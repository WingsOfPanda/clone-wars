# /clone-wars:consult v0.11.1 — Maintenance + Hardening

**Date:** 2026-05-04
**Status:** Draft
**Owner:** liupan

## Goal

Ship a v0.11.1 maintenance + hardening release on top of v0.11.0:

1. Refactor `lib/consult.sh` (~1280 lines) into 3 focused files plus a thin
   sourcing shim, so future hub-mode work and the recurring "split me" review
   feedback (flagged in 5+ code reviews during v0.11) is resolved.
2. Lift the slug regex `[A-Za-z0-9._-]+` into a shared constant
   `CW_SLUG_REGEX_BASE` in `lib/state.sh` so deploy + consult both reference
   one source of truth (no drift hazard).
3. Two hub-mode UX hardening features driven by /forcevision and
   /executeorder66 conventions:
   - Yoda auto-extracts target sub-projects from the topic text and confirms
     with the user (matches the "topic-as-canonical" pattern in both reference
     commands); explicit "all"/"everything"/"across all" keyword path
     auto-selects all leaves with zero AskUserQuestion (subject to a one-click
     confirm gate to avoid silent misdirection).
   - Trooper question events in hub mode get answered with the active
     sub-project's context, inferred via heuristic from the latest
     `### <leaf>` heading in findings.md.
4. Three defensive polish items:
   - Drilldown filename collision counter (-2/-3/... up to -99 cap).
   - Validator order documented in `bin/consult-design-doc.sh`; the
     acceptance-tests validator emits `log_warn` instead of silently skipping
     step-id cross-ref when `dag.md` is absent.
   - `bin/consult-design-doc.sh` warns if both `testing.md` AND
     `acceptance-tests.md` are present in `$DD_DIR` (mode-toggle indicator
     from manual hub-mode.txt edits).
5. Observability foundation: persist `_consult/findings-conformance.txt` per
   run so manual aggregation across runs informs v0.12+ decisions about
   whether to upgrade the question-protocol heuristic into an explicit
   trooper-side contract.

Single-repo behavior is byte-identical to v0.11.0; the regression baseline
fixture `tests/fixtures/v0.10-single-repo-design.md` continues to pass.
Hub-mode behavior is strict-superset enhancement (faster target selection,
better question answering, more mode-toggle resilience, no breaking contract
changes).

## Why

`lib/consult.sh` grew from ~720 lines (v0.10) to ~1280 lines (v0.11.0). Five
separate code reviews (Tasks 3 / 4 / 5 / 7 / 8 of v0.11) recommended splitting
it; the refactor was deferred to avoid mid-implementation rebase risk. v0.11
is now landed; v0.11.1 is the right window to do the split before the file
gains another ~500 lines from v0.12 work.

The slug regex `[A-Za-z0-9._-]+` is duplicated in 5+ call sites. A future PR
that tightens or loosens the character class in one site without the others
silently corrupts the cross-cutting validation contract — exactly the kind of
debt that compounds.

The two hub-mode UX items (#3 question protocol per-sub-project context, #6
auto-extract from topic) emerged from comparing /clone-wars:consult to its
reference cousins /forcevision and /executeorder66. Both reference commands
treat the topic text as canonical for target inference rather than asking
the user. v0.11.0's interactive picker is friction the user already pays for
in /forcevision-style runs but doesn't have to.

The defensive polish items (#4, #5, #7) are corner-case hardening surfaced
by code reviews during v0.11. They're cheap to land alongside the refactor
since most touch the same files.

The observability metric (#8) is the foundation for any future decision
about the v0.12 question-protocol upgrade: if the heuristic from #3 misfires
more than ~10% of runs, it's worth committing to an explicit trooper-side
`subproject` field. Without data we'd be guessing.

## Architecture

`/clone-wars:consult` v0.11.1 keeps the same 13-step pipeline and all
behavior contracts from v0.11.0. Five orthogonal changes:

1. **File-organization refactor** (`#1` + `#2`): `lib/consult.sh` (~1280
   lines) splits into three themed files (validators, prompts, hub) plus a
   thin sourcing shim that preserves existing `source lib/consult.sh`
   callers. Cross-cutting `CW_SLUG_REGEX_BASE` lifted to `lib/state.sh` so
   deploy + consult both reference one source of truth.
2. **Hub-mode UX hardening** (`#3` + `#6`): Yoda auto-extracts target
   sub-projects from the topic text and presents a single confirm-or-edit
   AskUserQuestion (matching `/forcevision`'s topic-as-canonical pattern);
   when "all"/"everything"/"across all" appears, a single confirm prompt
   covers the all-leaves case in one click. Trooper question events in
   hub mode get answered with the active sub-project's context (heuristic
   from latest findings.md `### <leaf>` heading).
3. **Validator + drilldown defensive polish** (`#4` + `#7`):
   `bin/consult-design-doc.sh` documents validator order; acceptance-tests
   validator emits `log_warn` when DAG is absent (instead of silent skip).
   Drilldown collision-counter prevents overwrites of prior round outputs.
4. **Mode-toggle resilience** (`#5`): `bin/consult-design-doc.sh` warns when
   both `testing.md` and `acceptance-tests.md` exist in `$DD_DIR` (indicates
   manual `hub-mode.txt` edit between runs). No active migration; user
   resolves manually.
5. **Observability foundation** (`#8`): per-run findings-conformance metric
   persisted to `_consult/findings-conformance.txt` for manual aggregation.

Single-repo behavior remains byte-identical to v0.11.0. Hub-mode behavior is
strict-superset enhancement; no contract changes that would break v0.11.0
state files in archive or in-flight.

The 3-way file split is the only change with non-trivial blast radius — all
86 existing tests must remain green. The shim's sourcing order matters
(`lib/state.sh` must be sourced before `lib/consult-hub.sh` because
`CW_SLUG_REGEX_BASE` is consumed there; `lib/consult.sh` shim sources its
3 split files in dependency order).

The shim must source state.sh-dependent files AFTER state.sh (callers
already do this). All existing `source lib/consult.sh` call sites in
`bin/consult-*.sh` and tests work unchanged.

### Slug constant cross-cutting (`#2`)

```
lib/state.sh:
  readonly CW_SLUG_REGEX_BASE='[A-Za-z0-9._-]+'

callers compose:
  consult-hub.sh::cw_consult_targets_persist:
    [[ "$line" =~ ^${CW_SLUG_REGEX_BASE}/${CW_SLUG_REGEX_BASE}$ ]]
  consult-hub.sh::cw_consult_detect_hub:

    [[ "$slug" =~ ^${CW_SLUG_REGEX_BASE}$ ]]
```

### Hub-mode target auto-extraction (`#6`)

Step 1.5 prelude flow in hub mode:

```
HUB_OUT=$(cw_consult_detect_hub "$(pwd)")
LEAVES_CSV=$(grep '^LEAVES=' <<< "$HUB_OUT" | cut -d= -f2)
HUBS_CSV=$(grep '^HUBS=' <<< "$HUB_OUT" | cut -d= -f2 || true)

EXTRACT=$(cw_consult_extract_targets_from_topic "$ARG_RAW" "$LEAVES_CSV") && EX_RC=0 || EX_RC=$?
INFERRED=$(grep '^INFERRED=' <<< "$EXTRACT" | cut -d= -f2)
KEYWORD_ALL=$(grep '^KEYWORD_ALL=' <<< "$EXTRACT" | cut -d= -f2 || echo 0)

case "$EX_RC,$KEYWORD_ALL" in
  0,1)
    # "all" / "everything" / "across all" → all leaves
    AskUserQuestion(
      "All leaves auto-selected (KEYWORD_ALL keyword in topic).
       Continue / Edit selection?"
      options=["Continue: $LEAVES_CSV", "Edit selection (open picker)"]
    )
    case $answer in
      "Continue: ...") CHOSEN_LEAVES="$LEAVES_CSV" ;;
      "Edit selection") <fall through to legacy Step 1.5 picker> ;;
    esac
    ;;
  0,0)
    # Inferred ≥1 leaf → confirm-or-edit AskUserQuestion
    AskUserQuestion(
      question="Inferred targets from your topic: $INFERRED. Confirm or edit?"
      options=[
        "Confirm: $INFERRED",
        "Edit selection (open picker)",
        "Abort"
      ]
    )
    case $answer in
      "Confirm: ...") CHOSEN_LEAVES="$INFERRED" ;;
      "Edit selection") <fall through to legacy Step 1.5 picker> ;;
      "Abort")          teardown + archive + exit ;;
    esac
    ;;
  *)
    # No inference → fall through to legacy Step 1.5 picker
    ;;
esac

printf '%s\n' "${CHOSEN_LEAVES//,/$'\n'}" | cw_consult_targets_persist "$TOPIC_DIR/_consult"
```

Single-repo mode: extraction never runs (gated by `HUB_MODE != single-repo`).

### Hub-mode question protocol heuristic (`#3`)

In `commands/consult.md` Step 3 (research-wait) and Step 5 (verify-wait), the
FS=question / VS=question handler is augmented:

```
TEXT=$(cw_consult_question_payload_read "$Q_FILE" TEXT)
OPTIONS=$(cw_consult_question_payload_read "$Q_FILE" OPTIONS)

if [[ "$HUB_MODE" != "single-repo" ]]; then
  ACTIVE_SP=$(cw_consult_findings_active_subproject "$FINDINGS_PATH") || ACTIVE_SP=""
  if [[ -n "$ACTIVE_SP" ]]; then
    log_info "active sub-project for question: $ACTIVE_SP"
    CONTEXT_SLICE=$(awk -v leaf="$ACTIVE_SP" '
      /^### / && $0 ~ leaf { in_block=1; print; next }
      /^### / && in_block { in_block=0 }
      in_block { print }
    ' "$FINDINGS_PATH")
  else
    CONTEXT_SLICE=$(cat "$FINDINGS_PATH")
  fi
else
  CONTEXT_SLICE=$(cat "$FINDINGS_PATH")
fi

# Existing critical/non-critical classification uses CONTEXT_SLICE
```

`cw_consult_findings_active_subproject` heuristic: parses findings.md,
returns the LAST `### <leaf>` heading found. The trooper's natural write
order means the most-recent heading is the leaf they're currently working
on. Returns rc=1 if no `### ` headings (single-repo / non-conformant
trooper) — caller falls back to full findings.

## Error Handling

### File-split shim sourcing failures (`#1`)

| Failure | Behavior |
|---|---|
| One of the 3 split files missing on disk | `source` fails, `lib/consult.sh` shim aborts. Caller (any bin script) gets unbound function error on first call. Treat as installation failure — refer user to `/clone-wars:medic`. |
| Function name collision between split files | Second `source` silently overrides the first. Mitigated by per-task ownership: each function lives in exactly one file. Test `test_consult_lib_shim_sources_all.sh` enumerates expected functions per file and asserts each is callable after sourcing only the shim. |
| `CW_SLUG_REGEX_BASE` unset when consult-hub.sh evaluates `[[ ... =~ ^${CW_SLUG_REGEX_BASE}/... ]]` | The `=~` would match against `^/[A-Za-z0-9._-]+$` (empty interpolation), accepting `<empty>/leaf` as valid — silent miscompare. Mitigation: state.sh sources before consult-*.sh in the canonical chain (already true today); shim test asserts the constant is defined post-source. |

### Slug constant accidental drift (`#2`)

| Failure | Behavior |
|---|---|
| Future PR adds a 6th caller that hardcodes `[A-Za-z0-9._-]+` instead of using `$CW_SLUG_REGEX_BASE` | No runtime failure — code works. But the constant's purpose (single source of truth) is silently defeated. Mitigation: `test_consult_slug_regex_constant.sh` greps for `[A-Za-z0-9._-]\+` literals across `lib/` and `bin/` and fails if any non-state.sh file has the bare regex. |
| Future PR tightens or loosens `CW_SLUG_REGEX_BASE` | All 5 callers behave consistently — that's the point. New regex semantics get covered by existing per-helper tests. |

### Topic-extraction false positives (`#6`)

| Failure | Behavior |
|---|---|
| Topic mentions a leaf name in non-target context (e.g. `"match the pattern in ARS-TaskServe when implementing the new ARS-Foo"`) — extractor returns both | AskUserQuestion confirm gate catches this — user picks "Edit selection" and uses the legacy picker. The cost of the false positive is exactly one extra click. |
| Topic contains "all" but user wanted a subset (e.g. `"document all the auth changes"`) | KEYWORD_ALL=1 fires; the explicit confirm AskUserQuestion ("All leaves auto-selected ... Continue / Edit selection?") is the safety net — turns the silent failure into a one-click recovery. |
| Topic contains a hub name only (`"refactor everything in ars_fleet"`) | Extractor returns all leaves under `ars_fleet`. Common case; intended behavior. AskUserQuestion confirm gate runs. |
| Topic has zero matches → fall through to legacy picker | Existing Step 1.5 behavior unchanged. |
| Substring ambiguity (e.g. `Gateway` matches `ARS-Gateway` + `ARS-LVMGateway` simultaneously) | Use word-boundary anchored matching to require full token match. If still ambiguous, surface ALL matches in INFERRED — confirm gate lets user disambiguate. |

### Active-subproject heuristic miss (`#3`)

| Failure | Behavior |
|---|---|
| `cw_consult_findings_active_subproject` returns rc=1 (no `### ` headings) | Caller falls back to passing the entire findings.md as context. Same as today's pre-v0.11.1 behavior. No regression. |
| Trooper jumped between leaves; question is about leaf-A but the latest heading is leaf-B | Yoda answers with leaf-B's context for a leaf-A question. Trooper either re-asks or proceeds; recoverable. The findings-conformance metric (#8) tracks how often this pattern appears across runs. |
| Trooper asks a meta question spanning leaves | Heuristic returns latest leaf; Yoda answers with that leaf's context only. Wrong scoping but recoverable. **Acceptance**: this is the stated trade-off of choosing B over A. |
| `findings.md` doesn't exist yet | `cw_consult_findings_active_subproject` returns rc=1 → fallback to "no findings" context. Existing v0.11 behavior. |

### Drilldown collision overflow (`#4`)

| Failure | Behavior |
|---|---|
| User runs >99 drill rounds on same section+sub-project+commander | `bin/consult-drilldown.sh` aborts with `too many same-section drilldown collisions; aborting`. User must manually clean `_scratch/`. |
| Concurrent drilldowns to same path | Bash test-then-rename is racy in principle but the directive serializes drilldown invocations. Not a practical concern. |

### Mode-toggle warn (`#5`)

| Failure | Behavior |
|---|---|
| Both `testing.md` and `acceptance-tests.md` present | `log_warn` fires, hub-mode wins (assemble uses `acceptance-tests.md`). User decides whether to rm the stale file. |
| Only one file present (normal case) | No warn, no behavior change. |
| Neither file present in hub mode | Hub-mode validator already errors (Task 10 of v0.11.0); unchanged. |

### Validator order doc (`#7`)

| Failure | Behavior |
|---|---|
| Future PR reorders validators | Acceptance-tests validator runs without dag.md populated → fires the new `log_warn`. The warning makes the regression visible in CI dogfood. Doc comment in design-doc.sh tells the next maintainer not to reorder. |
| Caller invokes acceptance-tests validator directly without dag.md present (e.g. unit test) | Same `log_warn` fires; cross-ref skips. Existing tests `test_consult_acceptance_tests_validate.sh` already exercise this path; updated to expect the warn on stderr. |

### Findings-conformance metric (`#8`)

| Failure | Behavior |
|---|---|
| Synthesis runs in single-repo mode | Both troopers get `<commander>=n/a` lines. |
| Trooper findings.md missing entirely | `cw_consult_findings_active_subproject` returns rc=1 → `<commander>=non-conformant`. Slight false positive (missing != non-conformant) but rare and the file is for trend analysis, not enforcement. |
| Conformance file write fails (disk full, permissions) | Synthesis continues (the metric is informational, not load-bearing). Log a `log_warn` and proceed. |

### Sourcing-shim backward-compat regression (`#1` cross-cutting)

| Failure | Behavior |
|---|---|
| External code does `source $CLAUDE_PLUGIN_ROOT/lib/consult.sh` and expects all functions defined in one file | Should still work — shim sources all 3 split files, so `declare -F` after `source lib/consult.sh` lists every function. No external-API surface change. |
| `lib/consult.sh` shim is `source`d twice in the same shell session | Functions get redefined (no error in bash). Idempotent. |
