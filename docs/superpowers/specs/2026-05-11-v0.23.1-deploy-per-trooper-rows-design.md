# v0.23.1 ‚Äî Deploy multi-repo: per-trooper sub-rows in Step 3b

**Date:** 2026-05-11
**Spec status:** approved (UX iteration on v0.23.0)
**Closes:** v0.23.0 dogfood UX feedback ‚Äî "we just shown ‚óº 3b DAG wave dispatch (multi-repo) in conductor progress, it is too little, i want at least show each trooper's task"

## Problem

`/clone-wars:deploy` multi-repo mode runs N codex/claude/opencode
troopers in parallel across DAG waves, each implementing a separate
sub-repo. The conductor task display rolls all of that activity into a
single `3b  DAG wave dispatch (multi-repo) [yoda+troopers]` row with
spinner `Multi-repo DAG dispatch`. From the user's seat the display
gives no signal about:

- which trooper is on which sub-repo
- which wave the deploy is currently in
- whether progress is happening or stalled per-trooper
- which sub-repo finished first when wave-internal completion is staggered

Each trooper's turn can take 30+ minutes (codex implementing a
non-trivial sub-repo unit). For 30+ minutes the conductor display says
nothing more than "DAG dispatch in progress". User asked for richer
visibility.

## Goal

After v0.23.1 lands:

- The upfront task table no longer contains a single `3b` row.
- At runtime, after `dag-waves.txt` is parsed in Step 3b, one
  `TaskCreate` call fires per `(wave, repo)` tuple.
- Subject string follows the canonical shape:
  `3b.<step> <Rank> <Cmdr> on <repo> [wave <w>]`
  (e.g. `3b.1 Captain Rex on auth-svc [wave 1]`).
- `activeForm` (spinner text when in_progress) reads:
  `<Rank> <Cmdr> implementing <repo>` (e.g. `Captain Rex implementing
  auth-svc`).
- Each sub-row flips `in_progress` when its spawn+send returns and
  `completed` when the wave-wait notification arrives with `TS=ok`.
- Single-repo deploy: zero touch (still uses `1.1/1/2/3` rows).
- `/consult` + everything else: zero touch.

## Architecture

Two changes, both small.

### Change 1 ‚Äî New `lib/commanders.sh` with `cw_cmdr_rank`

Tiny lookup helper. Source of truth for the canonical Star Wars rank
each commander holds in clone-army lore. Used only for sub-row label
rendering; not on the dispatch hot path.

```bash
# lib/commanders.sh
cw_cmdr_rank() {
  local cmdr="${1,,}"  # lowercase
  case "$cmdr" in
    rex|keeli|colt|trauma|blackout)        printf 'Captain' ;;
    cody|bly|wolffe|fox|gree|ponds|bacara|neyo|doom|faie)
                                            printf 'Commander' ;;
    hunter)                                 printf 'Sergeant' ;;
    havoc|thorn|thire|stone)                printf 'Lieutenant' ;;
    *)                                      printf 'Trooper' ;;
  esac
}
```

The case-list covers every entry in `config/commanders.yaml` that has a
canonical rank in lore (Rex Captain 501st; Cody/Bly/Wolffe/Fox/Gree/etc.
Commanders of their respective legions). Anything unknown falls through
to `Trooper` ‚Äî safe default that still renders cleanly. This list is
derived from Star Wars canon (Clone Wars TV series, Battlefront II,
Wookieepedia) and is not a load-bearing data shape ‚Äî extending it adds
flavor but never breaks anything.

### Change 2 ‚Äî Per-(wave, repo) `TaskCreate` in Step 3b

The upfront `TaskCreate √ó N BEFORE step 0` table drops the `3b` row.
Comment in the table explains that 3b sub-rows are dynamically created
at Step 3b entry once `dag-waves.txt` is known.

After `WAVE_GROUPS` and `WAVE_COUNT` are computed at the top of Step 3b
(directive prose lines ~715-770 of `commands/deploy.md`), insert a new
prose block:

```
source "$CLAUDE_PLUGIN_ROOT/lib/commanders.sh"

# For each (wave, step, repo) row in dag-waves.txt, create one TaskCreate
# call with subject "3b.<step> <Rank> <Cmdr> on <repo> [wave <w>]" and
# activeForm "<Rank> <Cmdr> implementing <repo>". Capture the returned
# task ID into REPO_TO_TASK_ID["<repo>"] for later in_progress / completed
# transitions.

declare -A REPO_TO_TASK_ID
while IFS=$'\t' read -r wave step repo desc; do
  cmdr="${REPO_TO_CMDR[$repo]}"
  rank=$(cw_cmdr_rank "$cmdr")
  # ISSUE: TaskCreate(
  #   subject="3b.$step $rank ${cmdr^} on $repo [wave $wave]",
  #   description="Plan + implement + self-verify $repo DAG unit (${REPO_TO_PROVIDER[$repo]})",
  #   activeForm="$rank ${cmdr^} implementing $repo"
  # )
  # CAPTURE task ID into REPO_TO_TASK_ID["$repo"]
done < "$ART_DIR/dag-waves.txt"
```

(`${cmdr^}` capitalizes the first letter ‚Äî `rex` ‚Üí `Rex`.)

In the wave loop, replace the existing `Set task '3b' ‚Üí 'in_progress'` /
`completed` calls with per-repo transitions:

```
# When dispatch returns rc=0 for a repo:
#   TaskUpdate(taskId=${REPO_TO_TASK_ID[$repo]}, status="in_progress")
#
# When that repo's wave-<cmdr>.txt arrives with TS=ok:
#   TaskUpdate(taskId=${REPO_TO_TASK_ID[$repo]}, status="completed")
```

The wave-wait + Stage 1 retry + Stage 2 partial-success branches all
operate per-repo already, so each branch threads through a single
`REPO_TO_TASK_ID` lookup.

### Conductor display effect

For a 3-sub-repo deploy with Wave 1: auth-svc + data-plane (parallel),
Wave 2: api-gw (depends on auth-svc):

```
‚úì 0   Audit + routing detect [yoda]
‚úì 3a  Preflight pane allocation (multi-repo) [yoda]
‚óº 3b.1 Captain Rex on auth-svc [wave 1]      ‚Üê spinner: Captain Rex implementing auth-svc
‚óº 3b.2 Commander Bly on data-plane [wave 1]  ‚Üê spinner: Commander Bly implementing data-plane
‚óã 3b.3 Commander Cody on api-gw [wave 2]
‚óã 3c  Final verification (multi-repo) [yoda]
‚óã 3d  Fix-loop (multi-repo) [yoda+troopers]
‚óã 4   Teardown + archive [yoda]
```

Two spinners run concurrently (one per in-flight trooper), each naming
its rank/commander/sub-repo. When wave 1 finishes both 3b.1+3b.2 flip
to ‚úì and 3b.3 (wave 2) starts.

## Failure modes

| Where | Trigger | Behavior |
|---|---|---|
| `cw_cmdr_rank` lookup | Unknown commander (e.g. user adds custom name to commanders.yaml) | Falls through to `Trooper`. Sub-row reads `Trooper Echo on ‚Ä¶`. Renders cleanly. |
| `TaskCreate` mid-step-3b | TaskCreate API failure (rare) | Same handling as elsewhere ‚Äî directive should `log_warn` and continue. The deploy itself proceeds; only the conductor display loses a row. |
| `REPO_TO_TASK_ID[$repo]` empty | Race where wave-wait fires before TaskCreate completes (impossible in practice ‚Äî TaskCreate is synchronous, wave-wait runs after spawn+send) | Defensive: directive checks `[[ -n "${REPO_TO_TASK_ID[$repo]:-}" ]]` before each `TaskUpdate`. |
| Stage 1 retry-once | Existing failure path ‚Äî wave's K spawns retried | Sub-rows for the failed repos remain `in_progress` across the retry (don't flip back to pending ‚Äî the work IS in progress, just on attempt #2). |
| Stage 2 partial-success | User picks "Proceed degraded" ‚Üí drop failed repo from `troopers.txt` | Mark dropped repo's sub-row `completed` (with description note "skipped per user choice") to clear the spinner. Future enhancement could use a "skipped" status if the API gains one. |

## Out of scope

- **Wave header sub-rows.** Could add `3b.wave1` / `3b.wave2` parent
  rows. Skipped ‚Äî adds clutter; wave is already encoded in each
  sub-row's `[wave <w>]` suffix.
- **`blockedBy` dependencies between sub-rows.** Could mark Wave 2
  rows blocked by Wave 1 rows. Skipped ‚Äî wave ordering is enforced by
  the directive logic, not by the conductor display; adding `blockedBy`
  would only affect display dimming, which doesn't add information the
  spinners + position don't already convey.
- **Live-updating activeForm with elapsed time** ("Captain Rex
  implementing auth-svc (5m)"). Skipped ‚Äî would require periodic
  TaskUpdate polling, not worth the complexity. Spinner + position
  already convey "still going."
- **Per-trooper rows for /consult.** Same shape would work for
  consult's Step 4/5/7/8 trooper rows but is out of scope here. Could
  be a v0.23.2 follow-up if the deploy visibility lands well.
- **Rich rank lookup loaded from YAML.** Could put rank table in
  `config/commanders.yaml`. Skipped ‚Äî bash case statement is fewer
  moving parts and edits are rare.
