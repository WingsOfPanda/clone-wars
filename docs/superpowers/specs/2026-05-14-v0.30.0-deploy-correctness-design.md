# v0.30.0 — Deploy correctness

**Date:** 2026-05-14
**Status:** Spec drafted, pending user review.
**Predecessor:** v0.29.0 simplification sweep (PR #94, merged).
**Successor / deferred:** v0.31.0 — project-local state relocation (separate brainstorm).

## Goal

Close four correctness gaps in `/clone-wars:consult` and `/clone-wars:deploy`
that surfaced in the dogfood of `consult-i-need-to-move-ars-a` →
`/clone-wars:deploy <design-doc>` (ARS-ReportLLM `skills/` move). User
reported: sub-repo commits landed on `main` with no rollback safety net,
the dirty-tree pre-flight stalled an autonomous loop, and the trooper
added scaffolding (`tests/run.sh`) not declared in the design doc. Each is
recoverable but each loses signal the user should have had at deploy
time. v0.30.0 fixes them at four chokepoints.

Item 5 (project-local state relocation, replacing
`~/.clone-wars/state/<repo-hash>/`) was scoped out into v0.31.0 because
its blast radius (touches every `cw_repo_hash`/`cw_consult_topic_dir`/
archive call site + every static-wiring lock) is independent of the
deploy-correctness theme and deserves its own brainstorm + migration
story.

## Background — why these four

The dogfood produced a flat 6-section design doc whose **content** was
unambiguously multi-repo (touched 3 trees: `ARS-ReportLLM/`,
`ARS-SkillServe/`, `ars_airdrop/` hub) but whose **header** had no
`**Target Sub-Project(s):**` declaration. `bin/deploy-init.sh` saw no
header → routed to single-repo mode → created
`feat/deploy-i-need-to-move-ars-a` only in `$TARGET_CWD = ars_airdrop/`.
The trooper followed the design's `### Sequencing` block correctly
(cd into each sub-repo as instructed), but the sub-repos had no feature
branch, so each `git commit` landed on `main`.

Root cause was upstream: `/clone-wars:consult`'s Step 10 multi-repo
auto-detector (`cw_consult_detect_multi_repo`) only examined the user's
one-sentence topic prose, which mentioned `ARS-ReportLLM` but not
`ARS-SkillServe`. The sub-repos that emerged during research were
invisible to the detector. Items 1–4 attack four layers of this failure
in defense-in-depth order:

1. **Item 1 — Detect at the right layer.** Swap Step 10's corpus from
   topic prose to `adjudicated.md` (the troopers' cross-verified output)
   so research-emergent sub-repos get caught.
2. **Item 2 — Catch the symptom if detection misses.** Deploy-side guard
   captures sibling sub-repo HEAD SHAs at Step 0 and verifies them at
   Step 4; rogue commits surface to user with revert + replay or
   accept-and-log options.
3. **Item 3 — Don't stall on benign pre-flight friction.** Yoda-side
   AskUserQuestion intercept on dirty-tree exit so autonomous loops
   don't deadlock.
4. **Item 4 — Catch trooper scope drift.** Diff-vs-Components match
   surfaces files the trooper added that aren't declared in the design
   doc.

Items 1 + 2 form a 1–2 punch on the same failure mode (Bug 1 from the
dogfood). Items 3 + 4 are independent improvements identified in the
same dogfood.

## Architecture

Single release. Two slash-command directives touched
(`commands/consult.md`, `commands/deploy.md`), one bin script amended
(`bin/deploy-init.sh` for structured exit codes), four new bin scripts
(`bin/deploy-sibling-baseline.sh`, `bin/deploy-sibling-verify.sh`, plus
the two parsers from item 4 if extracted), two new lib helpers
(`lib/deploy-sibling.sh`, `lib/deploy-scope.sh`). Every change preserves
single-repo, non-sibling-touching, clean-tree deploys at byte-equal
v0.29.0 behavior.

No schema changes to existing IPC files (`outbox.jsonl`, `state.txt`,
`pane.json`, `inbox.md`). All new state lives under `_deploy/` siblings
to existing files (`sibling-baseline.txt`, `sibling-rogue-accepted.txt`,
`pre-deploy-stash.txt`, `pre-deploy-commit.txt`, `scope-overrides.txt`).

## Item 1 — Consult Step 10 corpus swap

**Problem.** `commands/consult.md:745` calls
`cw_consult_detect_multi_repo "$PWD" "$(cat "$TOPIC_DIR/_consult/topic.txt")"`.
The topic.txt is the user's original sentence — for the dogfood case it
mentioned `ARS-ReportLLM` only. Sub-repos that became part of the design
through research (`ARS-SkillServe`, hub itself) weren't visible to the
detector.

**Change.** The caller in Step 10 swaps the second arg from
`topic.txt` to `adjudicated.md`. Lib helper signature unchanged
(`cw_consult_detect_multi_repo <cwd> <corpus>`), but the docstring is
updated to clarify: corpus is expected to be the troopers' cross-verified
output, not just the topic prose. The case-insensitive substring match
already does the right thing on a larger corpus.

**Hit-list display.** AskUserQuestion option labels gain occurrence
count to give the user a confidence signal. Compute via
`grep -ci '<slug>' adjudicated.md` per slug. Example label:
`Use auto-detected list (ARS-ReportLLM ×7, ARS-SkillServe ×4)`.
False-positive risk (sub-repo mentioned only in passing) is bounded by
the existing `Use / Edit / Proceed single-repo` choice — user always
has the override.

**Edge case — Step 10 runs before adjudicated.md exists.** Verify
sequencing in `commands/consult.md`: adjudicated.md is produced in
Step 9 (synthesis), Step 10 reads it. If `_consult/adjudicated.md` is
missing, fall back to topic.txt with a `log_warn` line so the failure
mode is observable. (Should not happen in normal flow; defensive only.)

**Files touched:**
- `commands/consult.md` (Step 10 corpus swap, AskUserQuestion label
  format)
- `lib/consult-walk.sh` (docstring update on
  `cw_consult_detect_multi_repo`)
- New tests:
  - `tests/test_consult_step10_corpus_swap.sh` — unit test asserting
    Step 10 reads adjudicated.md, falls back to topic.txt on missing
  - `tests/test_consult_detect_multi_repo_corpus_size.sh` — verify
    detector handles 100KB+ corpora without OOM (sanity)
- Static-wiring lock invariant in
  `tests/test_v0_30_0_static_wiring.sh`

**Out of scope.** Doesn't change the `Use / Edit / Proceed single-repo`
options. Doesn't add weighted scoring. Doesn't pre-promote without user
approval.

## Item 2 — Deploy adjacent-tree commit guard

**Problem.** `bin/deploy-init.sh` creates `feat/deploy-<topic>` only in
`$TARGET_CWD` (the hub for multi-repo, the single tree for single-repo).
Sibling sub-repos that the trooper touches but weren't declared in
`**Target Sub-Project(s):**` have no feature branch, so commits land on
`main` directly with no rollback path.

**Step 0 baseline.** New `bin/deploy-sibling-baseline.sh <art-dir>`:
1. Compute "siblings to watch":
   - Hub root = `$TARGET_CWD` for single-repo, `$HUB_CWD` for multi-repo.
     The hub itself is never in its own first-level listing, so
     self-inclusion isn't possible.
   - Enumerate first-level subdirectories of hub root: `<hub>/<entry>/`.
   - Skip `entry` whose name starts with `.` (hidden).
   - Keep `entry` only if `<hub>/<entry>/.git/` exists as a directory
     (so non-repo dirs like `docs/` or `scripts/` are skipped, and
     submodule gitlinks where `.git` is a file are also skipped).
   - Drop `entry` if its slug appears in declared
     `**Target Sub-Project(s):**` (read from
     `_deploy/multi-repo-targets.txt` if present, else empty list).
2. For each, capture `git -C <sibling> rev-parse refs/heads/main` (or
   the configured default branch via
   `git -C <sibling> symbolic-ref --short HEAD` at startup if no
   `main`).
3. Write `_deploy/sibling-baseline.txt` (TSV:
   `<sibling-slug>\t<baseline-sha>\t<branch-name>`).

**Step 4 verify.** New `bin/deploy-sibling-verify.sh <art-dir>` re-reads
each sibling's main SHA, compares against baseline. If any moved, lists
the rogue commits via
`git -C <sibling> log <baseline>..<head> --oneline`. Outputs a TSV
`_deploy/sibling-rogue.txt` (`<sibling-slug>\t<sha>\t<subject>`).

**Yoda intercept.** `commands/deploy.md` Step 4 reads
`_deploy/sibling-rogue.txt`. If non-empty, fires AskUserQuestion with
the rogue commit list inline:
- `Revert + replay on feat branch` (default) — runs
  `git -C <sib> revert --no-edit <sha-list>` (in baseline-to-current
  order to preserve dependencies), creates
  `feat/deploy-<topic>-rescue` in the sibling, cherry-picks the original
  commits onto it, leaves both branches in place for user to push or
  discard. Logs to `_deploy/sibling-rescue.txt`.
- `Keep on main (accept)` — no action, append to
  `_deploy/sibling-rogue-accepted.txt` for audit.
- `Send to trooper as fix-loop bug` — emit
  `_deploy/bugs.txt` entry referencing the rogue commits, fix round
  runs as normal.

**Lib helper.** `lib/deploy-sibling.sh` provides:
- `cw_deploy_enumerate_siblings <hub-cwd> <declared-targets-csv>` — emits
  one slug per line for siblings to watch.
- `cw_deploy_capture_sibling_baseline <sibling-cwd>` — emits TSV row.
- `cw_deploy_diff_sibling_against_baseline <sibling-cwd> <baseline-sha>`
  — emits commit list (oneline) since baseline.
- `cw_deploy_revert_and_replay <sibling-cwd> <topic> <sha-list>` —
  performs the revert + feat branch + cherry-pick orchestration. Returns
  rc=0 on success, rc=1 on revert conflict (escalates to user).

**Files touched:**
- `commands/deploy.md` (Step 0 baseline call + Step 4 verify call +
  AskUserQuestion intercept)
- `bin/deploy-sibling-baseline.sh` (new)
- `bin/deploy-sibling-verify.sh` (new)
- `lib/deploy-sibling.sh` (new)
- New tests:
  - `tests/test_deploy_sibling_enumerate.sh` — flat monorepo case,
    nested sub-repo case, declared-targets exclusion
  - `tests/test_deploy_sibling_baseline_capture.sh`
  - `tests/test_deploy_sibling_diff.sh`
  - `tests/test_deploy_sibling_revert_replay.sh` — clean revert,
    revert-with-conflict, multi-commit dependency order
  - `tests/test_deploy_sibling_v030_static_wiring.sh` invariant in
    main lock

**Edge cases handled:**
- Sibling has no `main` branch (uses `master` or other default) →
  capture via `symbolic-ref --short HEAD` at baseline time, store the
  branch name in baseline TSV, use stored name at verify time.
- Sibling is a submodule (gitlink) — skip; baseline only captures
  full git repos.
- Sibling has uncommitted work at baseline time — out of scope for
  v0.30.0; item 3 only handles the hub. v0.31.0+ may extend.
- Multi-repo deploy + sibling matches a declared target — excluded
  from baseline (the sibling has its own feat branch via
  `bin/deploy-multi-init.sh`).

## Item 3 — Deploy dirty-tree Yoda intercept

**Problem.** `bin/deploy-init.sh` correctly refuses to create a feature
branch when the working tree is dirty. For interactive use this is the
right safety check, but in autonomous loops (`/loop`-driven, scheduled,
etc.) it stalls without user attention.

**Structured exit code.** `bin/deploy-init.sh` is amended to exit `7`
specifically for the dirty-tree case (other failures keep their current
exit codes). Documented in the script's header comment.

**Step 0 intercept.** `commands/deploy.md` Step 0 traps rc=7. Fires
AskUserQuestion:
- `Stash and continue` (default) —
  `git -C $TARGET_CWD stash push -u -m "deploy <topic> WIP"`. Capture
  the stash ref via
  `git -C $TARGET_CWD stash list -1 --format=%H` immediately after the
  push (rather than relying on `stash@{0}`, which is index-relative and
  can be invalidated by concurrent stash operations during long deploys).
  Write the SHA + message to `_deploy/pre-deploy-stash.txt`. Continue
  deploy. Step 4 unwind matches by the captured SHA, not by index.
- `Commit first as chore: WIP` —
  `git -C $TARGET_CWD add -A && git commit -m "chore: WIP before deploy <topic>"`.
  Capture the SHA into `_deploy/pre-deploy-commit.txt`. Continue
  deploy.
- `Abort` — exit, no state written.

**Step 4 cleanup.**
- If `_deploy/pre-deploy-stash.txt` exists and deploy succeeded:
  attempt `git -C $TARGET_CWD stash pop` (specific stash ref). On
  conflict, surface to user with the stash ref + `git stash list`
  guidance — do not auto-resolve. Log outcome to
  `_deploy/post-deploy-stash-pop.txt`.
- If `_deploy/pre-deploy-stash.txt` exists and deploy aborted /
  failed: leave stash in place, surface ref to user with `git stash
  apply <ref>` guidance.
- If `_deploy/pre-deploy-commit.txt` exists: no special unwind. The
  commit lives on the feature branch alongside deploy work. User can
  `git rebase -i` post-merge to drop or squash.

**Hub-only scope (per brainstorm decision).** Sibling sub-repo
working-tree dirt is not pre-checked here — caught by item 2's
adjacent-tree guard if dirty work bleeds in via trooper commits.

**Autonomous-loop friendliness.** `Stash and continue` is reversible
and the default option, so autonomous loops won't deadlock. Per the
new global CLAUDE.md "execution phase" rules, Yoda picks the default
without halting if no human responds; the dialog is for visibility,
not a hard gate.

**Files touched:**
- `bin/deploy-init.sh` (one line: `exit 7` on dirty-tree, header
  comment update)
- `commands/deploy.md` (Step 0 rc=7 intercept block + Step 4 stash-pop
  cleanup block)
- New tests:
  - `tests/test_deploy_init_dirty_tree_rc7.sh` — assert rc=7 on dirty
    tree, rc=1 on other failures (regression-locks the convention)
  - `tests/test_deploy_dirty_intercept_stash_path.sh` — synthesize
    dirty tree, run intercept, verify stash captured + popped
- Static-wiring lock invariant.

## Item 4 — Deploy scope conformance check

**Problem.** Trooper added `tests/run.sh` (45 lines of scaffolding) not
listed in design doc's `## Components` "Files edited" table.
Single-turn deploy's self-verify only checks "did I implement the
design", not "did I implement only the design".

**Parser — `lib/deploy-scope.sh::cw_deploy_extract_components_paths
<design-doc>`.**
1. Scan from `## Components` heading until the next `## ` heading or
   EOF.
2. Within that range, find lines matching the markdown table syntax
   `| ... | ... |`.
3. Skip header row + separator row.
4. Extract first cell, strip backticks, strip leading/trailing
   whitespace.
5. Skip cells that don't look like file/directory paths
   (heuristic: contains `/` or matches `\.[a-zA-Z]+$`).
6. Output one path per line on stdout.

The parser tolerates:
- Tables under sub-headings (`### Files edited`) inside `## Components`.
- Multiple tables in `## Components` (concatenates all results).
- Surrounding prose between tables.

The parser does NOT tolerate (gracefully — emits no rows, returns rc=0
with empty output):
- Path lists in bullet form rather than table form.
- Path lists in code blocks.
v0.30.0 conventions enforce the table form (every existing design
doc in `docs/superpowers/specs/` uses tables); future docs that want
to use other forms can extend the parser.

**Matcher — `lib/deploy-scope.sh::cw_deploy_match_diff_against_components
<diff-paths-file> <components-paths-file>`.**

For each diff path, in-scope iff any of:
- `diff_path == listed_path` (exact match)
- `listed_path` ends with `/` AND `diff_path` starts with `listed_path`
  (explicit dir)
- `listed_path` does NOT end with `/` AND `diff_path` starts with
  `listed_path/` (implicit dir from a path the user wrote without
  trailing slash)

Out-of-scope paths printed to stdout, one per line. In-scope paths
suppressed.

**Step 4 enforcement.** After trooper done event, in
`commands/deploy.md` Step 4:
1. `git -C $TARGET_CWD diff --name-only $BASE..HEAD` per affected
   tree (hub for single-repo; hub + each declared sub-repo for
   multi-repo).
2. Concatenate diff paths into `_deploy/diff-paths.txt`.
3. Run parser → `_deploy/components-paths.txt`.
4. Run matcher → `_deploy/scope-out-of-scope.txt`.
5. If non-empty:
   - AskUserQuestion with the out-of-scope list inline:
     - `Accept and amend design retroactively` — Yoda offers a draft
       amendment to the Components table (added rows), user reviews
       via Edit tool, design doc updated in place.
     - `Send back to trooper to remove` — emit `_deploy/bugs.txt`
       entry, fix round runs.
     - `Force-keep without amending` — append to
       `_deploy/scope-overrides.txt` for audit.

**Files touched:**
- `lib/deploy-scope.sh` (new — both parser + matcher)
- `commands/deploy.md` (Step 4 scope-check block + AskUserQuestion)
- New tests:
  - `tests/test_deploy_scope_extract_components.sh` — single table,
    nested under sub-heading, multiple tables, no table edge case
  - `tests/test_deploy_scope_match_directory_prefix.sh` — exact match,
    explicit dir, implicit dir, mismatch
  - `tests/test_deploy_scope_step4_integration.sh` — synthesize design
    + diff, run end-to-end, verify out-of-scope list
- Static-wiring lock invariant.

## Risks

- **Item 1 false-positives.** Larger corpus = more incidental sub-repo
  mentions. Mitigation: occurrence-count display + existing override
  options. Worst case: user picks `Proceed single-repo`, behavior
  matches v0.29.0.
- **Item 2 revert+replay conflicts.** If the rogue commits conflict
  with the design's intended changes, `git revert` fails. Mitigation:
  `cw_deploy_revert_and_replay` returns rc=1 on conflict, escalates
  to user with the conflict file list. Don't auto-resolve.
- **Item 3 stash-pop conflicts.** Same risk: post-deploy
  `git stash pop` may conflict with deploy-introduced changes.
  Mitigation: don't auto-resolve; surface ref + guidance to user.
- **Item 4 parser robustness.** Markdown table format varies. Mitigation:
  enforce table form in design conventions (already true in all existing
  specs); parser tolerates common variations; failing parser surfaces
  zero out-of-scope paths (silent pass) rather than failing loud
  (avoid false positives blocking deploy).

## Rollback

Each item is independently revertible (separate commits). If item 2
conflicts with a real deploy, set
`CW_DEPLOY_DISABLE_SIBLING_GUARD=1` (env-var kill switch) before
running deploy — to be added in the implementation plan if surfaced
during dogfood. Same pattern for items 3 (`CW_DEPLOY_DISABLE_DIRTY_INTERCEPT`)
and 4 (`CW_DEPLOY_DISABLE_SCOPE_CHECK`). Item 1 has no kill switch
(the AskUserQuestion options are the override).
