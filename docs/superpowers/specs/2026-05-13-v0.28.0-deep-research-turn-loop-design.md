# v0.28.0 — `/clone-wars:deep-research` per-trooper turn loop design

**Status:** approved, ready for implementation plan
**Branch:** `feat/v0.28.0-deep-research-turn-loop`
**Date:** 2026-05-13

## Problem

`/clone-wars:deep-research` Phase 4 in v0.27.x is an **intra-turn loop**: Yoda enters Phase 4 once, then dispatches → foreground-waits → scores → dispatches inside a single Claude Code turn that can run for hours. During each foreground wait (up to 30 min via `bin/deep-research-experiment-wait.sh`), Yoda is fully blocked: cannot answer the user, cannot surface progress, cannot detect a stuck trooper.

Three concrete pain points from v0.27.x dogfooding:
1. **"Blank wait."** The user sees a wall of tool calls during 5–30 min experiments; no narrative, no live progress, no easy way to intervene.
2. **No liveness signal.** A hung trooper produces no `done` event; Yoda blocks until `experiment-wait.sh` times out. No early detection of "trooper crashed or stuck in a hyperparameter sweep that's clearly wrong."
3. **Coarse intervention seams.** Only at stagnation / time-budget does Yoda surface AskUserQuestion. Between those, no user touchpoints.

## Goal

Reshape Phase 4 from one intra-turn loop into **per-trooper independent turn cycles**:

- Each trooper has its own cadence; Yoda's next turn fires when **any one** trooper produces an event (completion, question, stuck signal, heartbeat).
- Between events, Yoda is **idle and responsive** — user can chat freely about the research or anything else.
- Liveness detection runs in the background; "stuck" troopers fire a notification with an escalation step (Yoda probes with `status?` before declaring failure).
- A custom completion-check (inspired by `/goal`'s pattern; NOT using `/goal` itself) replaces the v0.27.x stagnation-prompt + time-budget-prompt AskUserQuestions.
- Plugin-portable — no CLAUDE.md, no per-machine config; mechanism ships entirely inside `clone-wars/`.

## Non-goals (deferred to v0.29+)

- Inter-trooper messaging — troopers comparing notes directly without Yoda relay.
- Multi-session coordination — one user running multiple deep-research sessions concurrently.
- TUI pane snapshots for liveness — mtime-only + heartbeats is sufficient for v0.28.0.
- Migrating in-flight v0.27.x sessions to v0.28.0 schema. Clean break (per session full-replacement decision).

## Section 1 — Architecture overview

```
   Phase 3 spawns N codex troopers (existing).
              │
              ├─ Start N Monitor tasks (persistent: true), one per trooper,
              │  watching $trooper/outbox.jsonl for {done|error|question|
              │  heartbeat} events AND mtime-based stale-liveness signals.
              │
              ▼
   Yoda enters idle stance ───┐
              │               │
   user types? ─── yes ───────▼  Yoda's turn fires
   monitor fires? ── yes ────►  - reads queued notification(s) + scoreboard
                                - for each completion event: score, update
                                  rolling summary, completion-check + Yoda
                                  judgment, IF not done dispatch follow-up
                                  for that trooper (1-2 sentence direction),
                                  IF done mark complete
                                - for each stuck event: send "status?" probe
                                  or abort per Yoda judgment
                                - if user message, also respond conversationally
                                - update rolling summary, end turn
              │
              ▼
   All troopers complete OR halt-signal? ── yes ──► Phase 5 synthesis
                            │
                            no ──► back to idle stance
```

### Invariants

- **Yoda is never blocked between events.** No foreground `experiment-wait.sh` polling. Monitor handles waiting; Yoda only wakes on events.
- **Per-trooper independence.** Each trooper has its own state file; one trooper's failure does not affect others.
- **Single source of truth on disk.** All state persists under `_deep-research/`; any Yoda turn can replay from disk alone.
- **Monitor lifecycle bounded** by Phase 3 spawn ↔ Phase 4→5 transition. No dangling tasks.
- **Yoda gives direction, not detailed plans.** Each per-experiment dispatch is 1-2 sentences of strategic intent (≤ ~50 tokens / 3 lines). Reasons: keeps per-turn token cost low; preserves trooper autonomy. Continuity comes from `session-summary.md` (Section 6), not from re-deriving context every turn.

## Section 3 — Phase 4 directive prose structure

Phase 4 has **two entry modes**: initial entry (once, when `/clone-wars:deep-research` invoked) and re-entry (every subsequent turn, triggered by notification or user message).

### 3.a — Initial entry (once, end of Phase 3)

1. For each commander in `troopers.txt`:
   - Create `troopers/<commander>/{state.txt, experiments/, liveness-cursor.txt}` with `phase=idle, exp_counter=0`.
   - Start a `Monitor` task running `bin/deep-research-monitor.sh <art-dir> <commander>` with `persistent: true`. Record the task ID.
2. Append all task IDs to `monitor-tasks.txt`.
3. Touch `active.txt` with the topic slug.
4. Write initial `session-summary.md` (preflight info, empty scoreboard).
5. **First dispatch round.** For each trooper, Yoda composes a 1-2 sentence opening direction (informed by `topic.txt`, `metric.md`, `seed-from.txt`) and dispatches `exp-001` via `bin/deep-research-experiment-send.sh`. Update state: `phase=working, exp_counter=1, current_exp_id=exp-001`.
6. **End turn.** Yoda emits a chat message: "N troopers running first experiments. I'll pick up when they report back — you can ask anything in the meantime."

### 3.b — Re-entry handler (every subsequent turn while session is active)

```
1. Read state baseline:
   - scoreboard.md
   - all troopers/<cmdr>/state.txt
   - halt.flag (existence)
   - time-budget.txt + session-start.txt
   - count any new task-notifications since last turn

2. Hard-cap check (cheap, do first):
   - if halt.flag exists OR wall-clock cap exceeded → invoke bin/deep-research-finalize.sh
     and jump to Phase 5

3. Process queued notifications (in order):
   case event:
     done|error    → run bin/deep-research-score.sh → updates scoreboard.md
                     update trooper state.txt: phase=idle, clear current_exp_id,
                     last_event_ts=now
     question      → surface trooper's question to user; mark trooper phase=blocked;
                     do NOT auto-dispatch (wait for user direction)
     stale         → send "status?" probe via bin/send.sh; update state phase=stale,
                     probe_sent_ts=now. Debounce: skip if probe_sent_ts already set
                     within last LIVENESS_STUCK_S window.
     stuck         → Yoda judgment: abort (Ctrl-C the pane, mark phase=failed) or
                     extend (clear probe_sent_ts, give it more time)
     heartbeat     → just update last_event_ts; no further action

4. Completion check (across session):
   - call cw_deep_research_check_completion → TSV signal block
   - Yoda judgment overlay: should we stop?
   - if yes → mark all troopers phase=complete, touch halt.flag, jump to step 2

5. Dispatch round (idle troopers only):
   for each trooper where phase=idle AND halt.flag absent:
     - Yoda composes 1-2 sentence direction (informed by recent scoreboard + summary)
     - exp_counter++, EXP_ID=$(printf 'exp-%03d' $exp_counter)
     - bin/deep-research-experiment-send.sh dispatches it
     - update state: phase=working, current_exp_id, last_event_ts=now

6. Handle user message (if turn was triggered by user, not notification):
   - Detect halt intent ("stop", "halt", "we're done") → touch halt.flag, jump to step 2
   - Detect direction-change intent ("focus on Y for rex", "stop exploring X")
     → record in session-summary, factor into next dispatch
   - Otherwise respond conversationally (with status awareness)

7. Update session-summary.md (always last step before end turn)

8. End turn.
```

### 3.c — Re-entry trigger mechanism (UserPromptSubmit hook)

**The directive at `commands/deep-research.md` is read only on initial slash invocation. Re-entry needs a different mechanism.**

Mechanism: **`hooks/user-prompt-submit-active-session.sh`** shipped with the plugin, registered in `.claude-plugin/plugin.json` under `hooks.UserPromptSubmit`. ~25 lines bash.

Behavior on each UserPromptSubmit:
1. Resolve `$CLONE_WARS_HOME/state/<repo-hash>/`.
2. Scan for any `<topic>/_deep-research/active.txt`.
3. If found → emit a small context block telling Yoda to run handler 3.b. Block contents:
   - "Active deep-research session in progress: `<topic>`"
   - Path to handler 3.b directive section
   - "Run handler 3.b before responding to the user message."
4. If not found → exit 0, no output. Cheap no-op for 99% of Claude Code sessions.

For notification-fired turns (no user message), `<task-notification>` arrives natively; Yoda's directive context (still loaded from the most recent `/clone-wars:deep-research` invocation OR the hook on a prior user message) recognizes active.txt and runs 3.b.

## Section 4 — Liveness state machine

```
                  ┌───────────── done/error event ──────────────┐
                  │                                              │
   idle ──[Yoda dispatches]──► working ──[no events N min]──► stale
   ▲                            │  ▲                          │
   │                            │  │                          ▼
   │                       [Yoda sends "status?"]         (Yoda judgment)
   │                            │  │                       ╱      ╲
   │                            │  └── (new event) ───────╯      abort
   │                            │                                 ▼
   ├──────────────────────── question ◄───────────────────────► failed
   │                            ▲
   │                            │ (user answers in chat)
   │
   └── (completion-check OR halt.flag) ── complete
```

### Threshold semantics

- `$LIVENESS_PROBE_S` (default 300s) — outbox mtime older than this triggers `stale` notification.
- `$LIVENESS_STUCK_S` (default 600s) — outbox mtime older than this AND `probe_sent_ts` is set AND no new event since probe → `stuck` notification.

Both configurable per-topic at Phase 2 preflight; defaults usually fine.

### Monitor implementation

`bin/deep-research-monitor.sh` runs under `Monitor` with `persistent: true`. Loop:
1. Tail `outbox.jsonl` since last cursor. On `done|error|question|heartbeat`, emit one stdout line (notification body).
2. Every 60s, check outbox mtime against thresholds. Emit `stale` or `stuck` once per crossing (debounced via `liveness-cursor.txt`).
3. Exit cleanly when killed via TaskStop.

### Trooper-side heartbeats (optional, supplementary)

`config/prompt-templates/deep-research/experiment.md` gets a new instructional paragraph: "if your run will exceed ~5min wall-clock, emit `{event:\"heartbeat\",epoch:N}` to outbox every 2-3min." Codex may comply or not; when present, heartbeats keep mtime fresh and prevent false `stale` firings. When absent, mtime-only is the fallback — long quiet runs WILL trip `stale`, which Yoda's `status?` probe resolves cleanly.

### `question` phase

Sometimes troopers hit a real ambiguity (e.g. "validation or test set for the metric?"). Trooper emits `{event:"question", body:"..."}`. Yoda's handler surfaces it to user; marks `phase=blocked`. No auto-dispatch until user answers. User's chat answer flows back to trooper via `bin/send.sh`; trooper resumes, phase returns to `working`.

## Section 5 — Completion-check semantics

Two layers: a mechanical helper producing signals, and Yoda's judgmental decision consuming them.

### Helper: `cw_deep_research_check_completion <scoreboard> <metric>`

Prints TSV signal block to stdout:

```
floor_met=yes
target_met=yes
K_so_far=2
K_required=1
plateau=no
hard_cap=no
```

No decision, no recommendation — facts only.

Computation:
- `floor_met` — at least one `status=ok` row in scoreboard.md satisfying `min_acceptable` per `direction`.
- `target_met` — same against `target`.
- `K_so_far` — count of `status=ok` rows satisfying `target` (capped at `K_required`).
- `plateau` — last `plateau_window` (default 5) `status=ok` rows have max-min spread < `plateau_threshold` (default 0.01).
- `hard_cap` — wall-clock budget exceeded (`cw_deep_research_check_time_budget`).

### Yoda's decision policy

**Hard rules (no judgment):**
- `floor_met=no` AND `hard_cap=no` AND `halt.flag` absent → MUST keep going.
- `hard_cap=yes` OR `halt.flag` present → MUST stop (transition to Phase 5).

**Soft rules (Yoda judgment):**
- `floor_met=yes` AND `target_met=yes` AND `K_so_far >= K_required` → **default stop**, but Yoda may override if variance suspicious, user asked to keep exploring, etc.
- `floor_met=yes` AND `plateau=yes` AND `target_met=no` → **default stop**, but Yoda may override (pivot direction, request user input).
- `floor_met=yes` AND `target_met=yes` AND `K_so_far < K_required` → keep dispatching, look for corroboration.

Yoda's chat response on stop must cite the reason and show the signals — so the user can second-guess.

### Phase 1 dialogue additions

After existing target question, ask:
- "What's the minimum acceptable result you'd be OK shipping?" → `min_acceptable`
- "How many experiments at target before we call it done? (default 1)" → `K_corroboration`

`plateau_window` / `plateau_threshold` not asked — sensible defaults; user can edit metric.md directly.

### Renames (clarity, not semantics)

- `cw_deep_research_check_stagnation` → `cw_deep_research_check_plateau`. Same implementation; "plateau" is a feature signal, not a problem.

## Section 6 — Rolling summary (`session-summary.md`)

Yoda rewrites this every turn (atomic Write). Three audiences: user (glance during idle), Yoda's next turn (re-derive context cheaply), Phase 5 synthesis (raw material).

Section caps: top-5 scoreboard, last-10 events, last-5 decisions, ~3 sentences in Direction. Whole file stays ~200-400 lines max.

### Format

```markdown
# Research session — <topic-slug>
Updated: <ISO-8601 UTC>
Started: <session-start.txt value>
Time budget: <none | N hours, M remaining>

## Status

| Trooper | Phase | Current | Started | Last event |
|---|---|---|---|---|
| rex | working | exp-005 (modern LeNet + AMP) | 09:42 | 09:46 progress |
| keeli | stale | exp-004 (depthwise-sep) | 09:35 | 09:35 ack |
| bly | idle | — | — | 09:40 done (exp-003) |

## Scoreboard top 5

| Exp | Commander | Metric | Status | Runtime | Notes |
|---|---|---|---|---|---|
| exp-003 | rex | 0.9913 | ok | 412s | DW-sep + BN |
| ...

## Completion check

- Floor (>= 0.90): MET (since exp-001)
- Target (>= 0.99): MET (exp-003 at 0.9913)
- K corroboration: 1/1 → satisfied
- Plateau: NO (last 5 spread = 0.043)
- Hard cap: NO (3h remaining of 4h budget)

## Current direction

Both troopers exploring depthwise-separable convs since rex's exp-003 lead.
Trying weight decay sweep (rex: 1e-4, keeli: 5e-4) to test variance.
If both plateau, pivot to label smoothing.

## Recent decisions (last 5)

- exp-005 → rex: "WD 1e-4 on the exp-003 arch, same hyperparams otherwise"
- exp-004 → keeli: "depthwise-sep variant, target params < 80k"
- exp-003 → rex: "modern LeNet with batch norm + AMP" → 0.9913 ✓
- ...

## Recent events (last 10)

- 09:46 rex/exp-005 progress (epoch 12/60)
- 09:42 rex/exp-005 ack
- 09:42 rex dispatched exp-005
- 09:38 rex/exp-003 done (metric=0.9913)
- ...
```

### Update protocol per turn (step 7 of 3.b)

1. Read existing `session-summary.md` (or seed empty if first turn).
2. For each section, recompute mechanically from disk:
   - Status: read each `troopers/<cmdr>/state.txt` + last event from outbox.
   - Scoreboard top 5: `head -5` of sorted scoreboard.md.
   - Completion check: re-run `cw_deep_research_check_completion`.
   - Recent events: tail outbox.jsonl × N troopers, merge by ts, last 10.
3. Direction + Recent decisions: Yoda updates these with judgment (1-3 sentence diff vs prior).
4. Atomic write via `cw_atomic_write`.

### Helper

`cw_deep_research_render_summary <art-dir>` — renders sections 1, 2, 4, 5 mechanically (~80 LoC bash). Yoda fills in Direction + Recent decisions sections directly via Write.

### Why Recent decisions matters

This is the artifact that lets Yoda's short directions (Section 1 principle) stay coherent. When Yoda dispatches `exp-009` with one sentence, Yoda re-reads Recent decisions to see what was already tried, what worked, current strategy. Token-cheap continuity.

## Section 7 — Halt mechanisms + Phase 5 transition

Three halt paths converge on the same cleanup → Phase 5 trigger.

### Halt path A — user halt in chat

Yoda recognizes halt intent in step 6 of handler 3.b. Trigger phrases (case-insensitive substring): "stop", "halt", "we're done", "end research", "call it", plus structured phrasings like "stop research".

False-positive guard: if a halt phrase appears alongside a negation ("don't stop"), ignore. When uncertain, Yoda asks: "Halt now? (yes/no)".

Action: `touch _deep-research/halt.flag` with reason ("user-halted at 09:48").

### Halt path B — hard wall-clock cap

`cw_deep_research_check_time_budget` returns true. Step 2 of handler 3.b catches it. Action: `touch halt.flag` with reason `"time-budget-exhausted"`.

No AskUserQuestion at the cap — the v0.27.x prompt is deleted. If user wants extension, they say so in chat; Yoda updates time-budget.txt + session-start.txt + clears halt.flag.

### Halt path C — manual `/clone-wars:teardown <topic>`

Out-of-band. Existing teardown.sh archives the topic dir; Phase 4 handler 3.b detects this at step 1 (state files missing) and gracefully exits without writing further state. Monitor tasks become orphaned for ~60s until their watched outbox disappears; they error out and TaskStop is implicit. Best-effort cleanup runs at the next handler 3.b entry.

### Transition to Phase 5 (one cleanup path for all three)

`bin/deep-research-finalize.sh` runs the cleanup when `halt.flag` is detected:

1. Stop all monitor tasks via TaskStop with each ID from `monitor-tasks.txt`.
2. For each trooper currently `phase=working`: emit a final inbox prompt asking the trooper to wrap up cleanly (graceful abort, write partial result.json with `status=incomplete` if mid-run).
3. Wait briefly (~60s cap) for in-flight troopers to flush last events to scoreboard.md.
4. Update each trooper's `state.txt` to `phase=complete` or `phase=incomplete`.
5. Remove `active.txt` (so the UserPromptSubmit hook stops injecting handler 3.b context).
6. Final atomic write of session-summary.md with `## Halt` section noting reason.
7. Invoke Phase 5 synthesis (existing logic).

Phase 5 consumes `session-summary.md` as primary input alongside scoreboard.md.

Phases 6 (teardown) and 7 (present) — unchanged except for new file inventory in the archive.
