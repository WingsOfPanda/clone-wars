# v0.40.0 — Per-Session Isolation Design

> Closes the two same-repo cross-session bleed gaps surfaced by the
> v0.39.0 audit of intentions #1 (medic-once-globally) and #2
> (commands run in parallel across sessions without interfering).
> Intention #1 already holds; #2 is currently PARTIAL.

## Problem

When the user opens multiple Claude Code terminals in the **same repo**,
the v0.39.0 plugin has two interference points:

**Gap 1 — Hook returns the first `active.txt` it finds.**
`hooks/user-prompt-submit-active-session.sh:41`:

```bash
done < <(find "$STATE_ROOT" -maxdepth 4 -name 'active.txt' -path '*/_deep-research/*' 2>/dev/null)
```

Session A starts `/clone-wars:deep-research` in repo X. Session B then
submits any prompt in repo X — the hook sees A's `active.txt`, injects
the resume-handler directive, and B's Claude starts behaving as if it
is the advisor for A's research session. The hook is repo-scoped (per
v0.31.0) but not session-scoped. Today's design assumes one
deep-research per repo; reality is the user often runs multiple
terminals on the same repo.

**Gap 2 — `cw_commander_in_use` rejection has no session context.**
`bin/spawn.sh:168-172`:

```bash
if cw_commander_in_use "$COMMANDER" "$TOPIC"; then
  log_error "$COMMANDER is already deployed on $TOPIC; pick another commander"
  log_error "  or run: /clone-wars:teardown $COMMANDER $TOPIC"
  exit 1
fi
```

Two parallel sessions on identical `(topic, commander)` collide. The
rejection is correct (two writers on the same state dir would corrupt
each other) — but the message gives no hint about which session owns
the collision, so the user can't tell whether it's their own stale
state, a sibling terminal, or someone else on a shared dev box.

## Identifier

Claude Code exposes the session identifier in two complementary places,
both verified live on 2026-05-16:

| Where | How | Example |
|---|---|---|
| Subprocess env (Bash tool) | `$CLAUDE_CODE_SESSION_ID` | `1c6a4235-65ee-4bdb-b543-51a6f8b1d00a` |
| Hook stdin payload | JSON `.session_id` | same uuid |

The values match within a session, so init-time writes and hook-time
reads can correlate without a separate registry. Both forms are stable
across the lifetime of a single Claude Code session.

## Architecture

### Component 1 — Filename-stamped active marker (gap 1)

Rename the marker from `active.txt` to `active-<session-id>.txt`. Body
unchanged (topic slug).

```
~/.clone-wars/state/<repo-hash>/<topic>/_deep-research/
└── active-1c6a4235-65ee-4bdb-b543-51a6f8b1d00a.txt   ← body: "<topic-slug>"
```

Why filename-stamped (not body-encoded): the hook can short-circuit on
a single `find -name` glob without opening any file; ops can `ls
active-*.txt` to see every live session at a glance; the bare
`active.txt` reference becomes a search-and-replace target across the
codebase, so the lint can catch reintroductions trivially.

**Write path** — `bin/deep-research-init.sh:166`:

```bash
session_id="${CLAUDE_CODE_SESSION_ID:-unknown}"
printf '%s\n' "$TOPIC_NAME" > "$ART_DIR/active-${session_id}.txt"
```

**Delete path** — `bin/deep-research-finalize.sh:58`: remove the
session-specific file, not the bare one:

```bash
session_id="${CLAUDE_CODE_SESSION_ID:-unknown}"
rm -f "$ART/active-${session_id}.txt"
rm -f "$ART/active.txt"  # legacy v0.39.0 form — kept until 2 majors out
```

**Read path** — `hooks/user-prompt-submit-active-session.sh`:

```bash
# v0.40.0: read stdin JSON, extract session_id, match only this session's marker.
PAYLOAD=$(cat 2>/dev/null || true)
if command -v jq >/dev/null 2>&1; then
  SESSION_ID=$(printf '%s' "$PAYLOAD" | jq -r '.session_id // empty' 2>/dev/null)
else
  # Fallback: grep the field out of the JSON literal. CC payloads are
  # single-line, so a non-greedy match on the value works.
  SESSION_ID=$(printf '%s' "$PAYLOAD" \
    | sed -n 's/.*"session_id":[[:space:]]*"\([^"]*\)".*/\1/p' | head -1)
fi
[[ -n "$SESSION_ID" ]] || SESSION_ID="${CLAUDE_CODE_SESSION_ID:-}"
[[ -n "$SESSION_ID" ]] || exit 0   # no id, no match — silently no-op

# Defense in depth: session_id must look like a CC session id
# (uuid-shape). Anything else means a tampered payload — silently no-op
# rather than shelling out with attacker-controlled content.
[[ "$SESSION_ID" =~ ^[0-9a-zA-Z-]+$ ]] || exit 0
```

Then the find becomes:

```bash
done < <(find "$STATE_ROOT" -maxdepth 4 -name "active-${SESSION_ID}.txt" -path '*/_deep-research/*' 2>/dev/null)
```

### Component 2 — `.session_id` sibling in trooper state dirs (gap 2)

`lib/ipc.sh::cw_state_init` writes a `.session_id` file alongside the
freshly-initialized trooper state dir, containing
`${CLAUDE_CODE_SESSION_ID:-unknown}`.

```bash
cw_state_init() {
  local dir; dir=$(cw_trooper_dir "$1" "$2" "$3")
  mkdir -p "$dir"
  rm -f "$dir/identity.md" "$dir/inbox.md" "$dir/outbox.jsonl" "$dir/status.json" "$dir/pane.json" "$dir/.session_id"
  touch "$dir/outbox.jsonl"
  printf '%s\n' "${CLAUDE_CODE_SESSION_ID:-unknown}" > "$dir/.session_id"
}
```

The leading dot keeps it out of the `cw_is_artifact_dir` walk (which
only filters dirs whose name starts with `_`, but a hidden file in a
trooper dir is still ignored by `*/` globbing in
`cw_commanders_in_use_in_topic`).

**Read path** — `bin/spawn.sh:168-172`: enrich the existing error
message. Behavior unchanged (still rejects); message now points at the
owning session.

```bash
if cw_commander_in_use "$COMMANDER" "$TOPIC"; then
  trooper_dir=$(cw_trooper_dir "$COMMANDER" "$MODEL" "$TOPIC")
  owner=""
  if [[ -f "$trooper_dir/.session_id" ]]; then
    owner=$(head -1 "$trooper_dir/.session_id" 2>/dev/null | tr -d '\n')
  fi
  this_session="${CLAUDE_CODE_SESSION_ID:-unknown}"
  log_error "$COMMANDER is already deployed on $TOPIC; pick another commander"
  if [[ -n "$owner" && "$owner" != "$this_session" ]]; then
    log_error "  owned by another Claude Code session (id=${owner:0:8}…, mine=${this_session:0:8}…)"
  fi
  log_error "  or run: /clone-wars:teardown $COMMANDER $TOPIC"
  exit 1
fi
```

The `owner` field is informational only. When `.session_id` is missing
(state from a pre-v0.40.0 install) the message degrades to the v0.39.0
form — no functional regression.

**Why not scope state dir by session?** Considered and rejected. Path
shape is `<topic>/<commander>-<model>/` and is referenced from `lib/`,
`bin/`, `commands/`, and 30+ tests. Adding a session segment would
fork every path-aware test and break the trooper-identity model
documented in `docs/DESIGN.md` (where `<commander>-<model>-<topic>` is
the user-facing identity). The error message gives the user enough
context to disambiguate; the structural change carries no proportional
benefit.

## Lints

### Permanent lint — `tests/test_active_per_session_lint.sh` (no skip-guard)

Greps `bin/`, `lib/`, and `hooks/` for the bare token `active.txt`
(excluding the `active-*.txt` form, `providers-active.txt`, and lines
with an inline `# legacy` marker). Fails on any reintroduction.

```bash
#!/usr/bin/env bash
set -euo pipefail
cd "$(dirname "$0")/.."

# Find references to the bare "active.txt" filename. Exclusions:
#   - providers-active.txt (medic config — different file)
#   - active-*.txt (the new stamped form)
#   - any line containing "# legacy" or "# Legacy" (intentional bare ref
#     for backwards-compat cleanup of pre-v0.40.0 state)
#
# Word-boundary requirement: the match must be preceded by /, ", or
# start-of-line — so suffixes like "dev_active.txt" never trip it.
set +e
hits=$(grep -rn -E '(^|/|")active\.txt' bin/ lib/ hooks/ 2>/dev/null \
  | grep -vE 'providers-active\.txt' \
  | grep -vE '#[[:space:]]*[Ll]egacy')
rc=$?
set -e
if [[ $rc -gt 1 ]]; then
  echo "FAIL: grep error (rc=$rc)" >&2; exit 1
fi
if [[ -n "$hits" ]]; then
  echo "FAIL: bare active.txt references in bin/, lib/, or hooks/ (must be active-*.txt):" >&2
  echo "$hits" >&2; exit 1
fi
echo "PASS: no bare active.txt references in bin/, lib/, or hooks/"
```

### Version-stamped static-wiring — `tests/test_v0_40_0_static_wiring.sh`

Skips when `plugin.json` version != `0.40.0`. Activates and locks 7
invariants when version matches:

1. `plugin.json` and `marketplace.json` both read `0.40.0`.
2. `hooks/user-prompt-submit-active-session.sh` reads stdin (matches
   `cat 2>/dev/null` or equivalent).
3. Hook extracts `.session_id` (matches `jq -r '.session_id` OR
   `sed -n '.*"session_id"`).
4. Hook `find` pattern matches `active-${SESSION_ID}.txt` (not bare
   `active.txt`).
5. `bin/deep-research-init.sh` writes `active-${session_id}.txt`.
6. `lib/ipc.sh::cw_state_init` writes `.session_id` sibling.
7. `CLAUDE.md` "Current focus" header names v0.40.0.

## Migration

- Pre-v0.40.0 sessions that left a bare `active.txt` on disk become
  inert under the new hook (it ignores them). Cleaned up at next
  `/clone-wars:deep-research:finalize` invocation (finalize removes
  both forms).
- Pre-v0.40.0 trooper state dirs without `.session_id` continue to
  work — `cw_commander_in_use` rejection just doesn't include the owner
  prefix in those cases. No need to backfill.
- CHANGELOG entry calls out both behaviors so users know what to expect
  after upgrade.

### Dogfood (release-gate)

Two terminals open in the same clone-wars checkout.

1. **Terminal A:** `/clone-wars:deep-research what is the optimal CSS
   bundling strategy` → land at Phase 2 (time-budget pick).
2. **Terminal B:** submit any unrelated prompt (e.g. "what's in this
   PR?"). Expected: no deep-research resume injection, normal response.
3. **Terminal B:** submit a deep-research-relevant prompt (e.g. "tell
   me about CSS bundling"). Expected: still no injection — the hook
   only fires for terminal A's session id.
4. **Terminal A:** submit any prompt. Expected: resume handler fires,
   Yoda continues as advisor.
5. **Terminal A:** run finalize. Expected: A's `active-<sid>.txt`
   removed. Terminal B unaffected.

Mark v0.40.0 strict-dogfood row `[x] ok` in CHANGELOG when this passes
on a real machine.

## Out of scope (YAGNI)

- No global session registry, no cross-session crew sharing.
- No session id in pane labels (`<commander>-<model>-<topic>` stays
  the user-facing identity).
- No retroactive cleanup of v0.39.0 bare `active.txt` files outside
  of natural finalize calls — they're inert under the new hook.
- No change to `/clone-wars:consult`, `/clone-wars:deploy`,
  `/clone-wars:meditate` directives — they don't write `active.txt`
  and their state-dir collision behavior is governed by the
  topic-suffix retry already in `*-init.sh`.
