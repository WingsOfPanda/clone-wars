# v0.23.0 — DAG rescue UX: auto-proceed on verifiable extraction

**Date:** 2026-05-10
**Spec status:** approved (UX iteration on v0.21.0 rescue intercept)
**Closes:** v0.21.0 dogfood UX complaint — "the rescue intercept stops the auto-pipeline and feels like an error every time"

## Problem

v0.21.0's DAG rescue intercept (`commands/deploy.md` Step 5b) handles
hand-authored multi-repo design docs whose `## Execution DAG` section
is human prose (Unicode box diagrams, narrative wave descriptions)
instead of the parser-conforming `N. <slug> [(<abs-path>)] — <desc>`
format. The rescue extraction works correctly. But the UX has two
flaws:

1. **Alarming wording.** The directive prose says "Init failed —
   triggering rescue intercept" and the intercept block opens with
   "the failure may be a DAG-parse failure." From the user's
   perspective, nothing FAILED — the doc just used a different
   format and the system is recovering gracefully. The "failed"
   language makes a successful auto-recovery look like an error.

2. **Always asks for confirmation.** Sub-step 5b.4 fires
   `AskUserQuestion` on every rescue, even when Yoda's extraction is
   verifiably correct (slugs parse, paths exist on disk, every
   sub-repo has the expected `CLAUDE.md`/`AGENTS.md`). This breaks
   the auto-pipeline flow that `/clone-wars:deploy` is supposed to
   provide and adds 1 manual click per deploy on every hand-authored
   doc the user processes.

Reported by user during v0.22.0 dogfood on
`/home/liupan/ARS/ars_fleet/docs/designs/2026-05-10-three-standalone-skills.md`:

> "i used deploy again after reload, but do we have to show this
> every time? it not only stop the auto pipeline, but also make me
> feel it is errored"

## Goal

After v0.23.0 lands:

- The rescue path uses neutral, recovery-oriented wording. No
  "failed", no "intercept" framing in user-facing log_* lines or
  AskUserQuestion prompts.
- When Yoda's extraction is **verifiable** (every slug parses, every
  declared path exists with `CLAUDE.md`/`AGENTS.md`, parser+multi-init
  succeed), the rescue **auto-proceeds without confirmation**. One
  `log_ok` line summarizes the extraction; the deploy continues
  uninterrupted.
- When verification **fails** (slug invalid, path doesn't exist,
  required marker file missing), the AskUserQuestion fires as a
  safety net — but with a specific failure message, not a generic
  "looks right?" prompt.
- A `CW_DEPLOY_FORCE_RESCUE_PROMPT=1` env var lets users opt back
  into the v0.21.0/v0.22.0 confirm-every-time behavior.
- Single-repo + flat-monorepo + consult byte-equal v0.22.0.

## Architecture

Two changes, both directive-prose-only.

### Change 1 — Reword Step 5b for recovery-oriented language

Edits across the Step 5b block (intro paragraph, sub-step intros,
log lines):

| v0.22.0 wording | v0.23.0 wording |
|---|---|
| "DAG rescue intercept" | "DAG auto-extract" (heading + log lines) |
| "Init failed — triggering rescue" | "DAG section is prose; auto-extracting parser-conforming lines" |
| "the failure may be a DAG-parse failure" | "init exited non-zero because the `## Execution DAG` section uses a non-parser-conforming prose format" |
| "DAG rescue intercept: human-authored ... section detected" (log_info) | "DAG section is prose; auto-extracting parser-conforming lines" (log_info) |
| "rescue does not apply" (log_error) | unchanged — this IS an error path |
| "DAG rescue intercept complete; resuming Step 0 sub-step 6" (log_ok) | "DAG auto-extract complete (N lines); continuing" (log_ok) |
| AskUserQuestion `question:` field | "Auto-extracted DAG from prose section (verification PASSED). Use these lines?" (only fires under FORCE env var when verification is passing) |

The keyword "rescue" stays in the spec/plan/CLAUDE.md status rows
(internal nomenclature) but disappears from user-facing output.

### Change 2 — Add verification + conditional `AskUserQuestion`

New sub-step 5b.3.5 (between extract + confirm) — verification:

```bash
# Extract pass count (Yoda already produced parser-conforming lines)
NUM_EXTRACTED_LINES=<count>

# Per-line verification
VERIFY_FAILED=()
for each extracted line:
  parse: slug, optional_abs_path, desc, deps_csv
  # 1. slug regex
  if ! [[ "$slug" =~ ^[A-Za-z0-9_-]+$ ]]; then
    VERIFY_FAILED+=( "line N: slug '$slug' invalid (must match [A-Za-z0-9_-]+)" )
    continue
  fi
  # 2. path resolution
  if [[ -n "$optional_abs_path" ]]; then
    repo_cwd="$optional_abs_path"
  else
    repo_cwd="$TARGET_CWD/$slug"
  fi
  # 3. directory exists
  if [[ ! -d "$repo_cwd" ]]; then
    VERIFY_FAILED+=( "line N: directory not found: $repo_cwd" )
    continue
  fi
  # 4. marker file present
  if [[ ! -f "$repo_cwd/CLAUDE.md" && ! -f "$repo_cwd/AGENTS.md" ]]; then
    VERIFY_FAILED+=( "line N: $repo_cwd missing CLAUDE.md/AGENTS.md" )
    continue
  fi
done
```

Sub-step 5b.4 conditional logic:

```
if (( ${#VERIFY_FAILED[@]} > 0 )) || [[ "${CW_DEPLOY_FORCE_RESCUE_PROMPT:-}" == "1" ]]; then
  # Show AskUserQuestion. Verification failure messages cited inline
  # so user knows WHAT failed verification (not just "review these lines").
  question_body = (
    "Auto-extracted DAG from prose section. " +
    (verification failed → "Verification failed:\n" + join(VERIFY_FAILED, "\n") + "\n\n")
                       (forced via env  → "Force-prompt env var set; verification PASSED.\n\n") +
    "Lines:\n" + extracted_lines
  )
  AskUserQuestion(question=question_body, options=["Looks right — write & retry" (recommended), "Let me edit", "Abort deploy"])
  USER_CHOICE = answer
else
  # Auto-proceed. One log line summarizing.
  log_ok "DAG auto-extract: $NUM_EXTRACTED_LINES lines verified ($(printf '%s ' "${slug_summary[@]}"))"
  USER_CHOICE = "auto-verified"
fi
```

Sub-step 5b.5 (write into design-doc copy) is unchanged in
mechanism but the audit log gains the choice category:

```bash
printf 'rescued at %s; choice: %s; lines extracted: %d; verification: %s\n' \
  "$(date -u +%FT%TZ)" "$USER_CHOICE" "$NUM_EXTRACTED_LINES" "$VERIFY_STATUS" \
  > "$ART_DIR/dag-rescue.log"
```

`$VERIFY_STATUS` is one of: `auto-passed`, `prompted-passed`,
`prompted-failed-with-overrides`, `forced-prompt`.

### Ordering note

Verification runs AFTER extraction but BEFORE the `Edit` tool writes
into `design.md`. If verification fails AND user picks "Abort
deploy", design.md is never modified — clean state for next attempt.

If user picks "Let me edit" on a verification-failed prompt, they
provide corrected lines via the next chat message; Yoda re-runs
verification on the corrected lines; if those pass, auto-proceed.

## Backwards compatibility

- Default behavior change: rescue auto-proceeds when verification
  passes (was: always prompt). Considered breaking? No — the
  auto-proceed only fires when extraction is verifiably correct;
  the user gains time without losing safety. The verification check
  is stricter than the previous AskUserQuestion (which trusted user
  judgment without checking paths exist).
- `CW_DEPLOY_FORCE_RESCUE_PROMPT=1` lets users opt back into the
  v0.21.0/v0.22.0 always-prompt behavior. Documented in the
  Environment variables section of `commands/deploy.md`.
- Single-repo deploy: zero touch (Step 5b is multi-repo only and
  fires only when init exits non-zero on a multi-repo doc).
- Flat-monorepo multi-repo deploy: zero touch (rescue still only
  fires on init failure, which doesn't occur for parser-conforming
  DAG sections).
- /consult: zero touch.
- Audit log shape: extended `$VERIFY_STATUS` field appended; old
  readers (none in-tree) would still parse the prefix.

## Failure modes

| Where | Trigger | Behavior |
|---|---|---|
| Verification: slug regex | Yoda emitted a malformed slug | AskUserQuestion fires with "line N: slug 'X' invalid" cited |
| Verification: path missing | Yoda inferred a path that doesn't exist | AskUserQuestion fires with "line N: directory not found: <path>" cited |
| Verification: marker file missing | Path exists but no CLAUDE.md/AGENTS.md | AskUserQuestion fires with "line N: <path> missing CLAUDE.md/AGENTS.md" cited |
| Force env var set | `CW_DEPLOY_FORCE_RESCUE_PROMPT=1` | AskUserQuestion fires regardless of verification (audit-log notes "forced-prompt") |
| Re-parse after auto-proceed | `bin/deploy-dag-parse.sh` exits non-zero on the written DAG lines | Surface parser stderr + exit 1 (existing v0.22.0 behavior; rescue is one-shot, no second auto-extract attempt) |

## Out of scope

- **Heuristic confidence scoring.** Verification is binary
  (pass/fail) per check. No "70% confident, prompt anyway" middle
  ground.
- **Source-doc write-back.** Yoda only writes to `_deploy/<topic>/design.md`
  (the local copy). The user's source doc is never modified, so
  subsequent runs would re-extract. Acceptable: extraction is fast,
  and the user can manually add a `### DAG Lines` subsection to
  their source doc if they want to skip the extract path entirely.
- **Wave-grouping verification.** Verification doesn't cross-check
  Yoda's wave assignment against the prose (e.g., "Wave 1 has 3
  parallel boxes" → all 3 should have no inter-deps). The cycle
  check by `deploy-dag-parse` catches gross errors; finer-grained
  wave verification is v0.24+ if a real failure surfaces.
- **Per-doc preference memory.** No "remember my answer for this
  doc" feature. Each invocation is independent.
- **`/clone-wars:consult` analogue.** Consult doesn't have a parser
  intercept — its DAG is always emitted by `cw_consult_emit_soft_dag`.
  No work needed.
