# v0.33.0 — Schema Rework + Consensus Primitive Design

**Goal:** Close 5 dogfood findings that interlock around the result.json contract — disambiguate convergence-eligible metrics from trooper-self-reported numbers, surface schema-validation failures actionably, expand the schema for self-reported vs advisor-eligible distinction, ship the per-field consensus primitive every multi-round project needs, and document the inbox-stacking anti-pattern until the state-machine IPC redesign lands.

**Architecture:** Single schema migration touches `bin/deep-research-experiment-send.sh` (template render), `bin/deep-research-score.sh` (validation + scoreboard write), `lib/deep-research.sh` (validation helper + check_completion gate), `config/prompt-templates/deep-research/experiment.md` (trooper-facing schema doc). New script `bin/deep-research-consensus.sh` reads the scoreboard + result.json fields and writes a consensus.md side artifact. `bin/send.sh` gains a deep-research-trooper-aware warning. All changes preserve v0.32.0 result.json back-compat: existing archives still parse; new fields are optional.

**Tech Stack:** Bash 4.2+. No new dependencies. `jq` already optional (score.sh has grep fallback).

**Branch:** `feat/v0.33.0-schema-consensus`.

**Baseline:** v0.32.0 at main `265cb8b`; plugin.json reports `0.32.0`.

---

## Motivation

The CRF-extraction dogfood (15 rounds, ~24 hours, clone-wars v0.29.0) at `/home/liupan/crf/CLONE_WARS_DEEP_RESEARCH_FEEDBACK.md` surfaced an interlocked cluster around the result.json contract:

| # | Finding | Impact |
|---|---|---|
| 4 | Inbox stacking breaks dispatch-during-active-work (when `bin/send.sh` is used directly, bypassing experiment-send's phase guard) | medium |
| 5 | `cw_deep_research_check_completion` triggered `target_met=yes` after `exp-001` because trooper self-reported metric_value=1.0 with no relation to actual `field_agreement_rate` | high |
| 6 | `score.sh` silent `(skipping)` on malformed result.json fired on every score invocation, no fix path | medium |
| 10 | Every multi-round project needs per-field agreement; we hand-wrote `compute_consensus.py` + `split_agree_disagree.py` each round | high |
| 17 | Different troopers used different metric definitions (filled-count vs ratio vs agreement) — scoreboard unreliable for cross-trooper comparison | high |

## Design decisions

Five load-bearing choices made during brainstorming (2026-05-15):

**D1 — Metric source disambiguation (#5):** mandatory `metric_name` match. `result.json`'s `metric_name` field MUST equal `metric.md`'s `**Primary metric:**` value. `score.sh` rejects mismatched rows (writes per-experiment validation file, omits from scoreboard.md). `cw_deep_research_check_completion` reads only matching rows. Forces troopers to agree on what they're measuring. (Alternative considered: Yoda-computed advisor column. Rejected — requires Yoda to know how to compute every metric type, infeasible.)

**D2 — Validation feedback file (#6):** on schema-validation failure, `score.sh` writes `experiments/<exp-id>/result-validation.txt` with the specific failure reason (one line: `missing required field: metric_name` / `metric_name 'foo' != metric.md primary 'bar'` / `invalid status enum: 'fudged'`). Trooper sees it on its next inbox read; user can grep across experiments. Row still omitted from scoreboard.md. (Alternative considered: pre-flight validator script troopers must invoke. Rejected — troopers might skip it; doesn't help with already-emitted bad result.json.)

**D3 — Schema expansion (#17):** add three optional top-level fields to `result.json`: `self_reported_count` (number or null), `self_reported_ratio` (number or null), `self_reported_notes` (string or null). `metric_value` stays canonical and convergence-eligible (must match metric_name). Existing v0.32.0 archives without these fields still validate. Migration is just docstring + experiment template update. (Alternative considered: nested `{metrics: {primary, self_reported, advisor_computed}}`. Rejected — breaks every v0.32.0 test fixture.)

**D4 — Consensus primitive (#10):** new `bin/deep-research-consensus.sh <topic>`. Reads latest result.json from each trooper's most-recent ok experiment. Computes per-field agreement on `metric_name` + `metric_value` (epsilon=0.01 default, env override) + structured `notes` if JSON-parseable. Output: `_deep-research/consensus.md` with sections `## Agreed` / `## Contested` / `## All-missing`. Per-field provenance: which trooper(s) proposed each value. Limited to result.json shape — does NOT parse arbitrary filled outputs (that's the structured-extraction template's job, deferred to its own brainstorm). (Alternative considered: lib helper only. Rejected — Yoda discoverability is worse.)

**D5 — Inbox-stacking documentation (#4):** `bin/send.sh` adds a soft warning when the target pane is a deep-research trooper with `state.txt phase=working`. Format: `[WARN] trooper <cmdr> is mid-experiment (phase=working, current_exp_id=<id>); prefer bin/deep-research-experiment-send.sh after the next done event. Sending anyway.` Doesn't refuse — preserves the v0.32.0 abort script's call path. Real fix (per-experiment inbox files) deferred to architectural brainstorm B. (Alternative considered: per-experiment inbox files in v0.33.0. Rejected — ripples through experiment template, codex turn semantics, and consult/meditate which share send.sh.)

## Per-file changes

### `config/prompt-templates/deep-research/experiment.md`

**Schema doc rewrite — Section 4 (result.json schema):**

```
4. Atomically write {{BRANCH_DIR}}/result.json with this EXACT schema:

   {
     "branch_id":           "{{EXP_ID}}",
     "approach_label":      "{{APPROACH_LABEL}}",
     "metric_name":         "{{METRIC_NAME}}",
     "metric_value":        <number or null>,
     "status":              "ok" | "fail" | "timeout" | "cost_blown",
     "runtime_s":           <number>,
     "log_paths":           ["./stdout.log", "./stderr.log"],
     "notes":               "<free-form, max 500 chars>",

     "self_reported_count": <integer or null>,    // v0.33.0 optional
     "self_reported_ratio": <number or null>,     // v0.33.0 optional
     "self_reported_notes": "<string or null>"    // v0.33.0 optional
   }

   - metric_name MUST equal '{{METRIC_NAME}}' (rendered from metric.md
     primary_metric). Any other value is rejected by score.sh.
   - metric_value MUST be non-null when status="ok".
   - metric_value MUST be null when status != "ok".
   - self_reported_* fields are OPTIONAL advisory metrics; only metric_value
     drives convergence. Use them when your run measured multiple things
     and you want to surface them without confusing the scoreboard.
   - log_paths MUST exist on disk by the time you write result.json.
```

**Validation hint added at end:**

```
If score.sh writes experiments/{{EXP_ID}}/result-validation.txt after
your done event, your result.json was rejected. Read the file for the
specific failure, fix result.json, re-emit done.
```

### `lib/deep-research.sh`

**New helper `cw_deep_research_validate_result_json_v033`** — extends the existing `_cw_deep_research_validate_result_grep` with the v0.33.0 `metric_name` match check:

```bash
# cw_deep_research_validate_result_json_v033 <result-path> <expected-metric-name>
# Returns rc=0 if all required fields present + status enum valid + metric_name match
# + metric_value/status invariants hold. On failure: rc=1, single-line reason to stdout.
cw_deep_research_validate_result_json_v033() {
  local path="$1" expected_metric="$2"
  # ... existing grep-fallback checks (branch_id/approach_label/metric_name/...) ...
  # NEW: metric_name match
  local actual_metric
  actual_metric=$(grep -oE '"metric_name"[[:space:]]*:[[:space:]]*"[^"]*"' "$path" \
    | sed 's/.*"\([^"]*\)"/\1/')
  [[ "$actual_metric" == "$expected_metric" ]] \
    || { echo "metric_name '$actual_metric' != metric.md primary '$expected_metric'"; return 1; }
  return 0
}
```

Existing `_cw_deep_research_validate_result_grep` stays for back-compat (used by archive-replay code paths).

**Update `cw_deep_research_check_completion`** to filter scoreboard rows by `metric_name` matching `metric.md primary_metric`. Today (v0.32.0) it walks all rows regardless of metric_name; v0.33.0 reads the primary metric once at the top and skips non-matching rows.

### `bin/deep-research-score.sh`

**Validation gate (line 49-50 today):** replace silent `log_warn ... (skipping)` with:

```bash
# Get expected metric_name from metric.md
expected_metric=$(awk '/^\*\*Primary metric:\*\*/ {sub(/^\*\*Primary metric:\*\*[[:space:]]+/, ""); print; exit}' "$ART_DIR/metric.md")

# Validate with v0.33.0 metric_name match
if ! validation_err=$(cw_deep_research_validate_result_json_v033 "$result" "$expected_metric"); then
  : # validation passed
else
  # Write per-experiment audit file
  printf 'FAILED at %s: %s\n' "$(date -u +%Y-%m-%dT%H:%M:%SZ)" "$validation_err" \
    > "$branch_dir/result-validation.txt"
  log_warn "result.json invalid: $result (see result-validation.txt)"
  continue
fi

# Cleanup: if validation passed but a prior result-validation.txt exists, remove it
rm -f "$branch_dir/result-validation.txt"
```

(Implementation detail: the function returns rc=1 with reason on stderr; the assignment-and-test pattern above needs a slight refactor — actual impl will use the standard `if ! cmd; then ...; fi` pattern with the reason captured via redirect.)

**Scoreboard.md row format** — unchanged. The new `self_reported_*` fields don't appear in scoreboard (advisory only). They're queryable via direct result.json reads.

### `bin/deep-research-consensus.sh` (NEW, ~80 LoC)

```
Usage: bin/deep-research-consensus.sh <topic> [--epsilon=<float>]

Reads each commander's LATEST ok result.json under
$ART_DIR/troopers/<cmdr>/experiments/*/result.json. Computes per-field
agreement matrix. Writes $ART_DIR/consensus.md with three sections.

Exit codes: 0=ok, 1=no result.json files found, 2=usage error.
```

Output shape (`consensus.md`):

```markdown
# Consensus — <topic>

Generated: 2026-05-15T14:00:00Z
Epsilon for metric_value: 0.01

## Agreed (3 fields)

| Field | Value | Proposed by |
|---|---|---|
| metric_name | accuracy | rex, keeli, colt |
| status | ok | rex, keeli, colt |
| approach_label | resnet-small | rex, keeli, colt |

## Contested (1 field)

| Field | rex's value | keeli's value | colt's value |
|---|---|---|---|
| metric_value | 0.971 | 0.965 | 0.832 |

## All-missing (0 fields)

_(none)_
```

**Implementation:** awk-driven; uses jq if present, falls back to grep for field extraction. Epsilon comparison via awk's numeric `<=`. Field list is fixed (the v0.33.0 schema fields). Per-field provenance is sorted commander list.

### `bin/send.sh`

**Soft warning when target is a deep-research trooper with phase=working:**

```bash
# v0.33.0 #4 — soft warning if target is a mid-experiment deep-research trooper
# (real fix: state-machine IPC redesign in brainstorm B).
state_file="$ART_DIR/troopers/$COMMANDER/state.txt"
if [[ -f "$state_file" ]]; then
  phase=$(grep '^phase=' "$state_file" 2>/dev/null | cut -d= -f2 | tr -d '[:space:]')
  if [[ "$phase" == "working" ]]; then
    cur_exp=$(grep '^current_exp_id=' "$state_file" 2>/dev/null | cut -d= -f2 | tr -d '[:space:]')
    log_warn "trooper $COMMANDER is mid-experiment (phase=working, current_exp_id=$cur_exp); prefer bin/deep-research-experiment-send.sh after the next done event. Sending anyway."
  fi
fi
```

The check is best-effort — if state.txt doesn't exist (consult/meditate trooper), the warning silently doesn't fire. Doesn't refuse; doesn't change rc.
