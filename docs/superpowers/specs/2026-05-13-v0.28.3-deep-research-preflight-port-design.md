# v0.28.3 — Deep-Research Preflight Pane Allocation (Bug-Fix Port)

**Status:** approved 2026-05-13. Implementation pending writing-plans.

## Problem

The v0.28.2 strict-dogfood run on `/clone-wars:deep-research "optimize MNIST classifier accuracy under 100k params"` surfaced two linked bugs in Phase 3 (spawn roster):

1. **Sequential spawn despite parallel Bash calls.** The conductor issued two parallel `bin/spawn.sh` calls in a single message, but timestamp evidence shows them serializing:
   - rex `ready` event at `2026-05-13T11:03:42Z`
   - keeli `ready` event at `2026-05-13T11:04:34Z`
   - **Gap: 52s** (vs ~5s if truly parallel)

   Root cause is the same as the pre-v0.19.0 consult bug: `bin/spawn.sh:156-165` reads `<topic>/.last_pane` to choose which pane to split, then writes its own pane back. The N parallel processes race-and-serialize on this shared file. The 20s bootstrap sleep + identity-read wait per call compounds the effect.

2. **Uneven pane heights.** Each `cw_pane_spawn_down` halves the prior pane: T1 splits Yoda → 50% wide; T2 down-splits T1 → 25% tall; T3 (if N=3) → 12.5% tall. User-observed symptom: "pane is split into one, and into two, and into three, this cause the split tmux panes not evenly large".

`/clone-wars:meditate` and `/clone-wars:deploy` (multi-repo) already solved this by adopting the v0.19.0 consult preflight pattern. `/clone-wars:deep-research` is the only remaining N>1 spawner using the legacy sequential path.

User feedback logged at v0.28.2 dogfood: "i noticed that you did not spawn rex and keeli parallelly, you spawn rex first and then keeli. and there is no preflight on tmux pane. this might be a bug, log it, we fix it later."

## Goal

Port the v0.19.0 consult preflight architecture (`bin/preflight-layout.sh` + `bin/spawn.sh --target-pane`) to `/clone-wars:deep-research`'s Phase 3. After the port:

- All N panes appear within ~2s of the preflight call (visually atomic, no "1, then 2, then 3" appearance).
- Yoda's pane stays at `main-pane-width` (default ~50%); doesn't shrink as troopers spawn.
- All N `bin/spawn.sh` processes run in true parallel (only serialization is at the tmux server socket, ~50-150ms per `respawn-pane`).
- Single-trooper deep-research (not currently supported but not blocked) remains legacy-path-compatible if it returns.

Out of scope (explicit, will not be addressed in v0.28.3):
- 300s stale-threshold tuning (observed during dogfood; deserves its own discussion).
- Migrating `troopers.txt` to 2-col TSV (would touch 3 lib helpers + the v0.28.2 static-wiring lock).
- Deploy single-trooper legacy spawn path (out of v0.19.0 scope; still out of scope).
- N=1 deep-research support (not blocked by the port, but `cw_deep_research_pick_roster` minimum is 2; `preflight-layout.sh` minimum is 2).

## Architecture

Phase 3 is split into two sub-phases mirroring meditate Step 2 / consult Step 3a + 3b. A consult-shaped sidecar file feeds preflight without disturbing deep-research's native 1-column `troopers.txt`.

```
[Phase 3a: preflight — single foreground bash process]
conductor writes _deep-research/troopers-preflight.txt   (NEW sidecar)
                  └─ 2-col TSV: codex<TAB><commander>
                  └─ via cw_deep_research_write_preflight_sidecar (NEW lib helper)
       │
       ▼
conductor calls bin/preflight-layout.sh
                 --art-dir "$ART_DIR"
                 --troopers-from "$ART_DIR/troopers-preflight.txt"
                 "$DEEP_TOPIC" "$N"
       │
       ▼
   tmux split × N (off Yoda's pane)
   tmux select-layout main-vertical   (even heights)
   sentinel banner per pane (colored, per-commander label)
       │
       ▼
   _deep-research/preflight-panes.txt   (ordered: <commander>\t<pane_id>)

[Phase 3b: parallel dispatch — N parallel Bash tool calls]
conductor issues N parallel:
  bin/spawn.sh <commander> codex "$TOPIC"
               --target-pane <pane_id_from_preflight>
               --preflight-art-dir "$ART_DIR"
       │
       ▼
   bin/spawn.sh validates --target-pane against preflight-panes.txt
   tmux respawn-pane × N (each into its pre-assigned pane, parallel)
       │
       ▼
   sentinel banners replaced by codex TUIs
   troopers boot in parallel; cw_outbox_wait per trooper

[Stage 1 retry-once on any failure]
  meditate-pattern: full teardown + retry from Phase 3a.

[Stage 2 partial-success on second failure]
  meditate-pattern: AskUserQuestion → "Proceed degraded (M < N) / Abort".
```

Key properties (inherited from v0.19.0, verified by meditate v0.25.0 dogfood):

- **No `.last_pane` reads/writes** on the new path. Legacy `.last_pane` block in `bin/spawn.sh` remains for `--target-pane`-absent callers (deploy single-trooper).
- **Deterministic commander↔pane ordering** — `troopers-preflight.txt` order matches `preflight-panes.txt` order matches visual layout (Yoda left, troopers stacked right top→bottom in roster order).
- **True parallel dispatch** — each `bin/spawn.sh --target-pane` reads `preflight-panes.txt` read-only and calls `tmux respawn-pane`. No shared mutable file between spawn processes.

## Components

### NEW lib helper: `cw_deep_research_write_preflight_sidecar`

Location: `lib/deep-research.sh` (alongside `cw_deep_research_pick_roster`).

Signature: `cw_deep_research_write_preflight_sidecar <art-dir> <cmdr1> [<cmdr2> ...]`

- Receives roster as positional commander args (matches `cw_deep_research_pick_roster`'s line-per-commander output shape).
- Writes `<art-dir>/troopers-preflight.txt` via atomic tmp+mv (`cw_atomic_write` from `lib/state.sh` if present, else inline tmp+mv).
- Format: `codex\t<commander>` per line, no header, no comments. Trailing newline.
- Returns rc=0 on success; rc=1 on missing args or non-writable art-dir.
- Idempotent — overwriting a prior sidecar is fine.

```sh
# cw_deep_research_write_preflight_sidecar <art-dir> <cmdr1> [<cmdr2> ...]
# Writes consult-shaped 2-col TSV (codex\t<commander>) to <art-dir>/troopers-preflight.txt.
# Atomic; idempotent.
cw_deep_research_write_preflight_sidecar() {
  local art_dir="$1"; shift
  [[ -d "$art_dir" ]] || { log_error "art-dir not found: $art_dir"; return 1; }
  (( $# >= 1 )) || { log_error "need at least 1 commander"; return 1; }
  local tmp="$art_dir/troopers-preflight.txt.tmp"
  : > "$tmp"
  for cmdr in "$@"; do
    printf 'codex\t%s\n' "$cmdr" >> "$tmp"
  done
  mv "$tmp" "$art_dir/troopers-preflight.txt"
}
```

### `commands/deep-research.md` Phase 3 rewrite

Today's Phase 3 is a single block: N parallel `bin/spawn.sh <commander> codex "$DEEP_TOPIC"` calls + spawn-rollback runbook with auto-retry-once.

The rewrite splits Phase 3 into 3a + 3b. Phase 4 is unchanged.

**Phase 3a — Preflight pane allocation (foreground)**

1. Load ROSTER from `/tmp/cw-deep-research-roster.txt`.
2. Write the consult-shaped sidecar via the new lib helper:
   ```sh
   cw_deep_research_write_preflight_sidecar "$ART_DIR" "${ROSTER[@]}"
   ```
3. Invoke preflight-layout:
   ```sh
   "$CLAUDE_PLUGIN_ROOT/bin/preflight-layout.sh" \
     --art-dir "$ART_DIR" \
     --troopers-from "$ART_DIR/troopers-preflight.txt" \
     "$DEEP_TOPIC" "$N"
   ```
4. On rc=0: read `$ART_DIR/preflight-panes.txt` into a `PREFLIGHT_PANES` associative array (commander → pane_id).
5. On rc≠0: Stage 1 retry-once (full teardown + re-run). Same as Phase 3b retry logic — preflight is part of the retry budget per v0.19.0 spec.

**Phase 3b — Parallel dispatch (parallel Bash tool calls)**

1. Initialize `SPAWN_RETRY_COUNT=0` (same retry counter as the legacy block; shared with Phase 3a's retry).
2. **Issue N parallel Bash tool calls in a single message** — one per ROSTER entry:
   ```sh
   "$CLAUDE_PLUGIN_ROOT/bin/spawn.sh" \
     "$cmdr" codex "$DEEP_TOPIC" \
     --target-pane "${PREFLIGHT_PANES[$cmdr]}" \
     --preflight-art-dir "$ART_DIR"
   ```
3. Capture each rc separately. Evaluate tuple:
   - **All N succeed** → continue to Phase 4. Task `3` → `completed`.
   - **Any failed AND `SPAWN_RETRY_COUNT == 0`** → retry once. Tear down via `bin/deep-research-teardown.sh` (which now cleans preflight orphans — see below). Re-run Phase 3a + 3b. Set `SPAWN_RETRY_COUNT=1`.
   - **Any failed AND `SPAWN_RETRY_COUNT == 1`** → Stage 2 AskUserQuestion: `Proceed degraded (M-of-N troopers)` / `Abort deep-research`. On degrade, rewrite `troopers.txt` (1-col, native format) to drop failed entries; on abort, archive via teardown + exit.

### `bin/deep-research-teardown.sh` preflight-orphan cleanup

Today: `bin/deep-research-teardown.sh` runs after `bin/teardown.sh --pairs` (which kills the per-trooper panes) and `mv`s the topic dir to archive. It doesn't know about preflight sentinels.

v0.28.3 extension: before the `mv`, check for `$ART_DIR/preflight-panes.txt`. If present, for each `<commander>\t<pane_id>` line:
- If `<commander>` is NOT in `$ART_DIR/troopers.txt` (1-col native format), the pane is an orphan (preflight succeeded but spawn never replaced the sentinel). Kill it via `tmux kill-pane -t <pane_id> 2>/dev/null || true`.

This handles three cases:
- Stage 2 partial-success abort with surviving preflight sentinels.
- Preflight succeeded but user Ctrl-C'd between 3a and 3b.
- Stage 1 retry teardown (preflight panes from the failed attempt).

Implementation: ~10 lines added near the top of `bin/deep-research-teardown.sh`. No caller contract change.

Note: the consult-teardown's mechanically-equivalent extension (v0.19.0) is the precedent. `bin/teardown.sh --pairs` will not touch preflight orphans because they aren't in the per-trooper state dirs that `--pairs` iterates.

### `bin/spawn.sh` and `bin/preflight-layout.sh` — NO CHANGES

Both already support deep-research's needs:
- `bin/spawn.sh --target-pane <id> --preflight-art-dir <abs-path>` exists since v0.22.0 (deploy).
- `bin/preflight-layout.sh --art-dir <abs-path> --troopers-from <abs-path>` exists since v0.22.0.

Verified by code reading at:

- `bin/preflight-layout.sh:73-79` — `--troopers-from` precedence over default `$ART_DIR/troopers.txt`.

### Phase 4.a interaction (preserved)

Phase 4.a step 1 continues to write `troopers.txt` as 1-col (commander-only) via atomic tmp+mv. This is locked by `test_v0_28_2_static_wiring.sh` invariant 5 (atomic write to durable roster file).

The two roster files coexist:

| File | Schema | Writer | Readers | Purpose |
|---|---|---|---|---|
| `troopers.txt` | 1-col: `<commander>` per line | Phase 4.a step 1 (directive) | `cw_deep_research_render_summary`, `_list_commanders`, `_render_status_brief`, `bin/deep-research-score.sh` (filesystem fallback) | Continuity / status / synthesis |

## Failure modes

| Stage | Failure | Behavior |
|---|---|---|
| Phase 3a — sidecar write | Art-dir not writable | rc=1 from helper; conductor logs error + treats as Stage 1 retry trigger |
| Phase 3a — preflight-layout | tmux split fails midway | preflight-layout's trap-driven rollback kills already-created panes; rc≠0; Stage 1 retry |
| Phase 3a — preflight-layout | Yoda pane discovery fails | rc=1; Stage 1 retry; if retry also fails → AskUserQuestion: Abort |
| Phase 3a — preflight-layout | troopers-preflight.txt absent/malformed | rc=1; Stage 1 retry (regenerates sidecar) |
| Phase 3b — spawn | `--target-pane` not in preflight-panes.txt | Strict reject, rc=1 (spawn.sh validation is unchanged) |
| Phase 3b — spawn | M-of-N succeed after Stage 1 retry | Stage 2 AskUserQuestion: degrade-or-abort |
| Phase 3b — Stage 2 degrade | M < 2 succeeded | Force abort (deep-research min N is 2) |
| Teardown | preflight-panes.txt exists, troopers.txt does not | Kill all preflight panes (treat all as orphans) |
| Teardown | preflight-panes.txt absent (pre-v0.28.3 archive) | No-op (extension is a presence check) |
