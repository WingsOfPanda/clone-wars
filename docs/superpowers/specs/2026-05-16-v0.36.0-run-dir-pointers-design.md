# v0.36.0 — Eliminate `/tmp/cw-*` cross-session pointer races via project-local `_run/` dirs

**Status:** Spec — pending review
**Date:** 2026-05-16
**Branch:** `feat/v0.36.0-run-dir-pointers`
**Audience:** writing-plans skill (next step) + dogfood reviewer
**Predecessor:** v0.31.0 (project-local state relocation) — this completes that work

## Problem

`/clone-wars:*` directives use ephemeral `/tmp/cw-*` files to bridge state between separate Bash tool blocks within a single directive invocation. The names are session-global. When two `/clone-wars:*` invocations run in parallel (different terminals, different repos), they overwrite each other's pointer files.

### Observed symptom (reported during recent deploy)

> "during consult, one trooper report that 'The design doc changed on disk while I was validating it.'"
> "/tmp/cw-consult-* files keep getting overwritten by parallel sessions. Inlining the values directly."

### Causal chain

1. Session A in repo X, session B in repo Y both running `/clone-wars:consult`.
2. Session B's first Bash block writes its `$ARGS_FILE` path to `/tmp/cw-consult-args-path.txt`.
3. Session A's second Bash block (Step 12 walk-assemble) reads `/tmp/cw-consult-args-path.txt` → gets B's args path → derives B's topic → walk-assemble writes the design doc into B's `_consult/design-doc/` path.
4. B's drill-round trooper, mid-read of B's own design doc, sees the file replaced (atomic rename works, but the *content* flips). Outbox: "design doc changed on disk."

Symmetric: A or B can be writer or reader; either side's trooper can fire the message.

## Goal

Eliminate the cross-session pointer race by giving each directive invocation a fresh project-local working directory, while keeping the cross-Bash-block discovery mechanism that makes the pointer pattern necessary in the first place.

## Approach summary — Option A (project-local mktemp dir per invocation)

Mirror v0.31.0's `_args/` mktemp pattern with a sibling `_run/` dir purely for directive-internal pointers.

```
$state_root/                                 # = $PWD/.clone-wars (per v0.31.0)
├── _args/                                   # already exists — args files (v0.31.0)
└── _run/                                    # NEW (v0.36.0) — directive run dirs
    ├── .gitignore                           # auto-written: *
    ├── .last                                # cross-block discovery (see below)
    ├── consult.AbCdEf/                      # one dir per /clone-wars:consult invocation
    │   ├── args-path.txt
    │   ├── topic.txt
    │   ├── art-dir.txt
    │   └── fastpath-err
    └── deep-research.GhIjKl/                # one dir per /clone-wars:deep-research invocation
        ├── args-path.txt
        ├── topic.txt
        ├── art-dir.txt
        ├── topic-dir.txt
        ├── roster.txt
        └── repo-hash.txt
```

Cross-session collisions become structurally impossible:

- Different repos → different `cw_state_root` → different `_run/` paths.
- Same repo, parallel invocations → same `_run/` parent, but `mktemp -d` guarantees unique subdirs (`<command>.XXXXXX`).

## Approaches considered

1. **Project-local mktemp dir per invocation (chosen — Option A).** ~50 LoC of new bash; consistent with v0.31.0's `_args/` precedent. Cross-session collision is structurally impossible — different invocations in any combination of repo × session never share a path.

2. **Session-id pointers** (`/tmp/cw-${CLAUDE_SESSION_ID:-$$}-...txt`). Rejected: (a) Claude Code doesn't expose a stable session-id env var that survives across separate Bash tool invocations within a single session; (b) `$$` is per-Bash-block (each Bash tool call is a fresh shell — unique PID every call), so it doesn't help.

3. **Inline all values via Heredoc / nested `$()`** — what the user did as a workaround. Rejected as a permanent fix because directive Bash blocks have hard length limits and nested `$()` chains across 25 sites get unreadable fast; also breaks the conventional "one logical step per Bash block" rule.

4. **Atomic-rename retrofit on every reader** — rejected. The race is at the *writer* level (wrong session writes), not the reader. Atomic-rename in walk-assemble already exists (`bin/consult-walk-assemble.sh:100`) and addresses partial-read windows; it doesn't address whose-content-do-I-flip-to.

## Layer A — `cw_run_dir <command>` helper

New helper in `lib/state.sh`. Two responsibilities: (1) sweep stale run dirs, (2) mktemp a fresh one and writeback its path to `_run/.last`.

```bash
# cw_run_dir <command>
# Resolve a per-invocation directive scratch dir under $state_root/_run/.
# - Sweeps any sibling _run/<*>/ subdirs whose mtime is >24h old (idempotent;
#   crash-safe; concurrent live invocations untouched since they have fresh
#   mtime).
# - Creates _run/.gitignore = '*' on first call (mirrors v0.31.0's _args/).
# - Returns absolute path of a fresh mktemp -d _run/<command>.XXXXXX/ via stdout.
# - Writeback the same path to _run/.last so subsequent Bash blocks within the
#   same directive can read it without needing /tmp.
cw_run_dir() {
  local command="${1:-}" root run_root run_dir
  [[ -n "$command" ]] || { echo "cw_run_dir: missing <command> arg" >&2; return 2; }
  root=$(cw_state_root)
  cw_state_ensure
  run_root="$root/_run"
  mkdir -p "$run_root"
  [[ -f "$run_root/.gitignore" ]] || printf '*\n' > "$run_root/.gitignore"

  # Sweep: remove any sibling subdirs untouched in CW_RUN_SWEEP_S (default 86400s = 24h).
  local sweep_s="${CW_RUN_SWEEP_S:-86400}"
  local now d mtime
  now=$(date +%s)
  for d in "$run_root"/*/; do
    [[ -d "$d" ]] || continue
    mtime=$(stat -c '%Y' "$d" 2>/dev/null || stat -f '%m' "$d" 2>/dev/null || echo 0)
    if (( mtime > 0 )) && (( now - mtime > sweep_s )); then
      rm -rf "$d"
    fi
  done

  run_dir=$(mktemp -d -p "$run_root" "$command.XXXXXX") \
    || { echo "cw_run_dir: mktemp failed under $run_root" >&2; return 1; }
  printf '%s' "$run_dir" > "$run_root/.last.tmp" \
    && mv "$run_root/.last.tmp" "$run_root/.last"
  printf '%s\n' "$run_dir"
}

# cw_run_dir_last
# Read the most-recently-mktemp'd run dir for THIS project (i.e. THIS state_root).
# Used by directive Bash blocks 2..N to discover the run dir without /tmp.
# Errors if .last is missing (would mean the directive's first Bash block
# didn't call cw_run_dir).
cw_run_dir_last() {
  local root run_root last
  root=$(cw_state_root)
  run_root="$root/_run"
  last="$run_root/.last"
  [[ -f "$last" ]] || { echo "cw_run_dir_last: $last missing — first Bash block must call cw_run_dir" >&2; return 1; }
  cat "$last"
}
```

### Why `.last` works

- `cw_state_root` is project-local (v0.31.0) → `_run/.last` is project-local.
- Two parallel invocations *in different repos* see different `_run/.last` files → no collision.
- Two parallel invocations *in the same repo* → race on `.last`. **This is acceptable** because: (a) the same repo running `/clone-wars:consult` twice in parallel is the same user explicitly doing both, who can serialize them; (b) cross-Bash-block discovery within ONE invocation can't tolerate shared state with another invocation by design — the user's "inline the values" workaround is the only true solution to that pathological case. v0.36.0 fixes the cross-*repo* race (the actual reported scenario), not the same-repo-parallel race.

If the same-repo-parallel race becomes a real symptom later, v0.37.0 can add a TTY-keyed `.last.<tty>` shard. v0.36.0 closes the user-reported scenario without that complexity.

## Layer B — directive rewrites (4 files, ~25 sites)

Pattern:

**Before** (consult.md:128-141):
```bash
ARGS_DIR="${CLONE_WARS_HOME:-$HOME/.clone-wars}/_args"
mkdir -p "$ARGS_DIR"
ARGS_FILE=$(mktemp -p "$ARGS_DIR" -t 'consult.XXXXXX')
echo "$ARGS_FILE" > /tmp/cw-consult-args-path.txt
echo "$ARGS_FILE"
```
later block:
```bash
ARGS_FILE=$(cat /tmp/cw-consult-args-path.txt)
```

**After**:
```bash
source /home/liupan/CC/clone-wars/lib/log.sh
source /home/liupan/CC/clone-wars/lib/state.sh
RUN_DIR=$(cw_run_dir consult)
ARGS_DIR="$(cw_state_root)/_args"; mkdir -p "$ARGS_DIR"
ARGS_FILE=$(mktemp -p "$ARGS_DIR" -t 'consult.XXXXXX')
printf '%s' "$ARGS_FILE" > "$RUN_DIR/args-path.txt"
echo "$ARGS_FILE"
```
later block:
```bash
source /home/liupan/CC/clone-wars/lib/state.sh
RUN_DIR=$(cw_run_dir_last)
ARGS_FILE=$(cat "$RUN_DIR/args-path.txt")
```

Same shape applies to all 4 directives. The conversion is mechanical:

- `/tmp/cw-<command>-<key>.txt` → `$RUN_DIR/<key>.txt`
- First Bash block adds `RUN_DIR=$(cw_run_dir <command>)`
- Subsequent Bash blocks add `RUN_DIR=$(cw_run_dir_last)`

## Defaults

| Knob | Default | Env override | Meaning |
|---|---|---|---|
| `CW_RUN_SWEEP_S` | `86400` (24h) | `CW_RUN_SWEEP_S` | Stale threshold — run dirs older than this are swept on next `cw_run_dir` call |

24h covers v0.34.0 long-session deep-research workflows (typical 4-12h sessions; outliers up to 24h) without preserving multi-day clutter. Crashed sessions get cleaned by the next invocation in the same repo. Active sessions are untouched (they have fresh mtimes).

## Out of scope

- **Same-repo-parallel invocation** — `_run/.last` race acknowledged; deferred to v0.37.0 if observed in practice.
- **Atomic-rename retrofit** — `bin/consult-walk-assemble.sh:100` already does it. Nothing to add.
- **Migration of in-flight v0.35.0 sessions** — directive `/tmp/cw-*` files are ephemeral within an invocation; an in-flight v0.35.0 session has the relevant pointers in `/tmp` already. v0.36.0 just adds a NEW directive flow; in-flight sessions keep working until they exit. New invocations after upgrade use the new flow.
- **Backwards-compat for archived `/tmp/cw-*` files** — none required; these were always ephemeral.

## Release-gate dogfood

Run two parallel `/clone-wars:consult` sessions on different topics in different repos. Verify:

1. Each session writes to its own `<repo>/.clone-wars/_run/consult.XXXXXX/` dir (use `ls $repoX/.clone-wars/_run/` and `ls $repoY/.clone-wars/_run/` between sessions).
2. Neither session's drill trooper reports "design doc changed on disk."
3. After both sessions complete, `_run/.last` in each repo points at the session's own dir (not crossed).
4. Run a third `/clone-wars:consult` in repoX 25 hours later (or with `CW_RUN_SWEEP_S=1` for fast-test) — the prior session's dir gets swept; new session's dir is fresh.
5. `git status` in both repos shows `.clone-wars/` ignored (gitignore at `.clone-wars/.gitignore` from v0.31.0 + `_run/.gitignore` from v0.36.0).
6. `/clone-wars:meditate` + `/clone-wars:deep-research` exhibit the same isolation — each invocation gets its own `_run/<command>.XXXXXX/` dir.

## Open questions for reviewer

1. **24h sweep window** — long enough for your longest deep-research advisor sessions? You've reported up to 18h in past dogfood. 24h gives headroom; bump to 48h via `CW_RUN_SWEEP_S=172800` if you want more.
2. **`.last` race in same-repo parallel invocations** — accepted as deferred. If you actually run two `/clone-wars:consult` invocations in the *same repo* at the same time, second one's `.last` writeback wins and first one's later Bash blocks find the wrong dir. Same symptom as the bug we're fixing, just reduced from cross-repo to same-repo. Is that acceptable for v0.36.0, or should this also be in scope?
3. **Verbose `RUN_DIR=$(cw_run_dir_last)` boilerplate at the top of every Bash block** — 1 source line + 1 helper call per block, ~50 sites total. Could compress with a sourceable shim (`source /home/liupan/CC/clone-wars/lib/_directive-prelude.sh`) — but that adds an indirection. Status quo is more verbose but explicit. Preference?
