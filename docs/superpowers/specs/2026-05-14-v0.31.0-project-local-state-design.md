# v0.31.0 — Project-local state relocation

**Date:** 2026-05-14
**Status:** Spec drafted, pending user review.
**Predecessor:** v0.30.0 deploy correctness (PR #95, merged at commit 62d066e).
**Successor / deferred:** v0.32.0 — drop the vestigial back-compat shims
(`cw_topic_repo_hash`, `cw_repo_hash_for`, `CLONE_WARS_HOME` env var fallback)
once no callers remain.

## Goal

Eliminate the global `~/.clone-wars/state/` namespace by relocating all
clone-wars state to `<invoking-cwd>/.clone-wars/`. The change closes one
correctness bug surfaced during the v0.30.0 implementation
(cross-session bleed via the `UserPromptSubmit` hook) and one UX issue
the user flagged in the same session (args-file collision across
parallel slash-command invocations). Item 5 was scoped out of v0.30.0
specifically to get its own brainstorm + design.

**Hard cutover, no back-compat fallback.** Matches v0.14.0 / v0.16.0
precedent in CLAUDE.md ("breaking change for archived consult dirs
without back-compat per v0.14 precedent"). v0.31.0 reads/writes only
project-local state; existing `~/.clone-wars/state/` becomes invisible
to v0.31.0+ tooling. Migration is leave-alone: documented, user manages
their own files.

## Background — why these two fixes together

### Hook bleed (the new correctness bug)

`hooks/user-prompt-submit-active-session.sh` (registered in
`plugin.json`) fires on every `UserPromptSubmit` event in any Claude
Code session that has the clone-wars plugin loaded. In v0.30.0 the hook
scans the global state root:

```bash
STATE_ROOT="${CLONE_WARS_HOME:-$HOME/.clone-wars}/state"
find "$STATE_ROOT" -maxdepth 4 -name 'active.txt' -path '*/_deep-research/*'
```

Because the state root is shared across every project the user runs
clone-wars in, an active deep-research session in (say)
`/home/liupan/ARS/` fires reminder context in an unrelated
`/home/liupan/CC/clone-wars/` Claude Code conversation. The user
verified this concretely during the v0.30.0 implementation:

> "deep-research-fill-chance-2-crf is keep going. so as i noticed, we
> should manage those status file in different place with different
> path, so they won't conflict with each other"

### Args-file collision (the UX bug)

Every directive writes `$ARGUMENTS` to a stable file before invoking
the bin script:

```bash
ARGS_DIR="${CLONE_WARS_HOME:-$HOME/.clone-wars}/_args"
echo "$ARGS_DIR/consult.txt"
```

Two parallel `/clone-wars:consult` invocations in different Claude
Code sessions clobber each other's args before init reads them. Same
applies to medic / deploy / meditate / deep-research args files. The
user flagged this explicitly during the v0.30.0 brainstorm:

> "for each consult, we can not write into same path, we may use
> consult in different sessions"

Both bugs root in the same architectural choice (shared global
namespace) and benefit from the same fix (per-project namespace).
Bundling them keeps the migration story simple — one breaking change,
not two.

## Architecture

### Single canonical change: `cw_state_root` returns project-local

`lib/state.sh:13`:

```bash
# Before (v0.30.0):
cw_state_root() {
  printf '%s\n' "${CLONE_WARS_HOME:-$HOME/.clone-wars}"
}

# After (v0.31.0):
cw_state_root() {
  # CLONE_WARS_HOME is honored as a test/debug seam only — not advertised
  # in user-facing docs. Production flow never sets it; the env var
  # exists so the 96 existing tests that set CLONE_WARS_HOME=<tmpdir>
  # for sandbox isolation continue working without bulk rewrite. v0.32.0
  # may revisit and rip out the env var if no test still needs it.
  if [[ -n "${CLONE_WARS_HOME:-}" ]]; then
    printf '%s\n' "$CLONE_WARS_HOME"
  else
    printf '%s/.clone-wars\n' "$PWD"
  fi
}
```

`$PWD` is evaluated at function-call time, which equals the conductor's
invocation cwd because directive Bash blocks inherit the user's working
directory. Every other helper in `lib/state.sh` composes paths through
`cw_state_root` — they automatically inherit the new shape with no
further edits.

**Default behavior: project-local.** Hard cutover for user-facing
behavior. The `$HOME/.clone-wars` default is gone — the only way to
reach the old shape is to explicitly set `CLONE_WARS_HOME`, which is
a test-only escape valve. The hook patch (below) uses `$PWD`
directly (independent code path), so cross-session bleed is fixed
regardless of whether `CLONE_WARS_HOME` is set anywhere.

### Path-shape: `<hash>` segment kept for v0.31.0 (deferred cleanup)

In v0.30.0:

```
~/.clone-wars/state/<sha256-of-cwd>/<topic>/_consult/...
```

In v0.31.0:

```
<invoking-cwd>/.clone-wars/state/<sha256-of-cwd>/<topic>/_consult/...
```

The `<repo-hash>` segment is now cosmetic (project-local already
disambiguates by directory location), but **dropping it is deferred
to v0.32.0**. Pre-plan audit on 2026-05-15 found 8+ tests that
hard-code the `state/<hash>/<topic>` shape in their assertions
(`test_commanders.sh`, `test_consult_wait_question_rearm.sh`,
`test_ipc_archive.sh`, `test_consult_wait_state_file.sh`,
`test_consult_spawn_rollback.sh`, `test_deploy_wave_wait.sh`,
`test_state.sh`, `test_send_from_flag.sh`). Mass-updating those
tests is meaningful churn (~150 LoC) on top of the v0.31.0 user-
facing fixes. Defer to a focused v0.32.0 cleanup sweep that can
decide whether to drop the hash at all (project-local makes it
purely cosmetic — keeping it forever is also defensible).

`lib/state.sh` `cw_repo_state_dir` / `cw_topic_state_dir` keep their
current shape:

```bash
# v0.31.0 — unchanged from v0.30.0:
cw_repo_state_dir() {
  printf '%s/state/%s\n' "$(cw_state_root)" "$(cw_topic_repo_hash)"
}
cw_topic_state_dir() {
  printf '%s/state/%s/%s\n' "$(cw_state_root)" "$(cw_topic_repo_hash)" "$1"
}
```

`cw_topic_repo_hash` keeps its hash-computation logic (loses the
`CW_TOPIC_REPO_CWD` branch per the env-var removal — see below).

### Auto-write `.gitignore` on first state-dir creation

`lib/state.sh:17`:

```bash
# Before (v0.30.0):
cw_state_ensure() {
  local root; root=$(cw_state_root)
  mkdir -p "$root/state" "$root/archive"
}

# After (v0.31.0):
cw_state_ensure() {
  local root; root=$(cw_state_root)
  mkdir -p "$root/state" "$root/archive"
  [[ -f "$root/.gitignore" ]] || printf '*\n' > "$root/.gitignore"
}
```

The self-ignoring `*` line makes the entire `.clone-wars/` directory
invisible to git — including the gitignore file itself. Idempotent;
only writes if absent (so user customizations to `.gitignore` are
preserved across subsequent invocations).

### Vestigial helpers preserved for v0.31.0, dropped in v0.32.0

`cw_repo_hash_for <cwd>` and `cw_topic_repo_hash` no longer participate
in state-path resolution. They stay defined with an added docstring
note flagging them as compatibility shims. Audit-and-remove sweep
deferred to v0.32.0 once we verify no remaining callers.

### Hook patch (closes cross-session bleed)

`hooks/user-prompt-submit-active-session.sh:12`:

```bash
# Before (v0.30.0):
STATE_ROOT="${CLONE_WARS_HOME:-$HOME/.clone-wars}/state"

# After (v0.31.0):
STATE_ROOT="$PWD/.clone-wars/state"
```

Now the hook only scans the current Claude Code session's project.
Different invoking-cwd → different `.clone-wars/` → no bleed.

The hook also exits silently if `$STATE_ROOT` doesn't exist (existing
v0.28.0 guard at line 13) — that path remains correct under the new
shape (no `.clone-wars/` in this project → no clone-wars work happening
here → exit silently).

### Args-file uniquification

Every `commands/*.md` directive writes user's `$ARGUMENTS` to a file
via the Write tool, then invokes the bin script with `--args-file`.
The path is currently stable (`_args/<command>.txt`) which collides
across parallel sessions.

v0.31.0 swaps to per-invocation `mktemp`:

```bash
# Before (v0.30.0) — directive's first Bash block:
ARGS_DIR="${CLONE_WARS_HOME:-$HOME/.clone-wars}/_args"
mkdir -p "$ARGS_DIR"
echo "$ARGS_DIR/<command>.txt"

# After (v0.31.0):
ARGS_DIR="$PWD/.clone-wars/_args"
mkdir -p "$ARGS_DIR"
mktemp -p "$ARGS_DIR" -t '<command>.XXXXXX'
```

Each directive caches the printed `mktemp` path to
`/tmp/cw-<command>-args-path.txt` (existing pattern for sharing state
across Bash tool invocations within a directive). Subsequent Bash
blocks in the same directive read the cached path; downstream bin
scripts read it via `--args-file`.

New lib helper in `lib/argsfile.sh`:

```bash
# cw_args_file_consume <path>
# Delete the args-file after a bin script reads it. Called at the end
# of every bin script's --args-file branch so /clone-wars/_args/ doesn't
# accumulate orphans across long sessions.
cw_args_file_consume() {
  local path="${1:-}"
  [[ -n "$path" && -f "$path" ]] || return 0
  rm -f "$path"
}
```

Bin scripts that already source `lib/argsfile.sh` (medic, consult-init,
deploy-init, deep-research-init, meditate-init, spawn, send) add the
call at end of arg parse.

- **Removing vestigial helpers** (`cw_topic_repo_hash`,

- **Ripping out `CLONE_WARS_HOME` env-var fallback.** Kept as a

## Risks

- **Existing user state orphans on upgrade.** Per the leave-alone
  migration choice. The user has 1 live deep-research session that will
  orphan. Documented as a known consequence; user explicitly accepted
  this in the brainstorm.
- **Sandbox-style tests that override `CLONE_WARS_HOME`.** Audit
  on 2026-05-14 found 96 test files using
  `export CLONE_WARS_HOME="$SANDBOX"` for sandbox isolation. Per the
  brainstorm decision, the env var is kept as a test/debug seam
  (production behavior is project-local; tests use the env var to
  redirect into a temp dir without `cd`'ing). Zero test rewrites
  needed for this aspect. The 6 tests that set `CW_TOPIC_REPO_CWD`
  for multi-init flows ARE rewritten — separate from the
  `CLONE_WARS_HOME` test seam.
- **Subshell `cd` changing `$PWD` mid-script.** `cw_state_root` evaluates
  `$PWD` at call time (when `CLONE_WARS_HOME` is unset). If a bin
  script does `(cd <subdir> && some_fn)` and `some_fn` calls
  `cw_state_root`, it gets the wrong root. Audit on 2026-05-14
  confirmed v0.30.0 is safe: `bin/deploy-init.sh:115` uses
  `(cd $TARGET_CWD && cw_deploy_branch_create ...)` — the inner
  function calls `git` and `log_error` only, never `cw_state_root`.
  `bin/deploy-init.sh:136`'s `cw_state_root` call is OUTSIDE the
  subshell. No other `cd` subshells in the codebase reach
  `cw_state_root`. Document the invariant in `cw_state_root`'s
  docstring: "Always called from the conductor's invocation cwd;
  never call inside a `cd` subshell."

## Rollback

Each change is independently revertible (single commit per item). If
the cutover breaks a real workflow, set
`CW_FORCE_GLOBAL_STATE=1` (env var kill switch — TO BE ADDED in the
implementation plan if surfaced during dogfood) before running slash
commands; the helper detects it and falls back to the v0.30.0 path
shape. Same pattern for the args-file change
(`CW_FORCE_STABLE_ARGS_PATH=1`). The hook patch has no kill switch
(the patch is the fix; reverting the hook restores the bleed).
