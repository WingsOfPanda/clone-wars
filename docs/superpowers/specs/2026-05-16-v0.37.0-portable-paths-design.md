# v0.37.0 — Eliminate hardcoded `/home/<user>/` paths from runtime plugin code + permanent regression guard

**Status:** Spec — pending review
**Date:** 2026-05-16
**Branch:** `feat/v0.37.0-portable-paths` (off `feat/v0.36.0-run-dir-pointers` — stacks on the open v0.36.0 PR)
**Audience:** writing-plans skill (next step) + dogfood reviewer

## Problem

`commands/deep-research.md` (42 sites) and `commands/deep-research-resume.md` (5 sites) hardcode the author's absolute install path `/home/liupan/CC/clone-wars` instead of using the canonical `${CLAUDE_PLUGIN_ROOT}/<path>` form that every other directive uses. Any user who installs the plugin at a different path — i.e. **every user except the author** — gets `source: file not found` on every Bash block of `/clone-wars:deep-research` and `/clone-wars:deep-research-resume`. The plugin's two most-recent flagship features (advisor autoresearch from v0.27.0+ and the Monitor-driven resume handler from v0.28.0+) are completely non-functional for end users.

### Why this slipped

- Author wrote deep-research.md initially via copy-paste from local-dev test invocations that used absolute paths.
- Every other directive (`consult/meditate/deploy/medic/list/teardown`) was authored using `$CLAUDE_PLUGIN_ROOT` and the pattern was never enforced.
- The plugin's test suite (`tests/run.sh`) sources `lib/*` via a test-local `PLUGIN_ROOT=$(cd .. && pwd)` discovery, so directive paths are never exercised by CI.
- Directives are interpreted by Claude in conversation, not run as code — there's no runtime resolution check during development.
- v0.36.0's T4 mechanically mirrored the existing bad pattern when adding 13 new `source` lines to deep-research.md, expanding the bug from ~29 sites to 42. (The user spotted this in PR review, which is what surfaced the broader pre-existing issue.)

## Goal

(A) Make the plugin functional when installed at any path. (B) Add a permanent regression-guard test that prevents this entire bug class from re-appearing in any future contribution.

## Approach summary

### Layer A — mechanical conversion (47 sites)

Two forms to handle (matching what `commands/consult.md`, `medic.md`, `list.md`, `teardown.md` already do):

**Inside quoted `source` lines:**
```bash
# Before:
source /home/liupan/CC/clone-wars/lib/state.sh
# After:
source "$CLAUDE_PLUGIN_ROOT/lib/state.sh"
```

**Inside unquoted `bin/*.sh` invocations:**
```bash
# Before:
/home/liupan/CC/clone-wars/bin/spawn.sh <commander> codex "$TOPIC"
# After:
"${CLAUDE_PLUGIN_ROOT}/bin/spawn.sh" <commander> codex "$TOPIC"
```

The replacement is mechanical: `s|/home/liupan/CC/clone-wars|${CLAUDE_PLUGIN_ROOT}|g` works for the unquoted bin invocations; quoted `source` lines need `${CLAUDE_PLUGIN_ROOT}` → `"$CLAUDE_PLUGIN_ROOT"` per the existing style (no `${...}` braces inside double-quoted source paths in the other directives — keeping consistent).

In practice the migration uses two `Edit replace_all=true` passes per file:

1. `source /home/liupan/CC/clone-wars/` → `source "$CLAUDE_PLUGIN_ROOT/`
   followed by appending `"` to the line (the source path is the last token on the line).
2. `/home/liupan/CC/clone-wars` → `${CLAUDE_PLUGIN_ROOT}`

Pass 1's two-step shape is needed because the existing `source` lines aren't already quoted. Simpler: do a single `replace_all` from `/home/liupan/CC/clone-wars` to `${CLAUDE_PLUGIN_ROOT}` and accept that `source` lines end up using `${CLAUDE_PLUGIN_ROOT}` (braces) instead of `$CLAUDE_PLUGIN_ROOT` (no braces). Both are valid bash; the brace form is even slightly more robust. The lint doesn't care which form.

**Recommended:** one global replace per file. Yields uniform `${CLAUDE_PLUGIN_ROOT}` throughout. Two diff-sites per file = trivial review.

### Layer B — permanent regression-guard test

New `tests/test_no_hardcoded_paths.sh` (no `if [[ "$PV" != "0.37.0" ]]; then pass skip ...` gate — runs forever).

```bash
#!/usr/bin/env bash
# tests/test_no_hardcoded_paths.sh
# Permanent lint: runtime plugin code must NOT hardcode /home/<user>/ paths.
# Plugin gets installed at user-controlled paths via /plugin install;
# use ${CLAUDE_PLUGIN_ROOT}/<path> (Claude Code docs: plugin-install-path)
# OR, in bin/*.sh entry scripts, the resolution chain:
#   PLUGIN_ROOT="${CLAUDE_PLUGIN_ROOT:-$(cd "$(dirname "${BASH_SOURCE[0]}")/.." && pwd)}"
set -euo pipefail
cd "$(dirname "$0")"
source lib/assert.sh
PLUGIN_ROOT="$(cd .. && pwd)"

# Scan only runtime plugin code. tests/, docs/, CLAUDE.md, README excluded:
#   - tests/ can have legitimate fixture paths (test_deploy_v07_dogfood,
#     test_deploy_dag_capwords_path)
#   - docs/superpowers/{specs,plans}/ cite historical dogfood scenarios
#   - CLAUDE.md changelog references past dogfood paths
LINT_DIRS=(commands bin lib hooks config .claude-plugin)
FAILED=0
for d in "${LINT_DIRS[@]}"; do
  [[ -d "$PLUGIN_ROOT/$d" ]] || continue
  # Word-boundary regex: /home/<lowercase-username>/ — catches any
  # contributor's home dir, not just /home/liupan.
  hits=$(grep -rnE "/home/[a-z][a-zA-Z0-9_-]*/" "$PLUGIN_ROOT/$d" 2>/dev/null || true)
  if [[ -n "$hits" ]]; then
    echo "FAIL: hardcoded /home/<user>/ path(s) in $d/:" >&2
    echo "$hits" >&2
    FAILED=1
  fi
done

if (( FAILED != 0 )); then
  echo "" >&2
  echo "Fix: use \${CLAUDE_PLUGIN_ROOT}/<path> instead." >&2
  echo "For bin/*.sh entry scripts, use the standard resolution chain:" >&2
  echo "  PLUGIN_ROOT=\"\${CLAUDE_PLUGIN_ROOT:-\$(cd \"\$(dirname \"\${BASH_SOURCE[0]}\")/..\" && pwd)}\"" >&2
  exit 1
fi
pass "no hardcoded /home/<user>/ paths in: ${LINT_DIRS[*]}"
echo "test_no_hardcoded_paths: lint passed"
```

**Word-bounded regex `/home/[a-z][a-zA-Z0-9_-]*/`** catches any future contributor's home dir, not just `/home/liupan`. Slightly stricter than literal-string match. False-positive risk in runtime code: near-zero (no legitimate `/home/<user>/` should appear in shipped plugin code; if a legitimate case ever emerges, the lint message tells the contributor how to fix it).

### Why `tests/` is excluded from the lint

Two existing files in `tests/` legitimately reference `/home/liupan/` strings:

1. `test_deploy_v07_dogfood.sh:58,67` — `echo` statements telling the user where to `cd` for dogfood walkthroughs. They're prose for human readers, not paths the test executes.
2. `test_deploy_dag_capwords_path.sh:28,29` — DAG-line parser unit test that synthesizes an `ARS-` slug path. Test fixture data, not a runtime path.

Neither is a portability bug. The lint scope is shipped runtime code only.

## Out of scope

- **`${CLONE_WARS_HOME}` literal usage in directives** — should be `$(cw_state_root)` per v0.31.0. Different bug class; v0.31.0 already migrated most of it. Defer to v0.38 if observed.
- **CI integration** — there's no CI in the repo today. The `tests/run.sh` suite is run manually. Adding the lint to a CI job is a separate v0.38+ concern.
- **Author-style guide / CONTRIBUTING.md** — repo has no contributors besides me yet; documenting the pattern in CONTRIBUTING.md is premature. The lint itself is the documentation (when it fails, the error message tells you the canonical pattern).
- **Documentation prose** (specs, plans, CLAUDE.md changelog) — legitimate historical references; not runtime code.
