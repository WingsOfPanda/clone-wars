# v0.20.1 Deploy Multi-Repo Fixes — Design Doc

**Status:** approved 2026-05-09. Implementation pending. Patch on top of v0.20.0 (PR #71 — DO NOT MERGE until this lands).

## Problem

A `plugin-dev:skill-reviewer` pass on PR #71 (v0.20.0 multi-repo deploy)
found 6 P0 wiring defects + 4 P1 issues + 3 P2 nice-to-haves. The
multi-repo path is **unreachable** as currently wired (P0-1) and even
if reached would fail at the wait stage (P0-2) and silently no-op the
"feels unsafe" trigger (P0-3).

The single-repo path is byte-equal to v0.19.0 — backwards-compat is
preserved, so this patch only touches the multi-repo code surface.

Skill-reviewer's findings (verbatim summary):

1. **P0-1 Wiring**: `bin/deploy-init.sh` never invokes `deploy-dag-parse.sh`
   or `deploy-multi-init.sh`. The directive Step 3a's defensive check
   (`[[ -f dag-waves.txt ]]`) will fail on first invocation. Multi-repo
   path is unreachable.
2. **P0-2 Wait helper**: `bin/deploy-turn-wait.sh` is hard-coded to
   `cody/codex/round-aware`. Step 3b says "K parallel deploy-turn-wait.sh"
   but that script can't work for multi-trooper waves.
3. **P0-3 Branch-base capture**: Step 3c reads `$cmdr-branch-base.sha`
   per trooper, but no script writes it. The SHARED_PATHS detection
   (third "feels unsafe" trigger) silently no-ops.
4. **P0-4 multi-init signature drift**: takes `<topic>` only; derives
   sub-repo cwd from `$PWD`. Spec said `<design-doc-path> <topic>`.
   Brittle: relies on `$PWD == hub root`.
5. **P0-5 DAG-unit inbox prompt placeholders**: `<slug>`, `<N>`,
   `<total>`, `<upstream-slug-list>` are literal in directive — fresh
   Claude may pass them verbatim.
6. **P0-6 Bugs collection contract**: Step 3c finds bugs but doesn't
   write them; Step 3d's "for (sub-repo, bug)" loop has no source.
7. **P1-7** `Skill` missing from `allowed-tools`.
8. **P1-8** `WAVE_GROUPS` array computed but the outer wave loop never
   written.
9. **P1-9** Source-defaulting block uses `log_error` before `lib/log.sh`
   sourced.
10. **P1-10** `TOPIC_DIR=` string-construction duplicates
    `cw_deploy_topic_dir`; Stage 2 abort uses destructive `rm -rf
    "$TOPIC_DIR"` instead of `bin/deploy-archive.sh`.
11. **P2-11** No per-wave progress logging.
12. **P2-12** "Wave" concept never explicitly defined.
13. **P2-13** Step 4 final summary doesn't iterate troopers.txt for
    per-sub-repo commit counts.

## Goal

Fix all 13 findings in a single v0.20.1 patch on
`feat/v0.20.0-deploy-multi-repo-dag` (PR #71). After this patch:

- The multi-repo path runs end-to-end on a 3-sub-repo design doc:
  init parses DAG + assigns commanders + captures branch bases →
  preflight allocates panes → directive walks waves with K parallel
  spawn+wave-wait pairs → conductor reads multi-verify-bugs.txt to
  drive fix-loop → teardown cleans orphans.
- The fresh-Claude error surface in Step 3b is eliminated by a lib
  helper that returns a fully-resolved DAG-unit prompt.
- The "feels unsafe" Step 3c trigger fires correctly because
  per-cmdr branch-base.sha is captured at multi-init time (before any
  trooper modifies the working tree).
- Stage 2 abort preserves state via archive (not destruction).
- Single-repo deploy is byte-equal to v0.20.0 (and therefore v0.19.0).

## Components

### NEW `bin/deploy-wave-wait.sh`

Signature: `deploy-wave-wait.sh <topic> <commander> <provider>` (rc=0
on terminal event, rc≠0 only on bad args / missing state).

Behavior — mirrors `bin/consult-research-wait.sh`:

1. Validate args + topic regex.
2. Resolve trooper outbox path: `cw_outbox_path "$commander" "$provider"
   "$topic"`.
3. Resolve state file: `$ART_DIR/wave-<commander>.txt`.
4. Resolve done sentinel: `$ART_DIR/wave-<commander>.done`.
5. Loop reading `cw_outbox_wait_since` (or equivalent — same primitive
   consult-research-wait.sh uses) with timeout
   `${CW_DEPLOY_TURN_TIMEOUT:-14400}` (4hr default, matches existing
   single-repo Step 1).
6. On terminal event:
   - `{event:"done"}` → write `TS=ok` + `EVENT=done` to state file
   - `{event:"error",reason:"..."}` → `TS=failed` + `REASON=...`
   - timeout → `TS=timeout`
7. `touch <done-sentinel>` to signal the harness.
8. `exit 0` (the harness reads state file + sentinel after notification).

Exit codes:
- `0` — terminal event reached (state file populated)
- `2` — bad args
- `1` — fatal infra error (state-dir unwritable, etc.)

### NEW `cw_deploy_build_dag_unit_prompt` in `lib/deploy.sh`

Signature: `cw_deploy_build_dag_unit_prompt <slug> <design-path>
<step> <total> <upstream-csv>`. Echoes a fully-resolved heredoc to
stdout. rc=2 on bad args.

Mirrors existing `cw_deploy_build_turn_prompt_round1` (`lib/deploy.sh`
~line 236). Heredoc structure:

```
Read <design-path>. Your sub-repo is "<slug>".

Multi-repo design docs use `### <slug>` subsection headings inside the
Architecture and Components sections — focus on the subsections matching
your slug. The DAG context (Step <step> of <total>) is in the
"## Execution DAG" section; you depend on: <upstream-csv|none>.

Run the full superpowers ceremony for your sub-repo:
1. superpowers:writing-plans — produce an implementation plan from the
   design-doc's slice for "<slug>", saved to
   docs/superpowers/plans/YYYY-MM-DD-<slug>-plan.md
2. superpowers:subagent-driven-development — execute the plan task-by-
   task, two-stage review per task
3. superpowers:verification-before-completion — confirm tests pass,
   diff matches the plan, no half-finished work, before reporting done

Report status via outbox: emit {"event":"done"} when all tasks are
complete and verified. Emit {"event":"error", "reason":"..."} on any
unrecoverable failure.
END_OF_INSTRUCTION
```

When `<upstream-csv>` is empty or "none", emit `you depend on: none
(this is a wave-1 / root sub-repo).` instead. The placeholder
substitution is the helper's responsibility — directive callers just
pass values + interpolate the result into `cw_inbox_write`.

### MODIFIED `bin/deploy-multi-init.sh`

Two changes:

1. **Add optional `<hub-cwd>` second positional arg.** Defaults to
   `$PWD` if absent (preserves test/dev ergonomics). When passed,
   uses `<hub-cwd>` instead of `$PWD` for sub-repo resolution. Lets
   `bin/deploy-init.sh` pass `$TARGET_CWD` explicitly so the script
   doesn't depend on the caller's cwd. New usage:
   `deploy-multi-init.sh <topic> [<hub-cwd>]`.
2. **Per-cmdr branch-base capture.** Inside the existing
   `for repo in "${REPOS_ORDERED[@]}"` loop, after assigning
   `COMMANDER`, write:
   ```bash
   git -C "$CWD" rev-parse HEAD > "$ART_DIR/${COMMANDER}-branch-base.sha" \
     || { log_error "rev-parse HEAD failed for $CWD"; exit 1; }
   ```
   `$CWD` is already in scope (the per-repo absolute path). The capture
   happens BEFORE any trooper spawns or modifies the working tree, so
   `<cmdr>-branch-base.sha` is the pristine baseline for Step 3c's
   shared-paths diff.

### MODIFIED `bin/deploy-init.sh`

After the existing `routing.txt` write (the v0.20.0 block we added in
Task 7 of the v0.20.0 plan), append a multi-repo conditional:

```bash
if [[ "$ROUTING" == "multi-repo" ]]; then
```

#### Step 3b — "wave" definition (one-line, top of section)

> A **wave** is a set of sub-repos with no remaining unsatisfied
> dependencies that can run in parallel. `bin/deploy-dag-parse.sh`

## Failure modes

| Stage | Failure | Behavior |
|---|---|---|
| `deploy-init.sh` | `deploy-dag-parse.sh` fails | `log_error "deploy-dag-parse.sh failed"` + `exit 1` |
| `deploy-init.sh` | `deploy-multi-init.sh` fails | `log_error "deploy-multi-init.sh failed"` + `exit 1` |
| `deploy-multi-init.sh` | `git -C "$CWD" rev-parse HEAD` fails | `log_error "rev-parse HEAD failed for $CWD"` + `exit 1` |
| `deploy-multi-init.sh` | hub-cwd arg invalid (not absolute, not a dir) | `log_error "hub-cwd must be absolute existing dir"` + `exit 1` |
| `deploy-wave-wait.sh` | trooper outbox missing | `TS=failed` + `REASON=outbox missing` + done sentinel |
| `deploy-wave-wait.sh` | timeout (4hr default) | `TS=timeout` + done sentinel |
| `cw_deploy_build_dag_unit_prompt` | missing arg | rc=2 + log_error + return |
| Stage 2 abort (Step 3b) | partial-success "Abort all" picked | `deploy-teardown.sh` + `deploy-archive.sh` (NOT `rm -rf`) |
| Step 3c | bugs found | write to `multi-verify-bugs.txt`; Step 3d reads |
| Step 3d | `multi-verify-bugs.txt` absent or empty | log info + skip Step 3d entirely |
| Step 3d | per-repo MAX_FIX_ROUNDS reached | AskUserQuestion (give up / continue / escalate) — unchanged from v0.20.0 |
