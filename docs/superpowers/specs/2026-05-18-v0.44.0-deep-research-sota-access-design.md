# /clone-wars:deep-research — SOTA web access (v0.44.0)

**Status:** Locked 2026-05-18. Implementation plan to follow under
`docs/superpowers/plans/`.

**Predecessor signal:** post-v0.43.0 dogfood reflection — Yoda and
troopers both have web tools available today (Yoda: WebSearch + Tavily;
troopers: net access permitted-by-default since v0.27.0), but the
directive only mentions WebSearch+Tavily as *optional* in Phase 1
step 2 and dispatch prompts say nothing about web access. The result
is cold-started experiments that re-discover well-known SOTA
references the slow way. This spec wires explicit, well-affordanced
web access into both surfaces.

## Goal

Give the deep-research conductor (Yoda) and troopers (codex) explicit,
well-affordanced web access for SOTA discovery. Yoda runs one
dual-search SOTA sweep right after the metric is locked, writes a
shared `_deep-research/sota.md`. Every per-experiment dispatch inlines
sota.md into the trooper's `prompt.md` and grants explicit liberty to
web-search when stuck or before scale-up. Zero behavioral change for
runs that don't trigger Phase 1.5 (no such mode in v0.44.0; the sweep
always runs).

## Scope

**In scope:**

1. New `### Phase 1.5 — SOTA sweep` section in `commands/deep-research.md`,
   inserted between Phase 1 (metric lock) and Phase 2 (preflight). One
   round of dual-search (WebSearch + Tavily × 2 queries each = 4
   parallel calls), merged into a curated reference table written to
   `_deep-research/sota.md`.
2. New `cw_deep_research_format_sota_block` helper in
   `lib/deep-research.sh` — K=V heredoc input, Markdown table output,
   matches the existing `format_metric_block` pattern.
3. Trooper-side affordance in `bin/deep-research-experiment-send.sh`:
   `prompt.md` template gains a `## Reference: SOTA` section inlining
   sota.md verbatim, plus a two-liner explicitly granting web-search
   liberty when the trooper hits a plateau or before scale-up.
4. Permanent directive lint extended: `commands/deep-research.md` must
   carry the new `### Phase 1.5 — SOTA sweep` heading and must mention
   the two-liner verbatim in the dispatch payload template
   documentation.
5. Version-stamped static-wiring lock at 0.44.0 (skip-guarded until
   the version bump in T6).

**Explicitly out of scope (postponed):**

- **Mid-session sota.md refresh.** Write-once at Phase 1.5 only. If
  troopers find relevant new references mid-session, they record them
  in their own `notes.md`. Yoda does not mutate `sota.md` after the
  initial write. Candidate for v0.45.0 if dogfood signal demands it.
- **Per-trooper sota.md** (lane-specific reference sets). One shared
  reference for v0.44.0; per-lane refinement is over-engineered until
  dogfood shows a need.
- **Web-access opt-out flag.** Trooper net access has been
  permitted-by-default since v0.27.0; if a user needs hard isolation,
  the correct surface is OS-level (firewall, network namespace), not
  a skill flag. Documented as a security note in Phase 1.5.
- **Inter-trooper visibility** (Item 5 from the 2026-05-17 punch list,
  planned next-major per [[project_inter_trooper_messaging]]). Still
  prioritized next after v0.44.0.

## Architecture / contract

The deep-research turn structure gains one new sub-phase between
metric lock and preflight. Phase numbering preserved (Phase 2-7
unchanged) so existing test fixtures keep passing.

```
Phase 0 — args + init
Phase 1 — metric discussion → metric.md
Phase 1.5 — SOTA sweep (NEW) → sota.md
Phase 2 — preflight (roster + time limit)
Phase 3 — spawn roster
Phase 4 — per-trooper turn loop (4.a initial, handler in resume.md)
Phase 5 — synthesis
Phase 6 — teardown + archive
Phase 7 — present final doc
```

Phase 1.5 is unconditional: it always runs once metric.md is locked.
The dual-search cost is ~30-90s wall-clock and four web calls (two per
engine) per session — negligible against any time-budget the user
might pick in Phase 2.

`sota.md` is consumed in exactly two places:
- Yoda may reference it when composing per-experiment direction notes.
- `bin/deep-research-experiment-send.sh` inlines its contents into
  every `prompt.md` it writes.

After Phase 1.5 commits sota.md, the file is read-only for the
remainder of the session. The teardown flow archives it alongside
the other artifacts.

## Components

### Lane A — `cw_deep_research_format_sota_block` helper

**File:** `lib/deep-research.sh` (one new function appended; ~40 lines).

The helper accepts K=V pairs on stdin and emits the curated sota.md
Markdown. Mirrors the `cw_deep_research_format_metric_block` pattern
already in the file. Input grammar:

```
topic=<one-line topic phrase>
metric=<from metric.md primary_metric>
sweep_date=<ISO-8601 UTC date>
queries=<comma-separated list of queries Yoda actually fired>
ref_1=<approach family>|<best known result>|<constraint compliance>|<source URL>|<notes>
ref_2=...
...up to ref_7
```

Output (verbatim Markdown):

```
# SOTA reference — <topic>

> **Sweep date:** <ISO-8601 UTC>
> **Optimizing for:** <metric>
> **Queries fired:** <comma-list>

| Approach family | Best known | Constraint compliance | Source | Notes |
|---|---|---|---|---|
| <family> | <result> | <fits / over by N> | <URL> | <notes> |
| ...                                                              |
```

Empty `ref_N` rows are skipped. Trailing newline appended. Output is
deterministic for a given input — easy to unit test.

The helper does **not** call WebSearch/Tavily itself; Yoda fires the
searches via tool calls and constructs the K=V input from merged
results. Keeps the bash helper pure and testable.

### Lane B — Phase 1.5 directive section

**File:** `commands/deep-research.md` (new section ~80 lines, inserted
between Phase 1 and Phase 2).

Section structure:

```
### Phase 1.5 — SOTA sweep

Set task `1.5` → `in_progress`.

1. Read metric.md to pick search terms.
2. Fire dual-search (single tool-call message, 4 calls parallel):
   - WebSearch:    "SOTA <primary_metric> <topic-domain>"
   - WebSearch:    "<topic-domain> under <hard_constraint>"
   - Tavily:       "SOTA <primary_metric> <topic-domain>"
   - Tavily:       "<topic-domain> under <hard_constraint>"
3. Merge + dedupe by URL; prefer Tavily's longer snippets where they
   overlap (per global dual-search rule).
4. Pick ≤7 most relevant references — variety over completeness;
   prefer one row per approach family.
5. Write sota.md via cw_deep_research_format_sota_block.

[Bash block: K=V heredoc invocation]

Set task `1.5` → `completed`.

#### Security note

Web access in this phase relies on:
- Yoda's WebSearch + mcp__tavily__tavily-search tool availability
  (Claude Code default in 2026).
- Trooper-side net access, permitted-by-default since v0.27.0
  (sandboxing is honor-system; not enforced).

If you need hard-block, do it at OS/firewall/network-namespace level
before invoking /clone-wars:deep-research. The skill does not expose
an opt-out flag.
```

The TaskCreate row count in the directive's "Task list" section
increases from 8 to 9 (insert `1.5 SOTA sweep [yoda]`).

### Lane C — Trooper dispatch prompt template

**File:** `bin/deep-research-experiment-send.sh` (one new section
added to the prompt.md heredoc; ~15 lines of payload + 3 lines of
shell).

The dispatch script reads `$ART_DIR/sota.md` (if present) and inlines
it into the trooper's `prompt.md` under a new section:

```
## Reference: SOTA

<inlined contents of sota.md>

### Web search affordance

Consult this reference before starting. Web search (curl / pip
install / arXiv / HuggingFace / etc.) is allowed when you hit a
plateau or before scaling up. Record any consulted source in
`notes.md` under a `## Sources consulted` heading.
```

If `sota.md` is absent (e.g., legacy session, manual setup, or a
caller invokes experiment-send.sh outside the directive flow), the
script skips the section entirely — no error, no warning. Keeps
back-compat with existing tests that don't seed sota.md.

The inlined text is part of every per-experiment dispatch. Cost is
~1-2 KB per prompt.md — negligible against the trooper's existing
context allocation.

## Data structures

### sota.md format

Always single file at `$ART_DIR/sota.md` (where `$ART_DIR` is
`$CLONE_WARS_HOME/state/<repo-hash>/<topic>/_deep-research`). Written
atomically (tmp + mv pattern) by Phase 1.5. Read-only for the rest of
the session.

Maximum reference count: 7 rows (enough to span major approach
families without bloating every dispatched prompt). If Yoda's merged
result set exceeds 7, the dispatch decision (which to keep) is
documented in the directive prose: prefer variety across approach
families over depth within one family.

Frontmatter callout block:

```markdown
> **Sweep date:** 2026-05-18T09:42:11Z
> **Optimizing for:** accuracy
> **Queries fired:** SOTA accuracy MNIST, MNIST under 100k params
```

Reference table columns:

| Column | Type | Example |
|---|---|---|
| Approach family | short label | "Depthwise CNN" |
| Best known | metric value | "99.84%" |
| Constraint compliance | "fits" / "over by N" | "over by 60k params" |
| Source | URL | "https://arxiv.org/abs/..." |
| Notes | one-line | "Caps-Net variant; needs param shrink" |

### prompt.md addition

Inserted between the existing experiment-context section and the
existing direction-from-Yoda section. Position fixed: every dispatch
puts the SOTA reference in the same place so troopers can locate it
predictably.

## Error handling

- **Yoda's dual-search returns zero usable results.** Phase 1.5 still
  writes sota.md with the frontmatter and an empty table, with a Notes
  row that says "sweep returned no usable references; trooper-side
  web search remains available." Downstream prompt.md inlining still
  works (empty-table-but-present is fine).
- **sota.md present but malformed.** experiment-send.sh inlines
  verbatim — no parsing on the bash side, so malformed content is
  passed through to the trooper. Trade-off: simpler code, but
  Phase 1.5's formatter is the only safe writer.
- **sota.md missing at dispatch time** (legacy session, test
  harness without Phase 1.5 setup). Dispatch skips the SOTA section
  entirely, no error. Pre-v0.44.0 tests keep passing.
- **WebSearch or Tavily tool unavailable** (e.g., MCP server
  disconnected). Yoda runs whichever side is available; if both fail,
  proceed with the empty-table path above. Do not block the session
  on missing web tools — that's a user-environment problem, not a
  skill defect.

## Risks

- **Dual-search returns garbage / paywalled / off-topic.** Yoda
  curates aggressively (≤7 rows, variety bias). Cost is bounded at 4
  web calls per session, regardless of result quality.
- **Trooper context bloat.** sota.md is capped at ~1-2 KB by the 7-row
  limit. Inlining into every prompt.md adds ~1-2 KB to each dispatch.
  Trooper context allocation is 200K+ tokens — this is negligible.
- **Drift between Phase 1.5 and trooper expectations.** Directive
  lint pins the "## Reference: SOTA" section name; experiment-send.sh
  writes exactly that. Any future rename has to happen in both places
  or the lint catches it.
- **User wants Phase 1.5 to skip on tight time budgets.** Surfaced in
  brainstorm; rejected. ~30-90s is negligible. If dogfood demonstrates
  otherwise, v0.45.0 can add `--skip-sota-sweep` opt-out.
