# v0.38.0 — Split state-root concepts: per-machine config vs per-project state

**Status:** Spec — pending review
**Date:** 2026-05-16
**Branch:** `feat/v0.38.0-state-root-split` (off main, post-v0.37.0 merge)
**Audience:** writing-plans skill (next step) + dogfood reviewer
**Predecessor:** v0.31.0 (project-local state relocation) — this completes the split that v0.31.0 partially landed.

## Problem

v0.31.0 collapsed two distinct concepts under a single `cw_state_root` helper:

- **Per-MACHINE config** — what binaries are available (`providers-available.txt`), what shipping defaults look like (`contracts.yaml`, `commanders.yaml`), the user's roster choice (`providers-active.txt`). One per install; survives across all projects.
- **Per-PROJECT state** — this project's ongoing `/consult` topics, `/deploy` runs, `/deep-research` sessions, `/meditate` landscapes. One per repo; meaningful only inside that repo.

Before v0.31.0, both lived globally under `~/.clone-wars/`. v0.31.0 moved the whole tree project-local for the per-project state case (which was right) but inadvertently dragged the per-machine config along (which was wrong). Various scripts then inconsistently bypass `cw_state_root` using literal `${CLONE_WARS_HOME:-$HOME/.clone-wars}` because the authors knew the data was per-machine — but didn't formalize a separate helper.

### Observed symptom (user-reported)

Running `/clone-wars:medic` writes `<project>/.clone-wars/providers-available.txt`, then reads `~/.clone-wars/providers-available.txt` in Step A (different code path). On any fresh install, this means: medic detects providers correctly, writes the list project-local, then the interactive roster picker silently logs `warn: providers-available.txt not found; skipping trooper selection` because it looks in the wrong place. Roster picker never fires; consult silently uses defaults.

The user only saw the picker work because they had a pre-v0.31.0 `~/.clone-wars/providers-available.txt` left over from earlier installs. **Fresh-install users get a broken medic→consult chain.** Same class of break affects `/clone-wars:deep-research-init`'s codex-presence gate (reads `~/.clone-wars/providers-available.txt` even though medic now writes project-local) and `bin/deploy-multi-init.sh`'s `commanders.yaml` lookup (reads global, but `lib/commanders.sh:cw_commanders_path` looks project-local).

## Goal

(A) Restore the broken medic→consult/deploy/deep-research chain for fresh installs. (B) Add an explicit `cw_global_state_root` helper so per-machine config has a single source of truth. (C) Stop the workaround pattern where authors bypass `cw_state_root` with literal global paths because no helper exists for that case. (D) Add a permanent lint that enforces the discipline.

## Approach summary

### Layer A — new helpers in `lib/state.sh`

```bash
# cw_global_state_root — per-MACHINE config root.
# Always resolves to ${CLONE_WARS_HOME:-$HOME/.clone-wars}, regardless of $PWD.
# Use for config that is per-install, not per-project: medic's
# providers-{active,available}.txt, contracts.yaml, commanders.yaml,
# the archive/ subtree.
#
# Contrast with cw_state_root (per-PROJECT state at $PWD/.clone-wars/)
# for /consult, /deploy, /deep-research, /meditate per-topic work.
cw_global_state_root() {
  printf '%s\n' "${CLONE_WARS_HOME:-$HOME/.clone-wars}"
}

# cw_global_state_ensure — like cw_state_ensure but for the global root.
# Auto-creates .gitignore = '*' on first call (defensive; ~/.clone-wars/
# is not usually under git, but covers the case where a user nests it
# inside an unrelated repo).
cw_global_state_ensure() {
  local root; root=$(cw_global_state_root)
  mkdir -p "$root"
  [[ -f "$root/.gitignore" ]] || printf '*\n' > "$root/.gitignore" 2>/dev/null || true
}
```

### Layer B — migrate the per-machine paths

12 call sites across `lib/`, `bin/`, `commands/`. Mechanical conversion (`cw_state_root` → `cw_global_state_root`; literal `${CLONE_WARS_HOME:-$HOME/.clone-wars}` → `$(cw_global_state_root)`). Layer-by-layer order:

1. **`lib/state.sh`** — add helpers; rewrite `cw_active_providers_path` to call `cw_global_state_root`
2. **`lib/contracts.sh`** — `cw_contracts_path` calls `cw_global_state_root`
3. **`lib/commanders.sh`** — `cw_commanders_path` calls `cw_global_state_root`
4. **`lib/ipc.sh`** — archive base calls `cw_global_state_root`
5. **`bin/medic.sh`** — `state_root=$(cw_global_state_root)`; remove now-unused `cw_state_ensure` writability check (replace with `cw_global_state_ensure`)
6. **`bin/consult-drilldown.sh`** — contracts.yaml read via `cw_global_state_root`
7. **`bin/{consult,deploy}-archive.sh`** — `ARCHIVE_BASE` calls `cw_global_state_root`
8. **`bin/deploy-multi-init.sh`** — literal → `cw_global_state_root`
9. **`bin/deep-research-init.sh`** + 4 sibling scripts — literal → `cw_global_state_root`
10. **`commands/medic.md`** — drop Step 1 args-file boilerplate entirely (medic takes no args); literal `${CLONE_WARS_HOME:-...}` → `$(cw_global_state_root)`
11. **`commands/meditate.md:478`** — archive literal → `$(cw_global_state_root)`
12. **`commands/teardown.md:20`** — docs already say `$CLONE_WARS_HOME/archive/...`; clarify "global path; survives v0.31.0's project-local state move"

### Layer C — drop medic.md Step 1 (args-file boilerplate)

medic takes no arguments. The `_args/` staging pattern is a v0.31.0 boilerplate that applied to every directive uniformly. For medic specifically it's dead weight that pollutes `<project>/.clone-wars/_args/` with empty files. Drop steps 1, 2, 3 entirely from `commands/medic.md`; new Step 1 is just `"${CLAUDE_PLUGIN_ROOT}/bin/medic.sh"` (no `--args-file` flag).

`bin/medic.sh` keeps accepting `--args-file <path>` as a no-op for back-compat with anyone who upgraded mid-session (or for explicit invocation outside the directive). The script's `cw_args_file_consume` call already handles the empty-args case gracefully.

### Layer D — permanent lint `tests/test_state_root_discipline.sh`

Counterpart to v0.37.0's `test_no_hardcoded_paths.sh`. Runs every release. Enforces:


Failure mode: prints the offending file:line + suggested replacement. No version skip-guard.

## Out of scope

- **Migration code** — leave-alone, per v0.31.0 precedent. Users on the v0.31-v0.37 era who relied on project-local medic state need one `/medic` re-run after v0.38.0 upgrade. CLAUDE.md status row says this verbatim.
- **Renaming `cw_state_root`** — too invasive; `cw_state_root` keeps its "per-project" semantics. The new helper sits alongside.
- **Removing `${CLONE_WARS_HOME:-$HOME/.clone-wars}` env-var seam** — kept inside `lib/state.sh`'s helpers for tests/debug. Outside `lib/state.sh`, the literal is banned by the permanent lint.
- **Auto-detecting + cleaning up the v0.31-v0.37 project-local stragglers** — out of scope. They become inert; user can `rm -rf <project>/.clone-wars/{providers-*,contracts.yaml,commanders.yaml}` manually if they want to.
- **Migrating existing project-local archive/ subtrees** — out of scope. Old archive dirs at `<project>/.clone-wars/archive/` stay where they are; v0.38.0+ writes new archives to `~/.clone-wars/archive/`. Users can manually merge if they want.

## Open questions for reviewer

1. **`cw_global_state_ensure` `.gitignore` auto-write** — `~/.clone-wars/` isn't usually under git but the defensive write costs nothing. Worth keeping? Yes.
2. **`bin/medic.sh` accepting (and ignoring) `--args-file` for back-compat** — covers users who upgrade mid-session and their cached directive still passes `--args-file`. Cleaner alternative: hard-remove the flag and let the cached call fail. Recommendation: keep no-op for one release, drop in v0.39.
3. **Lint coverage for `${CLONE_WARS_HOME:-$HOME/.clone-wars}` literal outside `lib/state.sh`** — strict; will fail if anyone manually expands the env-var seam. Acceptable since the helper is now available.
