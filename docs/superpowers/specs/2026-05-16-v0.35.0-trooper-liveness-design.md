# v0.35.0 — Per-provider timeout multiplier + mtime liveness probe in `cw_consult_wait`

**Status:** Spec — pending review
**Date:** 2026-05-16
**Branch:** `feat/v0.35.0-trooper-liveness`
**Audience:** writing-plans skill (next step) + dogfood reviewer

## Problem

`cw_consult_wait` (`lib/consult-wait.sh`) writes `<state_key>=timeout` to the
per-trooper state file purely on wall-clock elapsed past
`cw_consult_timeout(<kind>)` — defaults `research=600`, `verify=300`,
`adversary=600`, `experiment=1800`. It never asks "is the trooper still
making progress?"

Symptom observed during the most recent `/clone-wars:consult` run:

- Wolffe (opencode / DeepSeek V4 Pro) recorded `FS=timeout`.
- `findings-wolffe.md` was 18 KB (active research output).
- `tmux` showed the wolffe pane still working.

opencode is consistently slower than codex/claude in wall-clock terms.
v0.13.0 already partially acknowledged this (`bootstrap_sleep_s 8 → 20`
on the cold-start path), but the per-phase research/verify/adversary
budgets still treat all providers identically.

## Goal

Stop false-positive `*S=timeout` writes for slow troopers without
sacrificing the safety-net for genuinely wedged ones. Two complementary
layers, both opt-in via configuration with sensible defaults.

## Approach summary

| Layer | What | Where |
|---|---|---|
| A | Per-provider `timeout_multiplier` field (default `1.0`); opencode ships `2.5` | `config/contracts.yaml` + new helper in `lib/contracts.sh` |
| B | mtime-based liveness probe before declaring timeout; extends deadline if outbox was touched recently, capped by hard ceiling | `lib/consult-wait.sh` |

Layer A alone closes the simple case (opencode just needs more time).
Layer B closes the harder case (a slow trooper that bursts past even
its multiplied baseline but is provably still writing). Together they
cover both axes.

## Approaches considered

1. **Multiplier + mtime liveness probe (chosen)** — small contracts.yaml
   change + ~15-line addition to consult-wait's poll loop. Universal
   across providers, parallels the proven `bin/deep-research-monitor.sh`
   pattern (which already uses outbox mtime against `PROBE_S` /
   `STUCK_S` thresholds).
2. **Trooper-emitted heartbeat events** — change `identity-template.md`
   to require periodic `{event:"heartbeat"}` JSONL. More accurate, but
   troopers don't always follow instructions. opencode in particular
   has skipped emit obligations before. Higher risk for the same
   outcome. **Deferred to a future brainstorm if mtime proves
   insufficient.**
3. **Per-provider absolute timeout overrides** (e.g. `consult.research_timeout_s.opencode: 1500`)
   — simpler math, noisier config (4 timeout keys × 4 providers up to
   16 fields). Multiplier wins on config discipline. **Rejected.**

## Layer A — per-provider timeout multiplier

### Config change (`config/contracts.yaml`)

Add an optional `timeout_multiplier` field per provider row. Default
`1.0` when absent (current behavior preserved for installs that don't
re-copy the shipped contracts.yaml).

```yaml
codex:
  ...                  # implicit multiplier 1.0

opencode:
  ...
  timeout_multiplier: 2.5   # DeepSeek V4 Pro is wall-clock slower
```

Three other providers (codex, gemini, claude) get the implicit `1.0`
and ship without the field. `gemini` and `claude` can grow their own
multiplier later if dogfood data shows they need one.

### New helper (`lib/contracts.sh`)

```bash
# cw_contract_timeout_multiplier <provider>
# Print the timeout_multiplier field for <provider>. Default "1.0" when
# the field is absent or malformed (non-positive, non-numeric).
cw_contract_timeout_multiplier() {
  local provider="$1" v
  v=$(_cw_contract_field "$provider" timeout_multiplier)
  # Accept positive int (1, 2) or simple decimal (1.5, 2.5).
  if [[ -n "$v" ]] && [[ "$v" =~ ^[0-9]+(\.[0-9]+)?$ ]] && \
     awk -v x="$v" 'BEGIN { exit !(x > 0) }'; then
    printf '%s\n' "$v"
  else
    printf '1.0\n'
  fi
}
```

Reuses the private `_cw_contract_field` helper added in v0.24.0
(simplification sweep cluster 4).

### Wire-up in `cw_consult_wait`

Currently:

```bash
timeout="${!timeout_env_var:-$(cw_consult_timeout "$timeout_key")}"
```

Becomes:

```bash
local base_timeout
base_timeout="${!timeout_env_var:-$(cw_consult_timeout "$timeout_key")}"
local mult
mult=$(cw_contract_timeout_multiplier "$model")
timeout=$(awk -v b="$base_timeout" -v m="$mult" 'BEGIN { printf "%d", b * m + 0.5 }')
log_info "[$kind-wait] $commander base=${base_timeout}s × ${mult} = ${timeout}s"
```

`$model` is already in scope (4th positional arg of `cw_consult_wait`).
The integer-rounded result is what every existing downstream check
uses, so no further changes are needed.

### Effective timeouts after Layer A

| Provider | research (base 600) | verify (base 300) | adversary (base 600) | experiment (base 1800) |
|---|---|---|---|---|
| codex | 600 | 300 | 600 | 1800 |
| gemini | 600 | 300 | 600 | 1800 |
| claude | 600 | 300 | 600 | 1800 |
| opencode | **1500** | **750** | **1500** | **4500** |

## Layer B — mtime-based liveness probe

### What and why

Before writing `<state_key>=timeout`, check the trooper's
`outbox.jsonl` mtime. If it was touched within `LIVENESS_PROBE_S`
seconds of now, the trooper is provably still writing — extend the
deadline by `LIVENESS_GRACE_S`. Cap by a hard ceiling
(`MAX_DEADLINE_FACTOR × baseline`) so a stuck-but-noisy trooper still
hits a real timeout eventually.

Outbox mtime is the universal "trooper is doing something" signal. The
same heuristic powers `bin/deep-research-monitor.sh` (see `PROBE_S` /
`STUCK_S` defaults of 900s / 1800s there). One probe site covers all
four `kind`s (research / verify / adversary / experiment) without
per-kind file lookups.

### Defaults

| Knob | Default | Env override | Meaning |
|---|---|---|---|
| `LIVENESS_PROBE_S` | 120 | `CW_CONSULT_LIVENESS_PROBE_S` | "Recent activity" window — outbox touched within this long counts as alive |
| `LIVENESS_GRACE_S` | 180 | `CW_CONSULT_LIVENESS_GRACE_S` | How long to extend on each successful liveness probe |
| `MAX_DEADLINE_FACTOR` | 2 | `CW_CONSULT_MAX_DEADLINE_FACTOR` | Multiplier × `timeout` (post Layer A) for the absolute hard cap |

These are intentionally tighter than deep-research-monitor's defaults
(900s / 1800s) because consult phases are shorter-lived and bursty —
findings.md tends to grow in 30-60s pulses, not 5-min ones.

### Effective hard caps after Layer A + Layer B

| Provider | research | verify | adversary | experiment |
|---|---|---|---|---|
| codex | 1200 (20m) | 600 (10m) | 1200 (20m) | 3600 (60m) |
| opencode | **3000 (50m)** | **1500 (25m)** | **3000 (50m)** | **9000 (150m)** |

For experiment kind, deep-research already has its own Monitor-driven
liveness via `bin/deep-research-monitor.sh`, so the cap mostly serves
as a fallback if the conductor isn't using Monitor (e.g. a wait shim
called directly from a test).

### Code change (`lib/consult-wait.sh`)

Insert before the existing `(( remaining > 0 )) || { matched="";...;break; }`:

```bash
local probe_s="${CW_CONSULT_LIVENESS_PROBE_S:-120}"
local grace_s="${CW_CONSULT_LIVENESS_GRACE_S:-180}"
local cap_factor="${CW_CONSULT_MAX_DEADLINE_FACTOR:-2}"
local hard_deadline=$(( $(date +%s) + timeout * cap_factor ))

while :; do
  local now=$(date +%s)
  local remaining=$(( poll_deadline - now ))
  if (( remaining <= 0 )); then
    # Layer B: liveness probe before declaring timeout.
    local mtime delta
    mtime=$(stat -c '%Y' "$outbox" 2>/dev/null || echo 0)
    delta=$(( now - mtime ))
    if (( mtime > 0 )) && (( delta <= probe_s )) && (( now < hard_deadline )); then
      poll_deadline=$(( now + grace_s ))
      (( poll_deadline > hard_deadline )) && poll_deadline=$hard_deadline
      log_info "[$kind-wait] $commander alive (outbox mtime ${delta}s ago); +${grace_s}s grace, hard cap in $(( hard_deadline - now ))s"
      continue
    fi
    matched=""; new_offset="$OFFSET"; event=""; break
  fi

  # ...existing poll body unchanged...
done
```

`stat -c '%Y'` is GNU stat (Linux). macOS uses `stat -f '%m'`. Wrap
with platform detection (mirror what `lib/state.sh` already does for
`stat`-style calls), or use a single `stat -c '%Y' "$f" 2>/dev/null ||
stat -f '%m' "$f" 2>/dev/null || echo 0` chain. **Decision: use the
chained fallback** — keeps the code in one place, costs a single
fork-exec of stat that errors out fast on the wrong platform.

## Tests

### `test_contract_timeout_multiplier.sh` (4 cases)

1. Default `1.0` when field is absent (codex / claude).
2. `2.5` when set on opencode.
3. Default `1.0` when value is malformed (`abc`, `0`, negative).
4. Decimal accepted (`1.5`, `3.0`).

### `test_consult_wait_liveness_probe.sh` (3 cases)

1. Stale outbox at deadline → real timeout (`*S=timeout` written).
2. Fresh outbox at deadline → grace extension (log line emitted, no
   `timeout` write yet); subsequent stale outbox → real timeout.
3. Repeated grace extensions hit hard cap → real timeout fires
   regardless of further activity (anti-stuck-noisy-trooper safety).

These tests synthesize outbox.jsonl files with controlled mtimes via
`touch -d '@<epoch>'`. They use `CW_CONSULT_LIVENESS_PROBE_S=2` /
`CW_CONSULT_LIVENESS_GRACE_S=2` / `CW_CONSULT_MAX_DEADLINE_FACTOR=2`
and `CW_CONSULT_RESEARCH_TIMEOUT_OVERRIDE=2` so the test runs in
single-digit seconds.

## Out of scope

- **Trooper-emitted heartbeat events** (Approach 2 above) — separate
  brainstorm if mtime-based probing proves insufficient.
- **deep-research's `bin/deep-research-monitor.sh`** — already
  mtime-aware via `PROBE_S` / `STUCK_S`. v0.35.0 leaves it untouched.
- **Per-kind multipliers** (e.g. opencode adversary 2× but research
  3×) — YAGNI; one multiplier per provider is enough until evidence
  says otherwise.
- **Multipliers on `gemini` / `claude`** — they ship at `1.0` until
  dogfood data motivates a change.

## Release-gate dogfood

Re-run `/clone-wars:consult` with wolffe in the roster on a topic that
previously timed out. Verify:

1. Wolffe's research phase logs `base=600s × 2.5 = 1500s` at start.
2. When wolffe is 600s in but writing every ~30s to findings, the
   wait-shim log shows `+180s grace, hard cap in <N>s` lines, not
   `FS=timeout`.
3. When wolffe truly wedges past the hard cap (3000s for research),
   `FS=timeout` fires correctly and the trooper is torn down.
4. `CW_CONSULT_LIVENESS_PROBE_S=0` env var disables the probe (forces
   v0.34-style behavior) — useful as an escape hatch for tests.
5. Codex / claude troopers behave byte-equal to v0.34.0 (multiplier
   defaults to `1.0`, no probe extension fires unless their outbox
   mtime is within `LIVENESS_PROBE_S` of deadline).

## Open questions for reviewer

1. Multiplier value for opencode — `2.5` chosen empirically from the
   wolffe symptom (18 KB findings at 600s deadline implies real work
   continues for some unknown additional time; 2.5× = 1500s is a
   plausible upper bound for typical research). Adjust to `2.0` or
   `3.0` if you have better data.
2. `LIVENESS_PROBE_S=120` and `LIVENESS_GRACE_S=180` — the ratio
   matters more than the absolute values. 120/180 says "if you wrote
   in the last 2 min, you get 3 more min." Could be `60/120` (more
   responsive but more frequent log noise) or `300/600` (matches
   deep-research-monitor's PROBE_S; quieter but slower to react).
3. Hard cap of `2×` — this is the one knob a malicious / runaway
   trooper can't escape. If it's wrong, increasing it is a one-line
   contracts.yaml edit (no code change).
4. Should the multiplier also apply to `cw_contract_ready_timeout`
   (the spawn-time wait)? Probably yes for symmetry, but that's a
   separate code path and adds blast radius. **Spec defers this to
   v0.36.0** if dogfood shows ready-timeouts still fire on opencode.
