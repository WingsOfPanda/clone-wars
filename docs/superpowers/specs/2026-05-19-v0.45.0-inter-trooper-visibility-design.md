## Goal

Each per-experiment dispatch inlines a snapshot of *peer* trooper
status (phase, current/last experiment, approach, best metric, 1-line
notes excerpt) into the trooper's `prompt.md`. Yoda stays the
dispatch authority — this just makes peer activity visible to the
working trooper so they can build on or diverge from what their peers
are exploring. Zero behavioral change for solo (N=1) sessions: the
`## Peers` section is omitted entirely.

## Scope

**In scope:**

1. New `cw_deep_research_format_peers_block <art-dir> <current-cmdr>`
   helper in `lib/deep-research.sh`. Wraps the existing
   `cw_deep_research_render_status_brief` data sources (`state.txt`,
   `scoreboard.md`, `result.json`) but filters OUT the current trooper
   so they don't see themselves. Emits a `## Peers` section + table
   + divergence-guidance paragraph. Emits nothing (rc=0, empty stdout)
   when the trooper roster has only the current commander.
2. New `{{PEERS_BLOCK}}` placeholder in
   `config/prompt-templates/deep-research/experiment.md`, inserted
   between the existing `{{SOTA_BLOCK}}` (v0.44.0) and `Branch sandbox:`
   sections.
3. `bin/deep-research-experiment-send.sh` builds `PEERS_BLOCK_VAL`
   from the helper, passes it as an `-v` awk arg, adds the gsub line.
   Empty value → no `## Peers` section in dispatched prompt.md
   (back-compat for solo sessions and any caller that doesn't seed
   multi-trooper state).
4. `commands/deep-research.md` Phase 4.a gains a v0.45.0 callout (in
   the same block as the existing v0.43.0 context-file and v0.44.0
   SOTA callouts) explaining peer visibility and the divergence
   default.
5. Permanent directive lint extended (+1 invariant) asserting the
   Phase 4.a callout references `## Peers`.
6. Version-stamped static-wiring lock at v0.45.0 (skip-guarded until
   the version bump in T6) — 5 invariants.

**Explicitly out of scope (postponed):**

- **Trooper-driven direct messaging.** rex actively writing into
  keeli's inbox. Yoda stays the routing layer. Trooper-initiated
  outbound messaging is a v0.46+ candidate if dogfood shows demand.
- **Mid-session sota.md mutation.** v0.44.0's write-once policy
  stands; trooper-discovered SOTA refs go in their own `notes.md`.
- **Peer notes.md inclusion.** Variable size; deferred until dogfood
  shows the 1-line notes excerpt from result.json isn't enough.
- **Pub/sub events from peer outbox.jsonl.** Too much plumbing;
  dispatch-time snapshot covers the documented use case.
- **Per-trooper "show me peer X's full result.json" lookup.** No
  evidence anyone needs it yet.

## Architecture / contract

Inter-trooper visibility is a **dispatch-time snapshot**, not a
runtime channel. At each per-experiment dispatch,
`bin/deep-research-experiment-send.sh` calls
`cw_deep_research_format_peers_block` with the current trooper's
commander name; the helper reads disk state (`state.txt`,
`scoreboard.md`, `result.json` per peer) and emits a rendered
Markdown block to stdout; awk substitutes it into the prompt.md
template's `{{PEERS_BLOCK}}` placeholder. The trooper reads this
once at dispatch and works from the snapshot.

Mirror of the v0.44.0 SOTA pattern: empty input → empty output → no
header in prompt.md (back-compat preserved). The trooper never reads
peer state directly; the only filesystem traversal happens in the
helper, which runs as the conductor (Yoda) not as the trooper.

```
Phase 4.a dispatch — per experiment:
   experiment-send.sh
     ↓
     reads $ART_DIR + current $COMMANDER
     ↓
     cw_deep_research_format_peers_block "$ART_DIR" "$COMMANDER"
       ├── lists commanders from $ART_DIR/troopers.txt (or troopers/ dirs)
       ├── filters out $COMMANDER
       ├── if remaining list empty → emit nothing, rc=0
       └── else → render ## Peers + table + divergence note
     ↓
     PEERS_BLOCK_VAL captured into awk -v arg
     ↓
     awk substitutes {{PEERS_BLOCK}} placeholder → prompt.md
     ↓
     trooper inbox.md cat's prompt.md + END_OF_INSTRUCTION sentinel
```

## Components

### Lane A — `cw_deep_research_format_peers_block` helper

**File:** `lib/deep-research.sh` (new helper appended after
`cw_deep_research_format_sota_block` introduced in v0.44.0; ~80
lines).

**Signature:** `cw_deep_research_format_peers_block <art-dir> <current-cmdr>`

**Inputs:** art-dir (`$CLONE_WARS_HOME/state/<repo-hash>/<topic>/_deep-research`),
current commander name (string, must match one of the rostered
troopers).

**Behavior:**

1. Read commander list — prefer `$art_dir/troopers.txt` (one
   commander per line, written by Phase 4.a per v0.28.2). Fallback
   to directory scan of `$art_dir/troopers/`.
2. If commander list is empty or contains only `<current-cmdr>` —
   emit nothing (rc=0), no header, no error.
3. For each peer commander (all rostered except current):
   - Read `phase` from `$art_dir/troopers/$peer/state.txt`.
   - Read `current_exp_id`; fall back to most-recent
     `exp-NNN/` directory via lexical sort.
   - Read approach label + best metric from peer's most recent
     `result.json` (when `phase==idle` or scoreboard has a row).
   - Read `notes` field from the most recent result.json (trim to
     ~80 chars, single line).
4. Emit:

   ```markdown
   ## Peers
   
   Your peer troopers — read for context, not as a target. Your job
   is to explore a different corner of the space. If you converge on
   a peer's approach, justify why in `notes.md`.
   
   | Trooper | Phase | Current/last | Approach | Best metric | Notes |
   |---------|-------|--------------|----------|-------------|-------|
   | <peer1> | <phase> | <exp-NNN> | <approach> | <metric> (status) | <notes excerpt> |
   ...
   ```

5. Errors that should NOT block dispatch (peer state files missing /
   malformed): emit `?` or `—` for the missing column, continue. The
   helper never returns rc≠0 on missing peer data — only on missing
   art-dir or missing current-commander arg (rc=2 usage error).

The helper does NOT call `cw_deep_research_render_status_brief`
directly — that helper renders a full status form (scoreboard top 3,
completion-check signal) that doesn't belong in a peer block. The
peers helper has its own narrower per-row rendering.

## Data structures

### Peers block format

Always rendered at dispatch time when N≥2 troopers; nothing rendered
when N=1. Markdown source:

```markdown
## Peers

Your peer troopers — read for context, not as a target. Your job is
to explore a different corner of the space. If you converge on a
peer's approach, justify why in `notes.md`.

| Trooper | Phase   | Current/last | Approach    | Best metric  | Notes                        |
|---------|---------|--------------|-------------|--------------|------------------------------|
| keeli   | working | exp-003      | Plain CNN   | 0.9821 (ok)  | param-shrink to 96k worked   |
| cody    | idle    | exp-004      | Transformer | 0.9742 (ok)  | seq-len 28 was bottleneck    |
```

Row count: always (N−1) where N is total troopers. No truncation —
roster is bounded to 2-3 (codex-only) so worst-case 2 peer rows.

Per-row cell defaults when data is unavailable:

| Cell | Source | Fallback when missing |
|---|---|---|
| Trooper | `troopers.txt` | (required; row skipped if commander name missing) |
| Phase | `state.txt:phase` | `?` |
| Current/last | `state.txt:current_exp_id` then most-recent `exp-NNN/` dir | `—` |
| Approach | most recent `exp-NNN/result.json:approach_label` | `—` |
| Best metric | most recent `exp-NNN/result.json:metric_value` + `:status` | `—` |
| Notes | most recent `exp-NNN/result.json:notes` trimmed to 80 chars, single line | `—` |

"Most recent" = lexically-greatest `exp-NNN/` directory under
`troopers/$peer/experiments/` that contains a readable `result.json`.
Scoreboard.md is NOT a source for any cell — the helper goes
straight to result.json so it doesn't depend on score.sh having run
successfully. Robustness over richness.

### Cost in trooper prompt

Per peer: ~one table row (~100 chars). With max 2 peers + header +
divergence paragraph: ~500 chars added per dispatch. Negligible
against the trooper's 200K+ token context allocation.

## Error handling

- **N=1 solo session.** Helper emits empty stdout, rc=0. Dispatch
  inlines empty string → no `## Peers` header in prompt.md.
  Regression lock test pins this.
- **Peer `state.txt` missing or malformed.** Row renders with `?`
  for phase, `—` for the other cells. Helper still returns rc=0.
- **Peer `result.json` missing** (peer hasn't completed any
  experiment yet). Row renders with phase + current_exp_id from
  state.txt, `—` for approach/metric/notes.
- **Peer's roster entry exists but `troopers/$peer/` directory
  doesn't.** Row skipped silently (defensive against partial state).
- **`<art-dir>` missing.** Helper returns rc=2 with stderr message.
  Dispatch script's `|| true` swallows this — `PEERS_BLOCK_VAL`
  stays empty, dispatch continues. (Same back-compat pattern as
  SOTA; a malformed art-dir is a bigger problem than missing peer
  info.)
- **Current commander not in troopers.txt.** Helper still runs the
  filter step; result is no peers excluded → all rostered commanders
  appear in the table including the "current" one (which is a bug,
  but a survivable one — better than blocking dispatch). The
  dispatch should only call this helper with a rostered commander
  name, so this edge case is theoretical.

1. In any session with N≥2 troopers, every dispatched `prompt.md`
   contains a `## Peers` section with (N−1) data rows; the
   dispatching trooper never appears in its own peers table.
2. In any session with N=1 trooper, no `## Peers` header appears in

6. Zero regressions in pre-existing tests.

## Risks

- **Peer awareness causes convergence anyway.** The divergence-bias
  prompt is just words. A trooper might still imitate a successful
  peer approach. Mitigation: explicit `notes.md` justification
  requirement gives Yoda visibility into convergence decisions at
  synthesis time. If dogfood shows persistent convergence,
  v0.46+ could escalate to "Yoda overrides convergence in next
  dispatch."
- **Stale snapshot during fast-moving peer activity.** A peer
  completes exp-005 between Yoda's `format_peers_block` call and the
  trooper actually reading prompt.md. The trooper sees exp-004 as
  peer's latest. Acceptable: the peer block is a hint, not a
  contract; Yoda's next dispatch will re-snapshot.
- **Helper performance on large rosters.** N is bounded to 2-3
  troopers in current code (deep-research-init.sh refuses N>3).
  Helper does O(N) state file reads; trivial. Not a concern.
- **Cross-trooper info leak.** The peer block exposes peer approach
  labels and metrics to the current trooper. This is by design
  (the whole point of v0.45.0). If a user wants strict isolation,
  the right surface is to run separate `/clone-wars:deep-research`
  sessions — not a v0.45.0 flag.
