# v0.50.0 — Trooper Escalation Protocol Design

**Date:** 2026-05-21
**Status:** Design locked
**Base:** main @ 5a58af1 (v0.49.0)
**Source:** May 21 audit of consult/deploy/deep-research archives (findings #1, #2, #4)

## Goal

Give troopers a protocol-level way to halt and ask the conductor when they hit a real blocker, instead of silently working around it. Close 3 of the 6 audit findings:

- **#1 (Hi/S)** Deploy turn-prompt template has no halt-on-blocker contract. May 21 incident: cody couldn't locate the archive at the literal path the design doc cited, went into "search the filesystem myself" mode, never emitted a `question` event. Yoda only intervened because the pause was noticed manually.
- **#2 (Med/S)** Inbox-reply has no acknowledgment contract. Trooper resumed with free-text `ack`; no proof inbox.md was actually read.
- **#4 (Med/S)** Same mitigation-language pattern in consult troopers ("located archived exp-006" / "located v7 tablebase"). Consult's research role is more legitimate, but normalizing the protocol across both lanes keeps the habit consistent.

Findings #3 (state.txt race), #5 (-FAILED forensics), #6 (result.json checkpoint_path) are deferred to a v0.51.0 hygiene round.

## Background: what already exists

Consult already has a complete question-event protocol:

- `lib/consult.sh:772-848` — `cw_consult_question_payload_write/read/validate/extract` family.
- `lib/consult-wait.sh:111-162` — wait loop listens for `done|error|question` and writes payload files on hit.
- Yoda directive in `commands/consult.md` handles question events by reading the payload, asking the user via AskUserQuestion, writing reply to inbox.md, then re-arming the wait.

Deploy has nothing equivalent:

- `bin/deploy-turn-wait.sh:37` only listens for `done|error`. A question event from a deploy trooper would be ignored until timeout.
- `lib/deploy.sh::cw_deploy_build_turn_prompt_round1` (lines 130-186) gives one halt example — the `branch-discipline` `{event:"error"}` at lines 174-182 — and zero examples of `{event:"question"}`.

So v0.50 is **not a new protocol**. It extends the existing consult protocol with a `claim` discriminator that lets Yoda auto-verify simple questions without bothering the user, and ports it into the deploy lane.

## Scope

Single refactor-only release. Pure bash 4.2+. No new runtime dependencies. Version-stamped static-wiring lock (matches v0.46/v0.47/v0.48/v0.49 release pattern).

## Protocol

### Question event shape

Trooper appends one JSONL line to outbox.jsonl:

```json
{"event":"question","text":"<why-asking>","claim":{"kind":"path|git|env|cmd|test","value":"<value>"},"ts":"<iso>"}
```

- `text` — required, free-text reason for asking.
- `claim` — optional. Present → Yoda routes through verify dispatcher. Absent → Yoda escalates straight to user.
- `claim.kind` discriminator (one of):
  - `path` — `[[ -e "$value" && -r "$value" ]]` + `ls -ld "$value"` for evidence.
  - `git` — `git rev-parse --verify "$value"` (refs, SHAs, paths inside trees).
  - `env` — `printenv "$value"` (presence + value).
  - `cmd` — `command -v "$value"` (binary on PATH).
  - `test` — `timeout 30 bash -c "$value"` capturing stdout/stderr/rc. **Bounded to diagnostic checks under 30s; explicitly not for running the trooper's test suite.**

### Yoda's verify dispatcher

New library `lib/trooper-questions.sh`:

- `cw_trooper_question_verify <kind> <value>` — returns rc=0 if the claim verifies (file exists, ref resolves, env set, cmd found, test exits 0), rc=1 if it concretely doesn't (file missing, ref unknown, env empty, cmd absent, test exits non-zero), rc=2 if unverifiable (unknown kind, timeout, malformed input). Prints evidence (ls output, rev-parse hash, env value, command path, test stdout+stderr+rc) to stdout.
- `cw_trooper_question_format_reply <kind> <value> <rc> <evidence-path>` — formats the inbox.md reply body that Yoda will push back. Reply includes: verdict line (FOUND / NOT FOUND / UNVERIFIABLE), absolute path of evidence, and a directive line telling the trooper to resume.
- `cw_trooper_question_validate_line <json-line>` — rc=0 iff the line is a parseable `{event:"question"}` with non-empty `text`. Used by wait scripts to gate the TS=question exit. Shared by deploy and consult.

Routing rules (Yoda directive prose, both deploy and consult):

1. Read payload file → extract `text`, `claim.kind`, `claim.value`.
2. If `claim` absent → AskUserQuestion with the free-text `text`, write user's reply to inbox.md.
3. If `claim` present → run `cw_trooper_question_verify`. If rc=0 or rc=1 → call `cw_trooper_question_format_reply`, write to inbox.md (no user prompt). If rc=2 → fall back to AskUserQuestion path.

### Inbox-ack contract

After the trooper reads inbox.md, the trooper calls `bin/inbox-ack.sh <topic> <commander> <inbox-path>`. The helper:

1. Reads inbox.md.
2. Computes `sha256sum < inbox.md | cut -d' ' -f1`.
3. Extracts the last non-blank line.
4. Appends `{"event":"ack","inbox_sha256":"<hex>","inbox_tail":"<line>","ts":"<iso>"}` to outbox.jsonl.

Yoda's directive prose verifies the sha matches Yoda's own computation of inbox.md before continuing the wait. On mismatch:

- First mismatch → re-push the same inbox.md (idempotent — same sha, same content).
- Second mismatch → emit `error` to outbox and log to Yoda's stderr. Don't loop forever.

### Trooper-ask helper

`bin/trooper-ask.sh <topic> <commander> <text> [<kind> <value>]` — appends a well-formed question event to the trooper's outbox.jsonl. Absolute path is baked into the prompt template so the trooper never has to remember the JSON shape.

Usage examples baked into the prompt:

```bash
# Path doesn't appear where the design doc said it would:
bin/trooper-ask.sh "$TOPIC" cody "design doc cites <path> but it's not there" path "/abs/path/from/design"

# Git ref the doc references doesn't resolve:
bin/trooper-ask.sh "$TOPIC" cody "design says branch X has file Y" git "X:python/utilities/foo.py"

# Opinion-bearing question (no claim — straight to user):
bin/trooper-ask.sh "$TOPIC" cody "should I keep the legacy fallback or remove it?"
```

## Risk surface

- **Prompt-template churn affects every future deploy/consult run.** Mitigation: the new section is additive (before `END_OF_INSTRUCTION`); existing prompts still work. Verification: dry-run `cw_deploy_build_turn_prompt_round1` and diff against committed v0.49.0 output to confirm only the new section appears.
- **`kind=test` is the soft spot.** Risk: trooper offloads work via "please run my test suite for me" questions. Mitigations: (1) prompt template explicitly bounds it ("diagnostic checks ≤30s, not full test suites"); (2) `cw_trooper_question_verify` enforces `timeout 30` on the bash invocation; (3) Yoda's reply for `kind=test` includes a one-line directive: "this was a diagnostic only — running your test suite is your job, not mine."
- **Inbox-ack sha mismatch loop.** Mitigation: first mismatch re-pushes idempotently; second mismatch fails loudly with an error event. No unbounded retry.
- **Consult's existing `cw_consult_question_*` family is left in place.** The new `lib/trooper-questions.sh` adds the verify+format helpers on top; consult's payload writer/reader continues to work as-is. No churn to consult's tested code paths.

## What this does NOT do

- No retry/backoff on question events (one question per round; if a trooper keeps asking, that's a design-doc clarity bug, not a protocol bug).
- No claim batching (`claims:[]` shape rejected — YAGNI; the May 21 incident was single-claim).
- No question handling in `bin/deploy-wave-wait.sh` (multi-repo DAG wave path) — defer to v0.51 if needed; the single-trooper deploy turn is the actual incident surface.
- No changes to `cw_consult_question_payload_*` family (consult's protocol stays; only the prompt-template prose gets aligned).
- No `bin/state-reconcile.sh` (finding #3), no `bin/failed-trooper-forensics.sh` (finding #5), no `result.json` schema change (finding #6) — all deferred to v0.51.
