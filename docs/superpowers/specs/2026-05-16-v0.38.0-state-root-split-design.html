Problem

v0.31.0 collapsed two distinct concepts under a single cw_state_root helper:

  • Per-MACHINE config — what binaries are available (providers-available.txt), shipping defaults (contracts.yaml, commanders.yaml), user's roster choice (providers-active.txt). One per install; survives across all projects.
  • Per-PROJECT state — this project's ongoing /consult topics, /deploy runs, /deep-research sessions. One per repo; meaningful only inside that repo.

Before v0.31.0, both lived globally under ~/.clone-wars/. v0.31.0 moved the whole tree project-local for the per-project case (right) but inadvertently dragged the per-machine config along (wrong). Various scripts then inconsistently bypass cw_state_root using literal ${CLONE_WARS_HOME:-$HOME/.clone-wars} because the authors knew the data was per-machine — but didn't formalize a separate helper.

Observed symptom (user-reported)

Running /clone-wars:medic writes <project>/.clone-wars/providers-available.txt, then reads ~/.clone-wars/providers-available.txt in Step A (different code path). On any fresh install, this means: medic detects providers correctly, writes the list project-local, then the interactive roster picker silently logs warn: providers-available.txt not found; skipping trooper selection because it looks in the wrong place. Roster picker never fires; consult silently uses defaults.

The user only saw the picker work because they had a pre-v0.31.0 ~/.clone-wars/providers-available.txt left over from earlier installs. Fresh-install users get a broken medic→consult chain. Same class of break affects /clone-wars:deep-research-init's codex-presence gate and bin/deploy-multi-init.sh's commanders.yaml lookup.

Goal

(A) Restore the broken medic→consult/deploy/deep-research chain for fresh installs. (B) Add an explicit cw_global_state_root helper so per-machine config has a single source of truth. (C) Stop the workaround pattern where authors bypass cw_state_root with literal global paths because no helper exists for that case. (D) Add a permanent lint that enforces the discipline.

Approach summary

Layer A — new helpers in lib/state.sh

# cw_global_state_root — per-MACHINE config root.
# Always resolves to ${CLONE_WARS_HOME:-$HOME/.clone-wars}, regardless of $PWD.
# Use for config that is per-install, not per-project: medic's
# providers-{active,available}.txt, contracts.yaml, commanders.yaml,
# the archive/ subtree.
#
# Contrast with cw_state_root (per-PROJECT state at $PWD/.clone-wars/)
# for /consult, /deploy, /deep-research, /meditate per-topic work.
cw_global_state_root() {
  printf '%s\n' "${CLONE_WARS_HOME:-$HOME/.clone-wars}"
}

# cw_global_state_ensure — like cw_state_ensure but for the global root.
cw_global_state_ensure() {
  local root; root=$(cw_global_state_root)
  mkdir -p "$root"
  [[ -f "$root/.gitignore" ]] || printf '*\n' > "$root/.gitignore" 2>/dev/null || true
}

Layer B — migrate the per-machine paths (12 call sites)

Mechanical conversion. Order: lib/ first, then bin/, then commands/. See the surface table for the per-file change list.

Layer C — drop commands/medic.md Step 1 (args-file boilerplate)

medic takes no arguments. The _args/ staging pattern is v0.31.0 boilerplate that applied uniformly to every directive. For medic it's dead weight that pollutes <project>/.clone-wars/_args/ with empty files. Drop steps 1, 2, 3 from commands/medic.md; new Step 1 is just "${CLAUDE_PLUGIN_ROOT}/bin/medic.sh" (no --args-file flag).

bin/medic.sh keeps accepting --args-file <path> as a no-op for back-compat with mid-upgrade sessions. The script's cw_args_file_consume call already handles the empty-args case gracefully.

Layer D — permanent lint tests/test_state_root_discipline.sh

Counterpart to v0.37.0's test_no_hardcoded_paths.sh. Runs every release. Enforces:

  1. medic.{md,sh} use cw_global_state_root, never cw_state_root.
  2. contracts/commanders helpers + cw_active_providers_path use cw_global_state_root.
  3. Archive base files use cw_global_state_root.
  4. No literal ${CLONE_WARS_HOME:-$HOME/.clone-wars} outside lib/state.sh.
  5. commands/medic.md has no _args/ reference (Step 1 dropped).

10 invariants per Layer E.

Out of scope

  • Migration code — leave-alone, per v0.31.0 precedent. Users on the v0.31–v0.37 era who relied on project-local medic state need one /medic re-run after upgrade.
  • Renaming cw_state_root — too invasive; cw_state_root keeps its per-project semantics. The new helper sits alongside.
  • Removing ${CLONE_WARS_HOME:-$HOME/.clone-wars} env-var seam — kept inside lib/state.sh's helpers for tests/debug. Outside lib/state.sh, the literal is banned by the permanent lint.
  • Auto-detecting + cleaning up v0.31–v0.37 project-local stragglers — out of scope. They become inert.
  • Migrating existing project-local archive/ subtrees — out of scope. v0.38.0+ writes new archives globally; old project-local archives stay where they are.

Release-gate dogfood

  1. /clone-wars:medic writes ~/.clone-wars/providers-available.txt (not project-local). Verify with ls ~/.clone-wars/.
  2. commands/medic.md no longer creates <project>/.clone-wars/_args/medic.XXXXXX (no Step 1 args-file write).
  3. Subsequent /clone-wars:consult reads the roster correctly — interactive picker runs (no skipping trooper selection warning).
  4. Fresh-install simulation: mv ~/.clone-wars ~/.clone-wars.bak && rm -rf <project>/.clone-wars/{providers-*,contracts.yaml,commanders.yaml}. Re-run /medic: contracts.yaml/commanders.yaml get copied from plugin to ~/.clone-wars/, providers detected, roster picker fires.
  5. Archive dir for new /consult, /deploy, /deep-research, /meditate runs lands at ~/.clone-wars/archive/<repo-hash>/<topic>-<ts>/ (uniformly).
  6. bash tests/test_state_root_discipline.sh exits 0.
  7. Deliberate-regression test: add cw_state_root call to bin/medic.sh, lint exits 1 with file:line + suggested fix.

Open questions

  1. cw_global_state_ensure .gitignore auto-write — ~/.clone-wars/ isn't usually under git but the defensive write costs nothing. Worth keeping? Yes.
  2. bin/medic.sh accepting (and ignoring) --args-file for back-compat — covers users who upgrade mid-session and their cached directive still passes --args-file. Recommendation: keep no-op for one release, drop in v0.39.
  3. Lint coverage for literal env-var seam — strict; will fail if anyone manually expands ${CLONE_WARS_HOME:-$HOME/.clone-wars} outside lib/state.sh. Acceptable since the helper is now available.
