**Date:** 2026-05-12
**Status:** approved by user (brainstormed 2026-05-12)

## 1. Problem

The v0.26.0 first-machine dogfood (2026-05-12) finished end-to-end with reasonable results (99.03% on MNIST under 100k params), but surfaced two structural complaints in addition to seven mechanical bugs:

1. **The metric is implicit.** v0.26.0 extracts a metric heuristically from the topic prose; if heuristic finds one, troopers start without the user ever confirming what "good" means. The user explicitly asked for a preflight metric confirmation step that produces a structured definition.
2. **The K×N round structure is too rigid.** A research advisor doesn't dispatch four parallel students per round on a fixed grid. They dispatch one or two tasks, read results, decide who does what next, and adjust. The user described the desired model as *"imagine you have 2-3 PhD students that doing research under your command. You may dispatch multiple experiment tasks to them one by one. Also, you may drop similar, or progress tasks to same troopers."*

These are not bugs to patch — they're a redesign of the research loop. The seven mechanical bugs from the v0.26.0 dogfood log fold into the same release because most of them become moot (no rounds → no per-round teardown bug; no budget flags → no aggressive-default complaint).

## 2. Goal

Replace v0.26.0's fixed K×N tree-search autoresearch with an **advisor-driven workflow**. The conductor (Yoda / claude) acts as a research advisor with **2-3 long-lived codex troopers** (PhD-student model). The conductor decides the metric framing, the experiments to dispatch, who does each, and when to stop — adaptively based on results, not by a fixed budget grid.

## 3. Architecture

### 3.1 Roster: 2-3 long-lived codex troopers

Spawned **once** at the start of the session (Phase 2). Persist through every experiment. Torn down **once** at the end (Phase 4). Conductor decides between N=2 and N=3 at preflight based on its read on topic complexity:

- **Narrow & deep** (single objective, tight constraint surface, e.g. "optimize MNIST accuracy under 100k params") → N=2.
- **Broad survey** (multiple sub-goals, exploratory, e.g. "find the best caching strategy for our hot-path APIs") → N=3.

**Where the judgment lives.** Yoda makes the N=2 vs N=3 call inline in the directive prose (not in bash). The bash helper is signature `cw_deep_research_pick_roster <N>` — it returns N codex commanders from the pool in deterministic order (rex, keeli, colt for N=3; rex, keeli for N=2). No keyword classifier in bash. The directive has a short "narrow vs broad" rubric with worked examples for Yoda to follow:

> **N=2 (default)** — single objective + tight constraint surface. Examples: "optimize MNIST accuracy under 100k params", "minimize p99 latency on /api/search", "find the lowest-loss config under fixed compute budget".
>
> **N=3** — multiple sub-goals OR broad survey OR no clear single optimum. Examples: "find the best caching strategy for our hot-path APIs" (multiple competing strategies, multiple workloads), "explore SOTA continuous-batching schedulers" (no single winner expected), "compare RAG vs fine-tune vs few-shot for the support classifier" (three families to staff one each).

If Yoda is unsure, default to N=2 — the safety-net stop-check protects the user from runaway cost regardless. Commanders come from the existing codex-eligible pool in `lib/deep-research.sh` (17 entries; rex, keeli, colt, …).

### 3.2 Four phases per session

| Phase | Step | Mechanism | Output |
|---|---|---|---|
| **1. Metric discussion** | Adaptive dialogue with user | AskUserQuestion loop in directive | `_deep-research/metric.md` |
| **2. Preflight** | Time-limit + spawn | Single AskUserQuestion + parallel `bin/spawn.sh` | `_deep-research/time-budget.txt`, trooper panes |
| **3. Research loop** | Conductor-driven dispatch/wait/score/stop-check | Loop of `bin/deep-research-experiment-{send,wait}.sh` + `score.sh` | `_deep-research/experiments/exp-NNN-<cmdr>/result.json`, rolling `scoreboard.md` |
| **4. Synthesis + teardown** | Write landscape doc, archive | Write tool + `bin/teardown.sh --pairs` + `bin/deep-research-teardown.sh` | `_deep-research/deep-research-<date>-<slug>.md`, archive dir |

#### Phase 1 — Metric discussion (adaptive dialogue)

Yoda runs a free-form back-and-forth to produce a structured metric block.

1. Run existing `cw_deep_research_extract_metric` heuristic on the topic to seed a guess.
2. Optionally pair `WebSearch` + `mcp__tavily__tavily-search` for additional context.
3. Open with ONE AskUserQuestion proposing the read of the goal. Example for a clear topic:
   > *"I read this as: maximize MNIST test accuracy, subject to <100k params. What's a target threshold — 99%, 99.5%, no specific cap?"*
4. User answers; Yoda asks zero or more follow-ups until the goal is structured.
5. Write `_deep-research/metric.md` (markdown block, not the v0.26.0 single-line `metric.txt`):

   ```markdown
   # Research goal

   **Primary metric:** accuracy
   **Direction:** maximize
   **Target (good):** >= 0.99
   **Acceptable:** >= 0.97
   **Hard constraints:** params < 100k

   **Notes:** MNIST test set; wall-clock per experiment ~5 min target
   ```

6. Final confirmation AskUserQuestion: *"Here's how I'll frame the goal — OK to proceed?"*. User confirms or refines.

Length adapts to topic clarity: 1-2 prompts for clear topics, 3-5 for ambiguous ones. Troopers see the metric.md body verbatim in their prompt under `## Research goal`.

Helper: `cw_deep_research_format_metric_block` renders the metric.md from `K=V` pairs given on stdin. Called from the directive after the dialogue settles.

#### Phase 2 — Preflight (one AskUserQuestion, then spawn)

Yoda explains its roster-size choice in chat, then fires ONE AskUserQuestion:

> *"Time limit on this research session? E.g. 12h, 4h, none."*
>
> Options: `No limit (default, recommended)` / `4 hours` / `12 hours` / `Other (Custom)`

User's choice is parsed and saved to `_deep-research/time-budget.txt`:
- `none` for `No limit`
- Integer seconds otherwise (e.g. `14400` for 4 hours, `43200` for 12 hours)

Edge cases: parse error → re-prompt; user enters `0` or negative → re-prompt. Troopers don't see this file — it's a conductor-only signal for the stop-check.

After this prompt, spawn N codex troopers via parallel Bash tool calls (one `bin/spawn.sh <cmdr> codex <topic>` per commander). Reuses v0.26.0's spawn-rollback runbook: auto-retry-once on cold-start failure via batched `bin/teardown.sh --pairs`. If retry also fails on a subset, AskUserQuestion: `Proceed with degraded roster (N-1)` / `Abort`.

#### Phase 3 — Research loop (advisor-driven)

No fixed structure. Conductor-led loop. Each iteration:

1. **Decide** — Yoda picks the next move: dispatch a new approach, dispatch a follow-up to a specific trooper that builds on their prior turn, or stop.
2. **Dispatch** — `bin/deep-research-experiment-send.sh <topic> <commander> exp-NNN <approach-label> <approach-brief>`.
   - `exp-NNN` is a global chronological counter (not per-round).
   - Branch dir: `_deep-research/experiments/exp-NNN-<commander>/`.
   - Prompt template interpolates `metric.md` body under `## Research goal`.
3. **Wait** — `bin/deep-research-experiment-wait.sh <topic> <commander> codex`. Foreground; returns when trooper's outbox emits `done` for this dispatch.
4. **Score** — `bin/deep-research-score.sh <topic>` (no round arg now). Reads all `experiments/exp-*/result.json`, writes flat `_deep-research/scoreboard.md` sorted by metric_value descending.
5. **Stop check:**
   - If `time-budget.txt` is a positive integer AND `(now - session_start) >= time_budget`:
     - Fire AskUserQuestion: `Continue` / `Stop` / `Extend (+T hours)`.
   - Else if `time-budget.txt == none` AND the last 5 experiments (by chronological order, starting from `exp-005`) all produced `metric_value` that did not beat the current best by >1%:
     - Fire AskUserQuestion: `Continue` / `Stop` / `Adjust direction`.
     - Floor: no stagnation prompt fires before experiment 5.
     - **Reset on Adjust direction:** `_deep-research/stagnation-cursor.txt` records the exp number from which to count. After Yoda updates metric.md (if needed) and resumes the loop, future stagnation checks ignore experiments before the cursor.
   - Else continue.
6. Loop.

A trooper can be re-dispatched. Their previous turn's transcript stays in their codex session, so a follow-up like *"That worked at 98.94%. Now try the same arch with weight decay 1e-4 — write the new run to exp-007-rex/code/"* builds on prior context naturally. The conductor's directive prose covers this pattern.

#### Phase 4 — Synthesis + teardown

When Yoda decides "done" (stop-check user-answer = `Stop`, or judgment: results saturated):

1. Write `_deep-research/deep-research-<date>-<slug>.md` via Write tool (atomic single-shot).
   - Same shape as v0.26.0 with **`## Experiment log`** section (chronological table of all `exp-*` results) replacing **`## Round-by-round`**.
   - Winner = highest `metric_value` row with `status: ok`.
   - `## Suggested next` emits `/clone-wars:deploy <archive-path>/<winner-exp-dir>/code/` when winner is `status: ok`.
2. `bin/teardown.sh --pairs <topic> <cmdr-1> [<cmdr-2>] [<cmdr-3>]` — single 9s graceful banner across the whole roster.
3. `bin/deep-research-teardown.sh <topic>` — archive entire topic state dir.

### 3.3 State shape (`_deep-research/`)

```
_deep-research/
├── topic.txt                       # verbatim topic string
├── metric.md                       # structured metric block (NEW, replaces metric.txt)
├── time-budget.txt                 # 'none' or integer seconds (NEW)
├── seed-from.txt                   # path to meditate landscape (when --seed-from given)
├── troopers.txt                    # 2-column TSV: <provider>\t<commander> (1 line per trooper)
├── session-start.txt               # ISO-8601 UTC of preflight completion (NEW, used by stop-check)
├── stagnation-cursor.txt           # exp number from which stagnation counts; '0' until first 'Adjust direction'
├── experiments/
│   └── exp-NNN-<commander>/        # one dir per dispatched experiment, NNN zero-padded 001..N
│       ├── code/                   # trooper's implementation
│       ├── result.json             # schema unchanged from v0.26.0
│       ├── stdout.log
│       ├── stderr.log
│       └── prompt.md               # exact prompt rendered for this experiment
├── scoreboard.md                   # rolling, rewritten after each score call; sorted desc by metric_value
└── deep-research-<date>-<slug>.md  # final landscape doc (written in Phase 4)
```

Per-trooper state files (outbox.jsonl, status.json, pane.json, inbox.md) continue to live at `<topic_dir>/<commander>-codex/` (unchanged from v0.0.x).

### 3.4 Result.json schema

Unchanged from v0.26.0:

```json
{
  "branch_id":      "exp-007",
  "approach_label": "Modern LeNet + weight decay 1e-4",
  "metric_name":    "accuracy",
  "metric_value":   0.9912,
  "status":         "ok",
  "runtime_s":      28.4,
  "log_paths":      ["./stdout.log", "./stderr.log"],
  "notes":          "free-form trooper notes"
}
```

Schema validation via `cw_deep_research_validate_result_json` is also unchanged (`branch_id` field name kept for back-compat; populated with the global exp-NNN id).

### 3.5 Sandboxing (unchanged from v0.26.0)

Honor-system. Troopers told (not enforced) to stay inside their experiment dir. `--allow-net=true` is the default (UX #5 fix); no flag. DANGER block in README updated.

### 3.6 Composability with `/clone-wars:meditate`

The `--seed-from <landscape-path>` flag is preserved. When given:
- `_deep-research/seed-from.txt` records the path.
- Phase 1 (metric discussion) is unchanged — seeds don't define the metric.
- Phase 3 (research loop) — Yoda uses the meditate landscape's `## Approaches` section to inform the first ~N dispatches. After that, the loop is advisor-driven as normal.

### 3.7 Stop logic state machine

```
session_start = ISO-8601 at end of Phase 2
time_budget = read from time-budget.txt

after each experiment completes:
  if time_budget != 'none' AND (now - session_start) >= time_budget:
    AskUserQuestion: Continue / Stop / Extend
  elif time_budget == 'none' AND exp_count >= 5 AND last_5_no_improvement_over_1pct:
    AskUserQuestion: Continue / Stop / Adjust direction
  else:
    continue
```

`last_5_no_improvement_over_1pct` is computed by reading `scoreboard.md` after each score call:
- Identify current best `metric_value` (top `status: ok` row).
- Walk the last 5 chronological experiments; each must have `(metric_value - best) / best * 100 < 1.0` (i.e. no improvement of >1% over the running best at any point in the window).
- Floor: don't fire before exp 5 even if window math would technically qualify.

## 5. Directive outline (`commands/deep-research.md`)

```
## Task list (TaskCreate × 8 BEFORE Phase 1)

| # | subject                                       |
|---|----------------------------------------------|
| 0 | 0 Args + init (no budget) [yoda]             |
| 1 | 1 Metric discussion [yoda + user]            |
| 2 | 2 Preflight (roster + time limit) [yoda + user]|
| 3 | 3 Spawn roster (parallel) [yoda]             |
| 4 | 4 Research loop (advisor-driven) [yoda]      |
| 5 | 5 Synthesis [yoda]                           |
| 6 | 6 Teardown + archive [yoda]                  |
| 7 | 7 Present final doc [yoda]                   |

Per-experiment sub-rows: `4.NNN <Rank> <Cmdr> on <approach-label>` are
fired by the directive at each dispatch (replaces v0.26.0's `2a.N.bID`
shape — `4.NNN` because the loop is task 4 and exp counter is global).
```

Detailed step-by-step:

- **Phase 1 (metric discussion)** — loop of AskUserQuestion → format → AskUserQuestion-confirm. Directive prose covers the adaptive length.
- **Phase 2 (preflight)** — single AskUserQuestion for time limit; commit `time-budget.txt`. Then explain roster choice in chat.
- **Phase 3 (spawn)** — N parallel Bash tool calls in a single message. Spawn-rollback runbook (auto-retry-once) inline.
- **Phase 4 (research loop)** — for-each iteration:
  - Yoda inline-decides what to dispatch (free-form prose).
  - One or more `experiment-send.sh` calls (parallel via single message if dispatching to multiple troopers simultaneously).
  - One or more `experiment-wait.sh` calls (parallel via single message).
  - `score.sh` call.
  - Stop-check via `cw_deep_research_check_stagnation` / `cw_deep_research_check_time_budget`.
- **Phase 5 (synthesis)** — single Write call for landscape doc.
- **Phase 6 (teardown)** — `bin/teardown.sh --pairs` then `bin/deep-research-teardown.sh`.
- **Phase 7 (present)** — show the user the archive path + winner code path + Suggested-next line.

All Bash tool code blocks in the directive use **absolute paths** verbatim (no reliance on env-var persistence across calls). Per-block prelude pattern:
