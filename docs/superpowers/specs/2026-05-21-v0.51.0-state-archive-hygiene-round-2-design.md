## #3 — Pre-finalize state reconcile

### Root cause (confirmed by audit)

`bin/deep-research-monitor.sh` emits liveness events (`stale`/`stuck`) but does NOT write `state.txt` directly. The directive `commands/deep-research-resume.md:51` is the actual writer of `phase=stale`. The race window:

1. Trooper's experiment finishes; `result.json` written at T+0.
2. Trooper emits `{"event":"done"...}` to outbox at T+0.
3. EITHER the session terminates before Yoda's resume handler processes the done event, OR the stale event (emitted earlier at T-100s) is processed AFTER done because of event ordering.
4. State.txt ends with `phase=stale` (or worse, the stale write overwrote a done write).
5. `finalize.sh::case "$cur_phase"` at L40-49 maps `phase=stale` → `phase=incomplete`.

The clean fix runs at finalize time, before the case-mapping: replay each trooper's outbox tail since the last processed cursor and apply terminal events (done/error) directly to `state.txt`.

### New helper

`cw_deep_research_trooper_state_reconcile <art_dir> <commander>` in `lib/deep-research.sh`:

- Reads trooper's `outbox.jsonl` starting from `<art_dir>/troopers/<commander>/liveness-cursor.txt` offset (or 0 if cursor missing).
- Scans for the LAST `done` or `error` event in that tail.
- Resolves `current_exp_id` from `state.txt`.
- Branching:
  - LAST event is `done` AND `result.json` exists at `<art_dir>/troopers/<commander>/experiments/<current_exp_id>/result.json` → write `phase=idle` via `cw_deep_research_trooper_state_write`. Matches the v0.28.1 score.sh semantics.
  - LAST event is `error` → write `phase=failed`.
  - LAST event is `done` BUT result.json missing → leave phase unchanged (incomplete result is real).
  - No terminal event in tail → leave phase unchanged.
- Idempotent — calling twice produces same final state.
- Returns rc=0 on any branch; rc=1 only on argument errors.

### Call site

`bin/deep-research-finalize.sh:36-50` — inside the existing `while read -r cmdr` loop, BEFORE the `cur_phase=$(...) case ...` block. One-line invocation:

```bash
cw_deep_research_trooper_state_reconcile "$ART" "$cmdr"
```

The subsequent case-mapping then sees the reconciled phase. A trooper that emitted done before session-end is now correctly mapped `idle → complete` rather than `stale → incomplete`.

## #5 — Spawn failure forensics

### Root cause

`bin/spawn.sh::_spawn_bootstrap_fail` (lines 251-260) captures pane scrollback via `tmux capture-pane -p -t "$PANE" | tail -n 25` and pipes to **stderr** (not disk). After `cw_state_archive` runs and renames the trooper dir with `-FAILED` suffix, the stderr is lost. Example: `~/.clone-wars/archive/.../design-a-deploy-plan/cody-codex-20260520T115806Z-FAILED/` has 0-byte outbox, no `status.json`, no captured scrollback.

### New helper

`cw_spawn_capture_failure_forensics <commander> <model> <topic> <pane_id> <reason> [<event_line>]` in `lib/ipc.sh` (where the rest of the spawn IPC helpers live):

- Resolves the trooper dir via `cw_state_dir "$COMMANDER" "$MODEL" "$TOPIC"` (existing helper).
- Returns rc=1 if the dir is missing or unwritable (best-effort: spawn flow continues to archive even on rc=1).
- Writes `<trooper-dir>/failure-reason.txt` with this layout:

```
# Spawn bootstrap failure
timestamp:     <iso UTC, date -u +%Y-%m-%dT%H:%M:%SZ>
commander:     <commander>
model:         <model>
topic:         <topic>
pane_id:       <pane_id>
fail_reason:   timeout|error_event
ready_timeout: <CW_SPAWN_READY_TIMEOUT seconds>

## Pane scrollback (last 50 lines, captured BEFORE pane kill)
<tmux capture-pane -p -t "$PANE" | tail -n 50>

## Event context
<the {"event":"error",...} JSONL line if fail path was error_event;
 else "no error event before timeout">
```

- Atomic write via `cw_atomic_write` (tmp + mv).

### Call site

`bin/spawn.sh::_spawn_bootstrap_fail` (lines 251-260) — call the helper BEFORE `cw_state_archive`. Pass `timeout` or `error_event` as the reason based on the existing branch, and the captured `$event_line` (if any) as the optional argument.

```bash
_spawn_bootstrap_fail() {
  local reason="${1:-timeout}" event_line="${2:-}"
  log_error "pane content (last 25 lines, captured BEFORE kill):"
  tmux capture-pane -p -t "$PANE" 2>/dev/null | tail -n 25 >&2 || true
  cw_spawn_capture_failure_forensics \
    "$COMMANDER" "$MODEL" "$TOPIC" "$PANE" "$reason" "$event_line" \
    || log_warn "failure-reason.txt write failed (continuing to archive)"
  cw_pane_kill_now "$PANE"
  local failed_archive
  failed_archive=$(cw_state_archive "$COMMANDER" "$MODEL" "$TOPIC" FAILED)
  log_error "state archived to: $failed_archive"
  exit 1
}
```

The two call sites (timeout at line 268, error_event at line 272) gain explicit reason args:

```bash
# Timeout path:
_spawn_bootstrap_fail timeout
# Error event path:
_spawn_bootstrap_fail error_event "$event_line"
```

The stderr echo (`tail -n 25 >&2`) is **preserved** — UX during fail is unchanged; the file write is additive.

## #6 — `result.json` schema add

### Root cause

`config/prompt-templates/deep-research/experiment.md:63-76` defines the schema troopers write. The schema currently has 11 keys; checkpoints are referenced ad-hoc inside `notes`. Deploy authors (and downstream consumers like `bin/consult-research-send.sh`) cannot programmatically read the path.

### Schema change

Add one nullable field to the schema block (line 70, between `log_paths` and `notes`):

```json
{
  "branch_id":           "{{EXP_ID}}",
  "approach_label":      "{{APPROACH_LABEL}}",
  "metric_name":         "{{METRIC_NAME}}",
  "metric_value":        <number or null>,
  "status":              "ok" | "fail" | "timeout" | "cost_blown",
  "runtime_s":           <number — wall-clock for the run phase only>,
  "log_paths":           ["./stdout.log", "./stderr.log"],
  "checkpoint_path":     <absolute path or null>,
  "notes":               "<free-form, max 500 chars>",

  "self_reported_count": <integer or null>,
  "self_reported_ratio": <number or null>,
  "self_reported_notes": "<string or null>"
}
```

Add one explanatory bullet after the existing `metric_value` / `log_paths` bullets:

```
- checkpoint_path: absolute path to any model checkpoint saved during
  this run, or null if no checkpoint was produced. Use this for
  downstream /clone-wars:deploy hand-off — the deploy lane reads this
  field directly instead of parsing free-text from notes.
```

### Why no validator + no static test against archives

- **No score.sh validator:** the field is informational, not metric-bearing. Failing a run because checkpoint_path is null would over-constrain — many runs legitimately produce no checkpoint.
- **No archive scanner:** legacy archives lack the field by design. A gating test would either fail or need a version gate that adds complexity for minimal value.
- **No consumer requires it yet:** existing `result.json` readers (`score.sh`, `consensus.sh`, `cw_deep_research_render_summary`, `cw_deep_research_json_field`) read fields they know about and ignore unknowns. Adding the field is pure additive.

The lock invariant just greps for the literal `"checkpoint_path"` in the template — future trooper runs include the field.

## Risk surface

- **#3 cursor edge cases.** Helper depends on `<trooper>/liveness-cursor.txt`. Missing/empty cursor falls back to reading the entire outbox (acceptable: outbox files are <100KB). Tested explicitly in case 4 (no terminal events).
- **#3 idempotency.** Calling twice in the same finalize run (won't happen in normal flow, but defensive) produces stable final state. Tested in case 5.
- **#5 best-effort.** If the trooper dir was deleted between spawn and fail (very rare), the helper returns rc=1 and `_spawn_bootstrap_fail` continues to archive without forensics. The original stderr echo still fires so the user sees scrollback in the fail-time log.
- **#5 _spawn_bootstrap_fail signature change.** Currently takes no args. After T4, takes `<reason> [<event_line>]`. Single caller, both call sites updated in the same commit.
- **#6 schema additive.** Existing consumers ignore unknown keys. New field is nullable. No legacy compat issue.
- **#6 fixture regression risk.** If a deep-research prompt fixture test exists and compares the rendered experiment.md byte-for-byte, the schema change will break it. Regenerate the fixture inline in T5 (same pattern as v0.50 T8 fixture regen for consult prompts).

## What this does NOT do

- No runtime handler change (no `commands/deep-research-resume.md` edit). Reconcile fires only at finalize.
- No `checkpoint_path` validator in `score.sh` or other readers. Field is informational.
- No legacy-archive static test. Pre-v0.51 result.json files don't have the field by design.
- No `cw_deep_research_state_reconcile` invocation outside `finalize.sh`. Runtime done-event processing still goes through the existing resume.md handler.
- No teardown.sh (consult/deploy lanes) integration of the spawn forensics helper. v0.51 covers the deep-research spawn fail path; consult/deploy teardown failure modes can ride a future release if needed.
