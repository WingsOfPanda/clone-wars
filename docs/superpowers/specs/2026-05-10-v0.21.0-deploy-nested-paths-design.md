# v0.21.0 ‚Äî Multi-repo deploy: nested + heterogeneous fleet support

**Date:** 2026-05-10
**Spec status:** approved (single-repo case unchanged)

## Problem

`/clone-wars:deploy` v0.20.5 multi-repo path is a closed contract between
`cw_consult_emit_soft_dag` (lib/consult-walk.sh) and `bin/deploy-dag-parse.sh`
+ `bin/deploy-multi-init.sh`, both targeting a flat-monorepo layout with
lowercase-dashed repo slugs. Real-world fleets like ARS
(`/home/liupan/ARS/{ars_fleet,ars_gateway}/ARS-{TaskServe,Perfusion,LVMGateway,Gateway}/`)
break the contract on two axes:

1. **DAG section format mismatch.** `bin/deploy-dag-parse.sh` requires
   parser-conforming lines (`N. <slug> ‚Äî <desc>`); human-authored design
   docs (`/forcevision`, `/holocron`, hand-written) ship Unicode box
   diagrams (`‚îå‚îÄ‚îÄ‚îê`, `‚îÇ TaskServe ‚îÇ`) that the parser ignores. Verified
   at `lib/deploy-dag.sh:19` regex `^([0-9]+)\.[[:space:]]+([a-z0-9-]+)[[:space:]]+‚Äî[[:space:]]+(.+)$`.
   Section is detected (`bin/deploy-init.sh:85-86`) but parser yields zero
   lines ‚Üí `bin/deploy-dag-parse.sh:64` exits 1 with `no DAG lines parsed`.

2. **Sub-repo path resolution.** `bin/deploy-multi-init.sh:60-64` stats
   `$HUB_CWD/$repo/CLAUDE.md` (flat one-level), and the slug regex
   rejects CapWords + underscore + slash. No legal slug encodes nested
   CapWords paths like `ars_fleet/ARS-TaskServe`. Verified at
   `bin/deploy-multi-init.sh:60-64` and `lib/deploy-dag.sh:19`.

The v0.20.0 spec
(`docs/superpowers/specs/2026-05-09-deploy-multi-repo-dag-design.md:130-138, 193, 460-461`)
explicitly assumes flat-sibling layout + consult-emitted DAG prose, so
both gaps are by-design v0.20.0 limitations rather than v0.20.x bugs.

## Goal

After v0.21.0 lands, `/clone-wars:deploy <design-doc>` succeeds when:

- The design doc was hand-written (or produced by `/forcevision` /
  `/holocron`) with a `## Execution DAG` section that contains box
  diagrams or other non-parser-conforming prose ‚Äî instead of aborting,
  the deploy directive **rescues** the section by extracting implicit
  DAG-lines via Yoda's judgment + `AskUserQuestion` confirmation, writes
  them into a `### DAG Lines` subsection of `_deploy/<topic>/design.md`
  (the local copy, not the user's source), and re-runs the parser.

- Sub-repo paths are nested + CapWords (e.g. `ars_fleet/ARS-TaskServe`):
  the DAG line carries an optional absolute path in parens after the slug,
  and `bin/deploy-multi-init.sh` honors it; flat-monorepo deploys (no
  parens) remain byte-equal to v0.20.5.

- Conductor's cwd matches the design doc's hub root (per the cross-hub
  constraint from the design doc itself ‚Äî that's a user-side discipline,
  not plugin code).

Single-repo `/clone-wars:deploy` is byte-equal to v0.20.5.

## Architecture

Two separable changes, one per failure.

### Change 1 ‚Äî Conductor-side DAG rescue intercept

`/clone-wars:consult` already emits parser-conforming DAG; **no change to
consult**. The rescue belongs in `commands/deploy.md` Step 0 because the
gap is human-authored docs, not consult output, and the rescue uses
Claude's judgment (extracting wave structure from box-art prose) rather
than new bash.

**Trigger detection (Step 0, after `bin/deploy-init.sh` exits non-zero):**

```
init_rc=$?
if (( init_rc != 0 )) && [[ -f "$ART_DIR/design.md" ]] \
   && grep -qE '^## Execution DAG\b' "$ART_DIR/design.md" \
   && [[ ! -f "$ART_DIR/dag-waves.txt" ]]; then
  : # rescue path
fi
```

The signature `(init failure) AND (design.md cp'd) AND (DAG section present)
AND (dag-waves.txt missing)` uniquely identifies a DAG-parse failure;
other init failures (branch create, multi-init missing sub-repo) leave a
different state shape on disk.

**Rescue mechanism (conductor-side):**

1. Read `$ART_DIR/design.md` `## Execution DAG` section.
2. Extract implicit DAG-lines using Yoda's judgment:
   - Wave boundaries (box rows, separator lines like `‚ñº`, headings).
   - Parallel groups within a wave (multiple boxes side-by-side).
   - Repo names (from box labels; cross-reference Components section
     for absolute paths if not stated in DAG).
   - Dependencies (wave N depends on wave N-1).
3. `AskUserQuestion`: 3 options ‚Äî "Looks right (write it & retry)",
   "Let me edit", "Abort deploy".
4. On confirm: use `Edit` tool to insert a `### DAG Lines` subsection
   under `## Execution DAG` in `$ART_DIR/design.md`. Body is the
   parser-conforming lines, one per step.
5. Re-invoke `bin/deploy-dag-parse.sh "$ART_DIR/design.md" "$ART_DIR"`
   then `bin/deploy-multi-init.sh "$TOPIC" "$TARGET_CWD"`.
6. Continue with init.sh's post-parse steps (target_cwd.txt write,
   branch-create, auto_provider write) by replaying them inline.

The rescue is **one-shot per deploy** ‚Äî if it fails (user picks Abort,
or re-parse still fails), the directive surfaces the error and exits
without retry.

**Audit trail:** rescue writes `$ART_DIR/dag-rescue.log` with the
extracted DAG-lines and the user's confirmation choice. Useful for
later post-mortems and for /consult-team discussions about whether
human-authored doc conventions should be folded into a future
emitter.

### Change 2 ‚Äî DAG line carries optional path; relaxed slug regex

The flat-sibling assumption (`$HUB_CWD/$repo/CLAUDE.md`) is the wall.
Three designs were considered:

| Design | Pros | Cons |
|---|---|---|
| A ‚Äî `## Sub-Project Paths` table convention | Explicit; reusable | New section spec; /consult emitter update; doc bloat |
| B ‚Äî Auto-extract from `### <Name> (\`<abs-path>\`)` Components headings | Zero new convention | Fragile (heading-format coupling); silent failures |
| **C ‚Äî DAG line carries optional path** | Minimal change; backward-compat; one source of truth | Slightly longer DAG lines |

**Chosen: C.** Parser regex extension:

```
old: ^([0-9]+)\.[[:space:]]+([a-z0-9-]+)[[:space:]]+‚Äî[[:space:]]+(.+)$
new: ^([0-9]+)\.[[:space:]]+([A-Za-z0-9_-]+)([[:space:]]+\((/[^)]+)\))?[[:space:]]+‚Äî[[:space:]]+(.+)$
                                ^^^^^^^^^^^^^^^^^^      ^^^^^^^^^^^^^^^^^^^^^^^^^
                                CapWords/underscore     optional absolute path
```

Old line (flat monorepo, byte-equal v0.20.5):
```
1. taskserve ‚Äî Edit deploy.py + tests
```

New line (nested fleet, opt-in):
```
1. ARS-TaskServe (/home/liupan/ARS/ars_fleet/ARS-TaskServe) ‚Äî Edit deploy.py + tests
```

`cw_deploy_dag_parse_line` returns 5-field TSV instead of 4:
`<step>\t<repo>\t<path|none>\t<desc>\t<deps-csv|none>`. Backward
compat: `<path>` is `none` when the optional `(<abs-path>)` group
isn't matched.

`bin/deploy-multi-init.sh` reads the 5-field TSV; uses the path
field if present, else falls back to `$HUB_CWD/$repo/CLAUDE.md`
flat-stat (current behavior). Branch-base capture (`<cmdr>-branch-base.sha`)
runs inside the resolved CWD regardless of which path source was used.

`bin/deploy-dag-parse.sh` writes 5-field TSV to `dag-waves.txt`. Step
3a's preflight reader (`bin/preflight-layout.sh --cwd-from`) reads
the path column from the 3-col `cmdr-cwd-map.txt` already produced by
multi-init ‚Äî no change needed there.

## Backwards compatibility

- `cw_deploy_dag_parse_line`: backward-compat for flat-monorepo
  4-field consumers preserved by appending the new `path` field as
  field 3 (between `repo` and `desc`). All readers updated atomically.
- `dag-waves.txt`: 5-field TSV; previously 4-field. **Breaking** for
  any external reader, but only `bin/deploy-multi-init.sh` and the
  Step 3b directive read it ‚Äî both updated in the same release.
- Regex: `[A-Za-z0-9_-]+` is a strict superset of `[a-z0-9-]+` ‚Äî every
  v0.20.5 valid slug remains valid in v0.21.0.
- Optional `(<abs-path>)` group is opt-in; flat-monorepo docs that
  don't include it route through the v0.20.5 fallback path
  (byte-equal behavior).
- Single-repo deploy path: zero touch ‚Äî all v0.21.0 changes are
  multi-repo branch only.

## Failure modes

| Where | Trigger | Behavior |
|---|---|---|
| Rescue extraction | DAG section is unstructured (no waves, no boxes, no enumeration) | Yoda surfaces "Cannot extract DAG; please add `### DAG Lines` subsection manually." Exit 1, no retry. |
| Rescue confirm | User picks "Abort deploy" | Exit 0 with `log_warn "rescue aborted by user"`; archive `_deploy/<topic>/` for inspection. |
| Re-parse after rescue | Re-invoked `bin/deploy-dag-parse.sh` still fails | Surface parser stderr + the inserted DAG-lines block; exit 1. No second rescue (one-shot). |
| Path field invalid | DAG line has `(<path>)` but path doesn't exist | `bin/deploy-multi-init.sh` exits 1 with `log_error "sub-repo '<repo>' not found at <path>"` (existing message; just the path source changed). |
| CLAUDE.md/AGENTS.md missing at path-field path | Same as flat-stat case | Same error message; multi-init's existing guard at line 63-64 covers both path sources. |

## Out of scope

- **`/consult` emitter for nested paths.** v0.21.0 doesn't extend
  `cw_consult_emit_soft_dag` to write `(<abs-path>)`. /consult still
  emits flat slugs only; nested paths require either hand-authored
  DAG-lines or the rescue intercept. Future v0.22+ if a /consult run
  on a heterogeneous fleet becomes a real use case.
- **Auto-extract paths from Components-section headings (Design B).**
  Considered and rejected for fragility (silent failure when heading
  format drifts).
- **`## Sub-Project Paths` table convention (Design A).** Considered
  and rejected for spec/emitter overhead.
- **Cross-hub conductor handling.** v0.21.0 still requires the
  conductor to run from the design doc's hub root (cross-hub deploy
  memory in design doc itself enforces this user-side). Future v0.22+
  may extend cross-hub.
- **Recovery from partial rescue.** If user accepts the rescue and
  multi-init then fails (e.g. one path is wrong), the deploy aborts
  cleanly with archive; no in-flight retry. v0.22+ may add this.
